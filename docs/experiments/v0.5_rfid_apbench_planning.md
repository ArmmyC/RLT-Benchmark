# v0.5 RFID-APBench Planning

## Purpose

v0.5 defines RFID-APBench, a public and synthetic RTL benchmark for RFID/NFC-style area plus activity evaluation.

AP means area plus activity. Activity is a VCD toggle-count proxy gathered from a declared simulation workload. It is not measured silicon power, final power, or a PPA claim.

This is a planning milestone only. It does not run benchmarks, create model outputs, add private RTL, or add fine-tuning code.

## Scope

RFID-APBench should provide small, public, reproducible RTL tasks that resemble control and datapath patterns common in RFID/NFC designs without copying private SiliconCraft RTL, private task text, or proprietary implementation details.

The initial benchmark should validate:

- public/synthetic task packaging
- deterministic correctness gates
- area metric capture from a matched synthesis flow
- activity metric capture from VCD toggle counts
- sanitized result exports
- honest reports that separate correctness, area, and activity

Private RTL is explicitly out of scope. Future private evaluation requires a separate access policy and must not be mixed into public benchmark artifacts.

## Relationship To v0.4

v0.4 established correctness-gated area-oriented scoring for public RTL-OPT flows. v0.5 moves from generic public RTL optimization toward a domain-shaped but still public/synthetic RFID/NFC benchmark.

The key change is replacing unsupported power claims with an explicit activity proxy:

- area comes from the declared synthesis flow
- activity comes from VCD toggle counts under a declared workload
- timing remains a required gate only when a task declares a timing constraint
- invalid samples contribute zero in aggregate comparisons

RFID-APBench must not claim measured silicon power. Lower toggle counts may indicate lower simulated switching activity for the benchmark workload, but they are not a substitute for a full power analysis.

## Benchmark Layout

Proposed public task root:

```text
benchmarks/rfid_apbench/
  README.md
  manifest.yaml
  tasks/
    ap_001_idle_counter/
      task.yaml
      prompt.md
      reference.sv
      testbench.sv
      constraints.sdc
      activity_workload.yaml
      expected/
        reference_metrics.yaml
```

Each task directory should contain only public/synthetic source material. The task text must be written for benchmark use and must not paraphrase private customer, product, or internal design requirements.

The public task schema is defined in `docs/specs/rfid_apbench_task_schema.md`.

## Evaluation Gates

A generated sample is eligible for area/activity scoring only after the required gates pass in order:

1. extraction
2. compile
3. simulation or equivalence
4. synthesis
5. timing if required
6. area metric availability
7. activity metric availability

The pipeline should stop scoring at the first failed required gate. Gate results must still be recorded so reports can distinguish generation failures, correctness failures, synthesis failures, and metric-availability failures.

Simulation is the default correctness gate for the initial synthetic tasks. Equivalence may be used when the task provides a reference suitable for a formal equivalence flow.

## Area/Activity Scoring

The initial scoring mode is `area_activity`.

```text
area_score = min(reference_area / generated_area, 2.0)
activity_score = min(reference_activity / generated_activity, 2.0)
score = 0.5 * area_score + 0.5 * activity_score
```

Inputs must be finite positive numbers from matched flows. Invalid, missing, zero, negative, non-finite, or unit-mismatched values make the sample invalid for scoring.

Per-sample analytical artifacts should use `score_status: invalid` and a null score for failed gates. Aggregate comparisons should also publish an all-sample mean where invalid samples contribute zero, so correctness failures cannot disappear from rankings.

## VCD Toggle-Count Flow

Each scored task must define an activity workload. The workload should be deterministic and public:

- reset sequence
- clock count or transaction count
- input stimulus sequence
- VCD start and stop window
- top module and signal inclusion policy
- ignored signals, if any

The evaluator should run the same workload against the reference RTL and generated RTL, emit VCD files locally, count toggles over the declared window, and export only sanitized aggregate activity metrics.

The primary activity metric for v0.5 is total weighted toggle count. The first version may use unweighted signal toggles if width-normalized counting is not yet implemented, but the selected method must be recorded in `activity_metric`.

Do not commit VCD files, raw simulation logs, generated RTL, raw prompts, or raw model responses.

RFID-APBench exports should be commit-safe and structured. Proposed fields include:

| --- | --- |

| `correctness_pass` | Simulation or equivalence gate result. |
| `correctness_method` | `simulation` or `equivalence`. |

| `timing_status` | `pass`, `fail`, `not_required`, or `unavailable`. |
| `area_metric_status` | Area metric availability. |
| `activity_metric_status` | Activity metric availability. |

| `generated_activity` | Generated activity count from the matched workload. |

Sanitized exports must not contain private RTL, private task text, raw prompts, raw model responses, generated RTL, VCD files, outputs, logs, secrets, model weights, training datasets, or adapter checkpoints.

## Initial Synthetic Task Proposals

### `ap_001_idle_counter`

- Behavior: implement a counter that advances only during an active RFID/NFC polling window and holds state during idle periods.
- Optimization opportunity: avoid unnecessary counter toggles and remove unused terminal-count logic while preserving boundary behavior.
- Correctness gate: simulation checks reset, active increments, idle hold behavior, wrap or saturate behavior as declared by the task.
- Area/activity expectations: optimized solutions should avoid extra comparators and reduce idle-window toggles relative to a naive always-counting implementation.
- Timing requirement: no special timing requirement in the MVP; record timing as `not_required`.
- RFID/NFC relevance: polling loops, frame wait timers, and low-duty-cycle control blocks are common in tag and reader control logic.

### `ap_002_command_decoder`

- Behavior: decode a small public synthetic command set into control signals for read, write, inventory, select, and idle operations.
- Optimization opportunity: simplify decode logic, avoid redundant default toggles, and keep inactive outputs stable.
- Correctness gate: simulation over all declared command values and illegal/default cases.
- Area/activity expectations: optimized solutions should reduce unnecessary wide compares and output switching on repeated commands.
- Timing requirement: optional single-cycle decode timing may be required in later stages; MVP can mark timing `not_required`.
- RFID/NFC relevance: command decoding is central to RFID/NFC protocol controllers without requiring private protocol extensions.

### `ap_003_register_bank_unnecessary_writes`

- Behavior: implement a small register bank with byte or word writes, reads, reset defaults, and write-enable controls.
- Optimization opportunity: gate register updates when writes are disabled or when write data matches current state, without changing observable reads.
- Correctness gate: simulation covers reset, read-after-write, disabled writes, repeated same-value writes, and address boundaries.
- Area/activity expectations: generated RTL may trade simple equality checks for lower register toggle activity; reports should expose whether the area cost is worth the activity reduction.
- Timing requirement: no required timing gate in the first MVP unless equality-gating paths are later constrained.
- RFID/NFC relevance: tag/session configuration and reader control registers often sit idle for long periods and should avoid unnecessary switching.

### `ap_004_crc_serial_parallel_tradeoff`

- Behavior: compute a small public synthetic CRC update over a declared RFID/NFC-style frame fragment using either serial or limited parallel logic.
- Optimization opportunity: balance combinational area against sequential activity and avoid overbuilt parallel CRC hardware when throughput constraints do not require it.
- Correctness gate: simulation compares CRC outputs against known public vectors and reset/seed behavior.
- Area/activity expectations: serial designs may reduce area but require more cycles; limited parallel designs may improve cycles with higher area. The task must declare the allowed latency so scoring is fair.
- Timing requirement: required only if the task declares a maximum clock period or per-byte latency; otherwise timing is `not_required`.
- RFID/NFC relevance: CRC generation and checking are common in RFID/NFC frame handling.

### `ap_005_fsm_controller_idle_activity`

- Behavior: implement a small controller FSM for idle, receive, decode, respond, and error states with public synthetic transitions.
- Optimization opportunity: encode states and outputs to minimize idle switching while keeping transitions and outputs correct.
- Correctness gate: simulation covers legal transitions, error recovery, reset, and extended idle periods.
- Area/activity expectations: optimized implementations should keep outputs stable in idle and avoid combinational feedback that toggles every cycle.
- Timing requirement: no special MVP timing requirement; later versions may require single-cycle transition timing.
- RFID/NFC relevance: protocol controllers spend substantial time waiting for field, command, or frame events.

## MVP Readiness

Fine-tuning work must wait until RFID-APBench has an MVP that can score 5 to 10 public/synthetic tasks end-to-end with:

- deterministic task packaging
- extraction, compile, correctness, synthesis, timing-if-required, area, and activity gates
- reference and generated area metrics
- reference and generated VCD toggle-count activity metrics
- sanitized commit-safe result artifacts
- reports that clearly label activity as a proxy, not measured power

Until that gate is met, RTLBench should focus on benchmark design, adapters, evaluation, scoring, sanitized artifacts, and reports.

## Non-Goals

Do not add:

- private RTL
- private task text
- raw prompts
- raw model responses
- generated RTL
- benchmark outputs or logs
- secrets
- model weights
- training datasets
- LoRA, QLoRA, or DoRA adapters
- fine-tuning scripts

## Staged Plan

1. Freeze the RFID-APBench task schema.
2. Freeze the area/activity scoring contract.
3. Implement synthetic task fixtures for the five initial proposals.
4. Add VCD toggle-count extraction on deterministic workloads.
5. Add sanitized export and summary reports.
6. Validate the MVP on 5 to 10 public/synthetic tasks.
7. Only then consider a future fine-tuning repository that calls into RTLBench for evaluation.

No benchmark execution is part of this planning document.
