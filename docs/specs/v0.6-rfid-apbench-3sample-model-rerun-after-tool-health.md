# Feature Spec: v0.6 RFID-APBench 3-Sample Model Rerun After Tool Health Restore

## Goal

Rerun the v0.6 RFID-APBench 3-sample model baseline now that the local EDA toolchain health has been restored and validated.

The previous v0.6 3-sample baseline completed generation but downstream compile/synthesis interpretation was invalidated by tool-health failures. After the tool-health bugfix and report refresh, reference validation and `reference_copy` candidate fixture validation show healthy Icarus, `vvp`, and Yosys with 5/5 passes.

This rerun should produce a fresh sanitized model-evaluation baseline with the same five public/synthetic tasks, exactly three samples per task, unchanged benchmark assets, and the tool-health-aware evaluator.

## Inputs

Read first:

- `reports/v0.5_rfid_apbench_reference_validation.md`
- `reports/v0.5_rfid_apbench_candidate_smoke.md`
- `reports/v0.6_rfid_apbench_tool_health_refresh.md`
- `reports/v0.6_rfid_apbench_3sample_baseline.md`
- `reports/v0.6_rfid_apbench_3sample_failure_analysis.md`
- `scripts/run_rfid_apbench_3sample_baseline.py`
- `scripts/evaluate_rfid_apbench_candidates.py`
- `src/rtlbench/tool_health.py`

## Required Scope

Run a fresh v0.6 model baseline that:

1. Uses the existing five RFID-APBench tasks only.
2. Runs exactly 3 samples per task.
3. Uses unchanged prompts, references, testbenches, activity workloads, reference metrics, extractor policy, evaluator policy, and scoring formula.
4. Uses the configured/default model, expected `qwen36-27b` when available.
5. Uses `neutral_baseline` or the existing default prompt behavior.
6. Uses low/deterministic generation settings where supported.
7. Uses the post-bugfix tool-health-aware evaluator.
8. Emits sanitized Markdown/CSV/JSONL reports.
9. Preserves the historical pre-tool-health baseline and does not overwrite its interpretation unless the report clearly records the new run id and fresh result.
10. Does not commit raw prompts, raw responses, generated RTL, VCDs, logs, compiled artifacts, synthesis scratch, private data, or training artifacts.

## Required Files

Update or create as appropriate:

- `reports/v0.6_rfid_apbench_3sample_baseline.md`
- `reports/v0.6_rfid_apbench_3sample_baseline.csv`
- `reports/v0.6_rfid_apbench_3sample_baseline.jsonl`
- optional `reports/v0.6_rfid_apbench_3sample_post_tool_health_summary.md`

Do not commit raw output directories or generated RTL.

## Report Requirements

The Markdown report must include:

- scope and non-goals
- run id
- model and prompt profile
- endpoint status without secrets
- tool health status for Icarus, `vvp`, and Yosys
- task count and samples per task
- total sample count
- generation/extraction/candidate availability counts
- compile/correctness/synthesis counts
- area/activity availability counts
- valid score count
- mean valid score
- mean all-sample score with invalid rows as zero
- per-task aggregate summary
- per-sample sanitized summary
- failure distribution
- comparison against the historical pre-tool-health v0.6 run
- statement that activity is a VCD toggle-count proxy, not measured power
- public synthetic / not private / not fine-tuning caveat

## Expected Behavior

If endpoint and tools are healthy:

- attempt all 15 samples
- evaluate extracted candidates through compile, simulation correctness, activity, synthesis, and scoring
- distinguish model-output failures from tool-health failures using the updated evaluator

If the endpoint fails:

- produce sanitized `request_failed` or `endpoint_unavailable` rows as appropriate
- do not fake generation

If tool health fails again:

- produce sanitized `tool_health_failed` or `tool_startup_failure` rows
- do not classify those as candidate `compile_failure`

## Tests

Run:

```bash
python -m pytest
```

Do not require a live model endpoint in unit tests.

## Forbidden

Do not commit:

- private SiliconCraft RTL
- private task text
- raw prompts
- raw model responses
- generated RTL
- model outputs
- raw benchmark run outputs
- VCD files
- simulator logs
- synthesis logs
- compiled simulation artifacts
- tool scratch files
- secrets or endpoint credentials
- model weights
- training datasets
- LoRA, QLoRA, or DoRA adapters
- fine-tuning scripts

## Non-Goals

Do not:

- add or modify benchmark tasks
- modify prompts, references, testbenches, extractor policy, evaluator policy, or scoring formula
- compare models
- tune prompts
- create training data
- fine-tune a model
- claim measured power
- claim final silicon PPA
- integrate private benchmarks

## Definition of Done

Done when:

1. A fresh post-tool-health v0.6 3-sample model baseline is recorded in sanitized Markdown/CSV/JSONL.
2. All five tasks have exactly three attempted samples.
3. Reports distinguish generation, extraction, candidate compile/simulation/synthesis, scoring, and any tool-health failures.
4. Valid and all-sample scores are reported.
5. `python -m pytest` passes.
6. No raw/private/model-output/training artifacts are committed.
