# Feature Spec: v0.7 RFID-APBench Targeted Empty-Response Observability Rerun

## Goal

Run a small targeted rerun on the two RFID-APBench tasks that previously produced concentrated empty responses, using the new sanitized runner/endpoint observability fields.

This rerun should exercise the newly added metadata fields and determine whether empty rows look like zero-token/empty-content successful responses, response-shape issues, timeouts, HTTP/transport failures, retry behavior, or another sanitized request outcome.

This is a bounded observability rerun only. It is not prompt tuning, model comparison, benchmark modification, private evaluation, fine-tuning, or training-data creation.

- `reports/v0.7_rfid_apbench_targeted_empty_response_investigation.md`

- `benchmarks/rfid_apbench/tasks/ap_010_retry_timeout_fsm/prompt.md`

## Required Scope

Run a targeted model-generation/evaluation rerun that is bounded to:

- tasks: `ap_001_idle_counter`, `ap_010_retry_timeout_fsm`
- samples: exactly 3 samples per task, 6 attempted samples total
- model: configured/default model, expected `qwen36-27b` when available
- prompt profile/settings: unchanged existing default or `neutral_baseline`
- temperature/top-p/max tokens: unchanged deterministic defaults where supported
- benchmark assets: unchanged
- extractor/evaluator/scoring: unchanged
- report outputs: sanitized only

If the endpoint is unavailable, tools are unhealthy, or requests fail, do not fake results. Emit sanitized blocker/failure rows using the new observability fields.

## Report Requirements

The Markdown report must include:

- scope and non-goals
- targeted task list
- run id, model, prompt profile, endpoint status without secrets
- tool health status
- total attempted samples
- generation/extraction/candidate/evaluation gate counts
- request outcome distribution
- metadata availability counts for the new observability fields
- per-sample sanitized summary including request outcome, attempt count, content presence, character count, finish reason, token counts, status class, and parse status
- interpretation of empty/non-empty outcomes using only sanitized metadata
- comparison against the previous empty-response evidence for these two tasks
- benchmark asset decision
- fine-tuning readiness decision
- exactly one recommended next step
- activity proxy caveat
- public synthetic / not private / not fine-tuning caveat

## Empty-Response Interpretation Rules

Use these sanitized rules:

- `success_empty` with `response_character_count=0`, `response_content_present=true`, and `completion_tokens=0` supports an endpoint-returned empty string or zero-token completion.
- `success_empty` with `response_content_present=false` supports null content normalized to empty text.
- `success_empty` with positive `response_character_count` supports whitespace-only content.
- `response_schema_failure` or `response_parse_failure` means the response was not a parsed empty content value.
- `timeout`, `http_error`, or `transport_error` means the row is not a successful empty completion.
- If key metadata is unavailable, state `insufficient_evidence` rather than overclaiming.

## Decision Rules

Recommend exactly one:

- `bounded_prompt_profile_experiment_spec` only if targeted empty behavior remains repeatable and metadata supports successful empty/zero-token/empty-content outcomes rather than endpoint or schema failures.
- `endpoint_or_client_bugfix_spec` if metadata shows parse/schema/client-normalization behavior that should be fixed before prompt work.
- `endpoint_stability_spec` if timeout, HTTP error, transport error, or retry instability dominates.
- `no_immediate_followup_beyond_documentation` if the rerun produces no useful additional evidence.

Do not recommend fine-tuning, private evaluation, model comparison, or benchmark modification from this rerun.

## Non-Goals

Do not:

- add or modify benchmark tasks
- change prompts or prompt profiles

- add private benchmarks
- claim measured power

- change area/activity scoring formula
- change evaluator policy unless needed only for targeted-run plumbing

## Definition of Done

Done when:

1. The targeted rerun reports exist in sanitized Markdown/CSV/JSONL.
2. The run attempts exactly 3 samples each for `ap_001_idle_counter` and `ap_010_retry_timeout_fsm`, unless endpoint/tool blockers are explicitly reported.
3. The new observability fields are present in CSV/JSONL rows.
4. The Markdown report interprets outcomes using sanitized metadata without raw content.
5. Benchmark assets remain unchanged unless a separately scoped issue is justified.
6. Fine-tuning is not recommended from this evidence alone.
7. Exactly one next bounded step is recommended.
8. `python -m pytest` passes.
9. No raw/private/model-output/training artifacts are committed.
