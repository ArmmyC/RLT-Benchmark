# Bugfix Spec: v0.6 RFID-APBench Tool-Health Report Refresh

## Goal

Refresh the sanitized RFID-APBench reports after the evaluator tool-health bugfix so committed reports no longer present tool-startup/runtime failures as confirmed candidate RTL compile failures.

The code now distinguishes path discovery, tool health, tool startup failure, candidate compile failure, and candidate synthesis failure. However, the existing `reports/v0.6_rfid_apbench_3sample_baseline.*` artifacts were produced before that distinction and still contain `compile_failure` rows for cases later analyzed as tool-startup/runtime failures.

This is a report refresh / clarification step. It should not change benchmark tasks, prompts, references, testbenches, extractor policy, area/activity scoring formula, or model settings.

## Inputs

Read first:

- `docs/specs/v0.6-rfid-apbench-evaluator-tool-health-bugfix.md`
- `reports/v0.6_rfid_apbench_3sample_baseline.md`
- `reports/v0.6_rfid_apbench_3sample_baseline.csv`
- `reports/v0.6_rfid_apbench_3sample_baseline.jsonl`
- `reports/v0.6_rfid_apbench_3sample_failure_analysis.md`
- `scripts/evaluate_rfid_apbench_candidates.py`
- `scripts/validate_rfid_apbench_references.py`
- `scripts/run_rfid_apbench_3sample_baseline.py`
- `src/rtlbench/tool_health.py`

## Required Scope

Do one of the following, whichever is safest and most reproducible in the current environment:

1. Regenerate affected sanitized RFID-APBench reports with the updated tool-health-aware evaluator; or
2. Add a sanitized post-bugfix clarification report and a clear supersession note to the stale v0.6 baseline report.

The result must make it obvious to future readers that the old `compile_failure` rows from the v0.6 3-sample baseline should not be interpreted as confirmed candidate RTL compile errors when tool health failed before candidate parsing.

## Required Files

Update or create as appropriate:

- `reports/v0.6_rfid_apbench_3sample_baseline.md`
- optional regenerated `reports/v0.6_rfid_apbench_3sample_baseline.csv`
- optional regenerated `reports/v0.6_rfid_apbench_3sample_baseline.jsonl`
- optional `reports/v0.6_rfid_apbench_tool_health_refresh.md`
- optional `reports/v0.6_rfid_apbench_tool_health_refresh.csv`

Do not commit raw run directories, raw model responses, generated RTL, VCD files, simulator logs, synthesis logs, compiled artifacts, or tool scratch files.

## Report Requirements

The refreshed or companion report must include:

- scope and non-goals
- tool-health-aware interpretation of the prior baseline
- current tool path and health status in sanitized form
- whether reports were regenerated or only clarified
- whether the original v0.6 3-sample generated-output record is preserved as historical data
- clear statement that old `compile_failure` rows are superseded by the failure-analysis/tool-health interpretation where applicable
- next operational step needed before rerunning model scoring, if any
- activity proxy caveat
- public synthetic / not private / not fine-tuning caveat

## Expected Behavior

If the current toolchain is healthy:

- reference validation and candidate fixture smoke should be rerunnable
- rerun the relevant sanitized reports if practical
- do not commit raw artifacts

If the current toolchain is unhealthy:

- do not fake successful compile/synthesis results
- produce sanitized `tool_health_failed` or `tool_startup_failure` report rows or clarification notes
- clearly state that a new model-scoring baseline must wait for a healthy toolchain

## Tests

Run:

```bash
python -m pytest
```

Add tests only if helper code is introduced. Do not require a live model endpoint in unit tests.

## Forbidden

Do not commit:

- private SiliconCraft RTL
- private task text
- raw prompts
- raw model responses
- generated RTL
- model outputs
- raw benchmark run outputs
- VCD files
- simulator logs
- synthesis logs
- compiled simulation artifacts
- tool scratch files
- secrets or endpoint credentials
- model weights
- training datasets
- LoRA, QLoRA, or DoRA adapters
- fine-tuning scripts

## Non-Goals

Do not:

- change benchmark task content
- change prompts
- change reference RTL
- change testbenches
- change extractor policy
- change area/activity scoring formula
- compare models
- tune prompts
- create training data
- fine-tune a model
- claim measured power
- claim final silicon PPA

## Definition of Done

Done when:

1. The committed report set no longer leaves the old v0.6 `compile_failure` rows unqualified as candidate RTL failures.
2. A sanitized refreshed or companion report explains the post-bugfix tool-health interpretation.
3. Any regenerated CSV/JSONL remains sanitized and avoids raw artifacts.
4. `python -m pytest` passes.
5. No raw/private/model-output/training artifacts are committed.
