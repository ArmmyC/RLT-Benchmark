# Feature Spec: v0.6 RFID-APBench Post-Tool-Health Baseline Analysis

## Goal

Analyze the fresh post-tool-health v0.6 RFID-APBench 3-sample model baseline and decide the next benchmark iteration.

The fresh run `20260617T150443Z` used the restored healthy toolchain and produced a meaningful model baseline:

- 15/15 generation completed
- 12/15 extraction passed
- 12/15 candidate files available
- 11/15 compile passed
- 10/15 correctness passed
- 11/15 synthesis passed
- 10/15 valid scores
- mean valid score: 0.961155
- all-sample zero-filled mean: 0.640770
- failure distribution: 10 passed, 3 empty responses, 1 compile failure, 1 simulation failure

This stage should classify the remaining invalid rows, summarize task-level stability, and recommend exactly one next action.

This is benchmark analysis only. It is not prompt tuning, model comparison, fine-tuning, private evaluation, or task expansion implementation.

## Inputs

Read first:

- `reports/v0.6_rfid_apbench_3sample_baseline.md`
- `reports/v0.6_rfid_apbench_3sample_baseline.csv`
- `reports/v0.6_rfid_apbench_3sample_baseline.jsonl`
- `reports/v0.5_rfid_apbench_model_smoke.md`
- `reports/v0.5_rfid_apbench_closeout.md`
- `reports/v0.6_rfid_apbench_tool_health_refresh.md`
- `reports/v0.5_rfid_apbench_reference_validation.md`
- `reports/v0.5_rfid_apbench_candidate_smoke.md`

## Required Scope

Create a sanitized analysis report that:

1. Summarizes the fresh post-tool-health baseline outcome.
2. Confirms all five tasks and exactly three samples per task were attempted.
3. Compares the fresh run against the historical pre-tool-health v0.6 run and v0.5 one-sample smoke.
4. Summarizes per-task stability and per-task score behavior.
5. Classifies the three empty responses.
6. Classifies the one compile failure.
7. Classifies the one simulation failure.
8. Identifies likely root-cause categories using only sanitized information.
9. States whether benchmark assets should remain unchanged.
10. Recommends exactly one next action.

## Required Files

Create or update as appropriate:

- `reports/v0.6_rfid_apbench_post_tool_health_baseline_analysis.md`
- optional `reports/v0.6_rfid_apbench_post_tool_health_baseline_analysis.csv`
- optional pure helper script/tests only if helper code is introduced

Do not commit raw run directories, raw model responses, generated RTL, VCD files, simulator logs, synthesis logs, compiled artifacts, or tool scratch files.

## Analysis Requirements

The Markdown report must include:

- scope and non-goals
- run id, model, prompt profile, endpoint/tool status
- aggregate gate summary
- score summary
- per-task stability table
- per-sample invalid-row summary
- comparison to v0.5 one-sample smoke
- comparison to historical pre-tool-health v0.6 run
- root-cause assessment
- benchmark asset decision
- exactly one recommended next action
- activity proxy caveat
- public synthetic / not private / not fine-tuning caveat

## Allowed High-Level Failure Categories

Use sanitized categories such as:

- empty_response
- model_output_empty
- compile_failure
- likely_candidate_syntax_or_structure_issue
- simulation_failure
- likely_functional_or_protocol_issue
- extraction_failure
- benchmark_asset_issue_not_supported
- tool_health_issue_not_observed
- endpoint_instability_not_observed
- insufficient_evidence

## Allowed Local Inspection

You may inspect ignored local/output artifacts to classify failures at a high level, but committed text must stay sanitized.

Allowed committed descriptions:

- "empty response"
- "candidate compile failed under healthy tools"
- "candidate simulation failed under healthy tools"
- "failure appears model-output related"
- "insufficient evidence from sanitized artifacts"

Forbidden committed content:

- raw generated RTL
- raw model response text
- raw prompt text beyond public file references
- simulator log excerpts
- synthesis log excerpts
- VCD snippets
- absolute output paths exposing raw artifacts
- private text or private RTL

## Decision Rules

Recommend exactly one:

- `v0.6_task_expansion_spec` if benchmark assets are stable and the five-task baseline is good enough to expand to 10 tasks.
- `bounded_prompt_profile_experiment_spec` only if invalid rows are dominated by format/empty-response problems and task assets remain stable.
- `targeted_task_investigation_spec` only if a specific task appears benchmark-ambiguous or unstable.
- `extractor_bugfix_spec` only if extraction damage is supported.
- `evaluator_bugfix_spec` only if healthy-tool evaluation still appears incorrectly classified.
- `endpoint_stability_spec` only if request failures or transport errors dominate.
- `no_followup_needed` only if the analysis is already sufficient for current milestone closure.

Do not recommend fine-tuning from this analysis.

## Definition of Done

Done when:

1. A sanitized post-tool-health baseline analysis Markdown report exists.
2. The 10 valid rows and 5 invalid rows are summarized clearly.
3. Per-task stability and score behavior are explained.
4. Benchmark assets are either kept unchanged or a scoped issue is justified.
5. Exactly one next action is recommended.
6. `python -m pytest` passes.
7. No raw/private/model-output/training artifacts are committed.
