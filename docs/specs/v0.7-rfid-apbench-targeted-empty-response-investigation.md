# Feature Spec: v0.7 RFID-APBench Targeted Empty-Response Investigation

## Goal

Investigate the task-concentrated empty responses observed in the expanded v0.6 RFID-APBench baseline, without changing benchmark assets or running a benchmark-wide prompt experiment.

The v0.7 model-output robustness report recommends `targeted_empty_response_investigation_spec` because all six expanded-run empty responses were concentrated in two tasks:

- `ap_001_idle_counter`: 3/3 empty responses
- `ap_010_retry_timeout_fsm`: 3/3 empty responses

The investigation should determine, using sanitized evidence only, whether empty output appears repeatable and task-concentrated, or whether additional serving/runtime observability is needed before any prompt-profile experiment.

This is a targeted observability and analysis step. It is not prompt tuning, model comparison, benchmark modification, private evaluation, fine-tuning, or training-data creation.

## Inputs

Read first:

- `reports/v0.7_rfid_apbench_model_output_robustness.md`
- `reports/v0.6_rfid_apbench_expanded_baseline_analysis.md`
- `reports/v0.6_rfid_apbench_expanded_3sample_baseline.md`
- `reports/v0.6_rfid_apbench_expanded_3sample_baseline.csv`
- `reports/v0.6_rfid_apbench_expanded_3sample_baseline.jsonl`
- `reports/v0.6_rfid_apbench_expanded_closeout.md`
- `scripts/run_rfid_apbench_3sample_baseline.py`
- `src/rtlbench/tool_health.py`
- `benchmarks/rfid_apbench/tasks/ap_001_idle_counter/prompt.md`
- `benchmarks/rfid_apbench/tasks/ap_010_retry_timeout_fsm/prompt.md`

## Required Scope

Create a sanitized investigation report that:

1. Focuses only on `ap_001_idle_counter` and `ap_010_retry_timeout_fsm`.
2. Summarizes existing empty-response evidence from v0.6 and v0.7 robustness analysis.
3. Checks whether current committed sanitized fields can distinguish:
   - accepted request with empty text
   - request/transport failure
   - endpoint unavailable
   - timeout
   - tool or evaluator failure
4. Identifies missing sanitized observability fields, if any.
5. Decides whether a small targeted rerun is needed before prompt-profile changes.
6. Recommends exactly one next bounded step.
7. Keeps benchmark assets unchanged.
8. Keeps prompt tuning, model comparison, fine-tuning, and private evaluation out of scope.

## Optional Targeted Rerun

A small targeted rerun is allowed only if needed to characterize repeatability. If performed, it must be bounded to:

- tasks: `ap_001_idle_counter`, `ap_010_retry_timeout_fsm`
- samples: at most 3 additional samples per task
- model: configured/default model, expected `qwen36-27b` when available
- prompt profile/settings: unchanged existing default or `neutral_baseline`
- no prompt changes
- no task changes
- sanitized report outputs only

If the endpoint is unavailable or unstable, do not fake results. Emit sanitized blocker rows or state that the rerun was not performed.

## Required Files

Create:

- `reports/v0.7_rfid_apbench_targeted_empty_response_investigation.md`

Optional if a targeted rerun is performed:

- `reports/v0.7_rfid_apbench_targeted_empty_response_investigation.csv`
- `reports/v0.7_rfid_apbench_targeted_empty_response_investigation.jsonl`

Optional helper script/tests only if helper code is introduced.

Do not commit raw run directories, raw model responses, generated RTL, VCD files, simulator logs, synthesis logs, compiled artifacts, or tool scratch files.

## Report Requirements

The Markdown report must include:

- scope and non-goals
- source evidence used
- targeted task list
- existing empty-response summary
- optional targeted rerun summary if performed
- sanitized observability assessment
- missing observability fields, if any
- endpoint/tool/evaluator/benchmark assessment
- benchmark asset decision
- fine-tuning readiness decision
- exactly one recommended next step
- activity proxy caveat
- public synthetic / not private / not fine-tuning caveat

## Allowed High-Level Categories

Use sanitized categories such as:

- model_output_empty
- endpoint_instability_not_observed
- endpoint_or_serving_cause_insufficiently_observed
- request_failed
- timeout
- tool_health_issue_not_observed
- evaluator_bug_not_supported
- benchmark_asset_issue_not_supported
- insufficient_evidence

## Decision Rules

Recommend exactly one:

- `runner_or_endpoint_observability_spec` if current fields are insufficient to distinguish empty model output from accepted-request serving/runtime behavior.
- `bounded_prompt_profile_experiment_spec` only if repeatable empty output remains task-concentrated and observability is sufficient to treat it as a prompt/completion-reliability problem.
- `targeted_promptless_rerun_spec` if the evidence is insufficient and no targeted rerun was performed.
- `no_immediate_followup_beyond_documentation` only if the investigation is sufficient and no next experiment is warranted.

Do not recommend fine-tuning, private evaluation, model comparison, or benchmark modification from this investigation.

## Tests

```bash
python -m pytest
```

Add tests only if helper code is introduced. Do not require a live model endpoint in unit tests.

## Forbidden

Do not commit:

- private SiliconCraft RTL
- private task text
- raw prompts beyond public benchmark prompts
- raw model responses
- generated model RTL
- model outputs outside sanitized reports
- raw benchmark output directories
- VCD files
- simulator logs
- synthesis logs
- compiled simulation artifacts
- tool scratch files
- secrets or endpoint credentials
- model weights
- training datasets

## Non-Goals

Do not:

- add or modify benchmark tasks
- change prompts or prompt profiles
- run a benchmark-wide model baseline
- compare models
- tune prompts in this step
- create training data
- fine-tune a model
- add private benchmarks
- claim measured power
- claim final silicon PPA
- change area/activity scoring formula
- change evaluator policy unless a clear observability bug is found and separately specified

## Definition of Done

Done when:

1. `reports/v0.7_rfid_apbench_targeted_empty_response_investigation.md` exists.
2. It summarizes empty-response evidence for `ap_001` and `ap_010` clearly.
3. It states whether current sanitized observability is sufficient or insufficient.
4. It keeps benchmark assets unchanged unless strong evidence says otherwise.
5. It states fine-tuning is not recommended from this evidence alone.
6. It recommends exactly one next bounded step.
7. `python -m pytest` passes.
8. No raw/private/model-output/training artifacts are committed.
