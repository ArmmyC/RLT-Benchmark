# Bugfix Spec: v0.6 RFID-APBench Evaluator Tool Health Checks

## Goal

Fix the RFID-APBench candidate evaluator so it distinguishes tool path discovery from actual tool execution health.

The v0.6 3-sample failure analysis found that 10 extracted candidates were recorded as `compile_failure`, but bounded inspection showed both Icarus and Yosys terminated before parsing candidate RTL because of a missing runtime dependency. These rows should not be interpreted as confirmed candidate RTL compile errors.

This bugfix should add executable health checks and preserve a sanitized distinction between:

- tool unavailable or unhealthy
- tool startup/runtime failure
- candidate parse/compile failure
- candidate synthesis failure

Read first:

- `reports/v0.6_rfid_apbench_3sample_failure_analysis.md`

## Required Scope

Update the evaluator/tool handling so that:

1. Tool detection does not treat path discovery alone as proof of health.
2. Icarus Verilog health is checked by executing a lightweight version/help command or equivalent safe probe.
3. `vvp` health is checked similarly when simulation is required.
4. Yosys health is checked by executing a lightweight version/help command or equivalent safe probe.
5. Health-check failures are reported as sanitized tool health failures, not candidate compile failures.
6. Candidate compile failures are used only when the compile tool starts and returns a candidate-related failure.
7. Candidate synthesis failures are used only when the synthesis tool starts and returns a candidate-related failure.
8. Reports preserve enough sanitized detail to distinguish `tool_unavailable`, `tool_health_failed`, `tool_startup_failure`, `compile_failure`, and `synthesis_failure`.
9. Existing valid reference/candidate-fixture paths continue to work under a healthy toolchain.
10. Raw logs, generated RTL, paths exposing raw artifacts, and private data are not committed.

## Required Files

## Expected Behavior

### Tool path missing

If `iverilog`, `vvp`, or `yosys` path is missing:

- classify as `tool_unavailable`
- do not call candidate compile/synthesis failed

### Tool executable found but cannot start

If a tool path exists but the process fails before candidate parsing, such as missing runtime dependency:

- classify as `tool_health_failed` or `tool_startup_failure`
- include only sanitized notes such as `compile tool failed health check` or `synthesis tool failed health check`
- do not include raw stderr/log excerpts
- do not classify as candidate `compile_failure`

### Tool starts and candidate fails

If the tool starts normally and rejects the candidate RTL:

- classify as `compile_failure` or `synthesis_failure`
- keep notes sanitized
- do not commit raw candidate code or raw logs

### Healthy reference/candidate fixture

Reference validation and `reference_copy` candidate fixture evaluation should still pass under healthy tools.

## Report/Record Expectations

Sanitized rows should be able to represent:

- `tool_unavailable`
- `tool_health_failed`
- `tool_startup_failure`
- `compile_failure`
- `synthesis_failure`
- `simulation_failure`
- `candidate_missing`
- `passed`

Existing CSV/JSONL schemas may be extended if needed, but must remain backward-readable where practical.

## Tests

Run:

```bash
python -m pytest
```

Add tests that simulate or monkeypatch:

- missing tool path -> `tool_unavailable`
- discovered tool path but failed health check -> `tool_health_failed` or `tool_startup_failure`
- tool starts but candidate compile returns failure -> `compile_failure`
- report writers keep notes sanitized

Do not require a live model endpoint in unit tests.

## Definition of Done

Done when:

1. Tool path discovery and tool health are distinguished.
2. Tool startup/runtime failure is not reported as candidate compile failure.
3. Candidate compile/synthesis failures remain available when tools start correctly and candidate RTL fails.
4. Unit tests cover missing tools, unhealthy tools, and candidate failures.
5. `python -m pytest` passes.
6. No raw/private/model-output/training artifacts are committed.
