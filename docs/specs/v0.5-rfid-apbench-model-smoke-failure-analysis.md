# Feature Spec: v0.5 RFID-APBench Model Smoke Failure Analysis

## Goal

Analyze the first real RFID-APBench model-smoke result and prepare a bounded follow-up run plan without committing raw model outputs or training artifacts.

The first real model smoke completed generation for all five public/synthetic tasks, but only four candidates extracted and evaluated successfully. `ap_001_idle_counter` failed extraction, while the other four tasks produced valid `area_activity` scores.

This stage is benchmark analysis only. It is not fine-tuning.

## Inputs

Read first:

- `reports/v0.5_rfid_apbench_model_smoke.md`
- `reports/v0.5_rfid_apbench_model_smoke.csv`
- `reports/v0.5_rfid_apbench_model_smoke.jsonl`
- `scripts/run_rfid_apbench_model_smoke.py`
- `scripts/evaluate_rfid_apbench_candidates.py`
- `src/rtlbench/extraction.py`

## Required Scope

Implement a safe failure-analysis and bounded-retry planning pass that:

1. Summarizes the first model smoke results from sanitized reports.
2. Produces a failure taxonomy for extraction, compile, correctness, synthesis, area, activity, and score regressions.
3. Investigates why `ap_001_idle_counter` failed extraction using raw output only in ignored local/output paths.
4. Does not commit raw model responses, raw prompts, generated RTL, logs, VCDs, or output directory contents.
5. Identifies whether the failure is likely prompt formatting, model behavior, extraction logic, or task/module naming mismatch.
6. Adds a bounded retry plan for only the failed task or a small 5-task rerun.
7. Keeps model comparison, prompt tuning campaigns, and fine-tuning out of scope.

## Required Files

Create or update as appropriate:

- `reports/v0.5_rfid_apbench_model_smoke_analysis.md`
- `reports/v0.5_rfid_apbench_model_smoke_analysis.csv` if useful
- optional small script/helper for summarizing sanitized smoke reports
- optional tests for pure summary/helper functions

Do not commit raw run directories or generated RTL.

## Analysis Requirements

The Markdown analysis report must include:

- scope and non-goals
- model, task count, sample count, and run id
- generation/extraction/gate pass counts
- valid score count
- mean valid score
- all-sample mean score with invalid rows as zero
- per-task summary from sanitized fields
- failure taxonomy
- specific note for `ap_001_idle_counter` extraction failure
- score observations for the four valid tasks
- recommended bounded retry plan
- caveat that activity is a VCD toggle-count proxy, not measured power
- caveat that this is public synthetic smoke, not private SiliconCraft evaluation and not fine-tuning

## Bounded Retry Plan

Recommend one of these:

### Option A: targeted failed-task retry

Run `ap_001_idle_counter` only with the same model and settings for 3 samples, then evaluate extracted candidates through the existing candidate evaluator.

### Option B: exact 5-task rerun

Run all five tasks again with one sample each if the raw failure suggests non-deterministic formatting.

Do not run broad model comparisons or prompt campaigns in this stage.

## Sanitization Rules

Committed analysis may include only sanitized fields already present in the Markdown/CSV/JSONL reports, plus human-written summaries.

Do not commit:

- raw prompts
- raw model responses
- generated RTL
- output paths exposing raw artifacts
- VCD contents
- simulator logs
- synthesis logs
- secrets or endpoint credentials
- private RTL
- private task text
- model weights
- training datasets
- LoRA, QLoRA, or DoRA adapters
- fine-tuning scripts

## Tests

Run:

```bash
python -m pytest
```

Add tests only for pure helpers if helper code is introduced. Do not require live model endpoints in unit tests.

## Non-Goals

Do not:

- fine-tune a model
- create training data
- create a fine-tuning repository
- compare multiple models
- run large sample counts
- tune prompts broadly
- claim measured power
- claim final silicon PPA
- integrate private benchmarks

## Definition of Done

Done when:

1. A sanitized model-smoke analysis report exists.
2. The extraction failure is categorized without committing raw output.
3. Score observations for valid tasks are summarized.
4. A bounded retry plan is documented.
5. `python -m pytest` passes.
6. No raw/private/model-output/training artifacts are committed.
