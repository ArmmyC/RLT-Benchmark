## 4. Next Stage

### Stage G: Candidate-selection report and larger-run plan

Status: next.

Create a human-readable decision report that summarizes the 12-row smoke matrix, selects `neutral_baseline` and `strict_rtl_only` as larger-run candidates, and defines the next larger validation slice.

Suggested larger validation slice, to be planned but not automatically executed:

- VerilogEval: 20 tasks, pass@1, candidate profiles only
- RTLLM: 10 tasks, pass@1, candidate profiles only
- RTL-OPT equivalence: 10 tasks, candidate profiles only

The exact limits can be adjusted in the report, but the plan must stay smaller than a full benchmark.

### Stage H: Larger validation execution

Only after Stage G is reviewed, run the selected larger validation slice. Do not run it in the Stage G task.

## 5. Required Artifacts

Add:

- `reports/v0.3_qwen36_27b_prompt_candidate_decision.md`
- `reports/v0.3_qwen36_27b_prompt_candidate_decision.csv`

The Markdown report should include:

- smoke result table by benchmark and profile
- combined pass counts across all smoke benchmarks
- selected candidates
- rejected or held-back profiles with reasons
- recommended larger validation plan
- explicit note that smoke results are too small for final conclusions

The CSV should include one row per prompt profile with at least:

- prompt_profile
- verilogeval_passed
- rtllm2_passed
- rtlopt_passed
- total_passed
- total_samples
- decision
- reason

## 6. Commit Rules

Allowed:

- candidate decision reports
- documentation/spec updates
- scripts/tests only if needed to generate the decision report cleanly

Forbidden:

- `outputs/`
- `raw_responses/`
- `extracted_rtl/`
- `logs/`
- `error_logs/`
- generated RTL code
- model responses
- prompts containing private tasks
- API keys or secrets
- caches, virtual environments, model weights

## 7. Non-goals

Do not:

- run the larger validation slice yet
- run full benchmarks
- add new models
- change benchmark logic
- change evaluator behavior
- modify Baseline v0.1 or v0.2 artifacts
- implement LANTA serving, SSH, Slurm swapping, OpenWebUI, or LiteLLM logic
- claim final PPA or functional superiority from smoke results alone

## 8. Definition of Done for Next Codex Task

The next Codex task is done when:

1. Candidate decision Markdown report exists.
2. Candidate decision CSV exists.
3. The report selects `neutral_baseline` and `strict_rtl_only` for larger validation.
4. The report holds back `low_area_low_power` and `no_explanation_code_only` with reasons.
5. The report includes a larger validation plan but does not execute it.
6. `python -m pytest` passes.
7. No raw experiment outputs are committed.
