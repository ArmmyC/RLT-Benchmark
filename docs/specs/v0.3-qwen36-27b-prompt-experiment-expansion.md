# Feature Spec: v0.3 qwen36-27b Prompt Experiment Expansion

## 1. Goal

Use the completed v0.3 larger-validation results to produce a final prompt decision report for `qwen36-27b`.

Smoke testing, candidate selection, larger-validation scaffolding, and larger-validation execution are complete. The larger validation compared:

- `neutral_baseline`
- `strict_rtl_only`

The next step is not another run. The next step is a decision report that explains whether to keep `neutral_baseline`, switch to `strict_rtl_only`, or expand further.

## 2. Completed Larger Validation Results

### VerilogEval, 20 tasks per profile

- `neutral_baseline`: 19/20 functional pass, 19/20 syntax/compile pass, 1 compile failure
- `strict_rtl_only`: 20/20 functional pass, 20/20 syntax/compile pass

### RTLLM, 10 tasks per profile

- `neutral_baseline`: 4/10 functional pass, 5/10 syntax/compile pass, 5 compile failures, 1 simulation failure
- `strict_rtl_only`: 4/10 functional pass, 5/10 syntax/compile pass, 5 compile failures, 1 simulation failure

### RTL-OPT equivalence, 10 tasks per profile

- `neutral_baseline`: 8/10 equivalence pass, 10/10 syntax/compile pass, 2 equivalence failures
- `strict_rtl_only`: 7/10 equivalence pass, 10/10 syntax/compile pass, 3 equivalence failures

### Combined larger-validation totals

- `neutral_baseline`: 31/40 final pass
- `strict_rtl_only`: 31/40 final pass

The larger validation does not produce a clear overall winner. `strict_rtl_only` is slightly better on VerilogEval, while `neutral_baseline` is slightly better on RTL-OPT equivalence. RTLLM is tied.

## 3. Current Interpretation

Recommended provisional default:

- Keep `neutral_baseline` as the default prompt profile for now.

Reason:

- It remains the historical/control prompt.
- It ties `strict_rtl_only` overall at 31/40.
- It performs slightly better on RTL-OPT equivalence, which is important for optimization-oriented work.

Recommended optional challenger:

- Keep `strict_rtl_only` as a challenger profile for future experiments.

Reason:

- It improves the VerilogEval larger slice by 1 task.
- It has stricter formatting behavior.
- It does not beat `neutral_baseline` overall yet.

Continue holding back:

- `low_area_low_power`
- `no_explanation_code_only`

## 4. Next Stage

### Stage I: Larger-validation decision report

Status: next.

Create a final v0.3 decision report that summarizes smoke and larger-validation results, explains the tied larger-validation outcome, and recommends a conservative default.

## 6. Commit Rules

Allowed:

- larger decision reports
- documentation/spec updates only if needed
- tests or helper fixes only if needed

Forbidden:

- `outputs/`
- `raw_responses/`
- `extracted_rtl/`
- `logs/`
- `error_logs/`
- generated RTL code
- model responses
- prompts containing private tasks
- API keys or secrets
- caches, virtual environments, model weights

## 7. Non-goals

Do not:

- run more benchmarks
- run full benchmarks
- add new models
- change benchmark logic
- change evaluator behavior
- modify Baseline v0.1 or v0.2 artifacts
- implement LANTA serving, SSH, Slurm swapping, OpenWebUI, or LiteLLM logic
- claim final silicon PPA quality
