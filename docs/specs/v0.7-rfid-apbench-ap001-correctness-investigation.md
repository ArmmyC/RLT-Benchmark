# Feature Spec: v0.7 RFID-APBench ap_001 Correctness Investigation

## Goal

Investigate the persistent `ap_001_idle_counter` functional simulation failures now that the response-boundary/null-content issue is resolved by forwarding `chat_template_kwargs.enable_thinking=false`.

The v0.7 thinking-disabled closeout recommends `ap001_correctness_investigation_spec` because `ap_001_idle_counter` produced three interpretable non-empty candidates that extracted, compiled, and synthesized but failed functional simulation. This is a candidate correctness problem, not an empty-response recurrence.

This investigation should classify the correctness failure pattern using sanitized evidence only. It must not change benchmark assets, task prompts, evaluator policy, scoring, model, token limit, or prompt profile.

Read first:

- `reports/v0.7_rfid_apbench_thinking_disabled_closeout.md`
- `reports/v0.7_rfid_apbench_targeted_3sample_thinking_disabled_validation.md`
- `reports/v0.7_rfid_apbench_targeted_3sample_thinking_disabled_validation.csv`

- `benchmarks/rfid_apbench/tasks/ap_001_idle_counter/prompt.md`
- `benchmarks/rfid_apbench/tasks/ap_001_idle_counter/reference.sv`

- `scripts/evaluate_rfid_apbench_candidates.py`

## Required Scope

Create a sanitized correctness investigation report for `ap_001_idle_counter` only.

The report should:

1. Summarize the post-fix `ap_001_idle_counter` evidence: non-empty generation, extraction pass, compile pass, synthesis pass, simulation failure.
2. Confirm this is not an empty-response, endpoint, token-limit, extractor, synthesis, or benchmark asset issue based on committed evidence.
3. Inspect the public task specification, reference, and testbench to identify the expected behavior at a high level.
4. If allowed by available sanitized artifacts, classify the model-candidate failure pattern without committing raw generated RTL or raw simulator logs.
5. If raw generated candidate/log inspection is needed locally, keep it ephemeral/uncommitted and report only sanitized categories.
6. Decide whether more model samples, a task-specific analysis note, or a broader fixed-run baseline is the next bounded step.
7. Keep benchmark assets unchanged unless there is strong evidence of a public benchmark bug.
8. Keep fine-tuning out of scope and not recommended.
9. Recommend exactly one next bounded direction.

## Optional Bounded Diagnostic

A live diagnostic is allowed only if it is narrowly scoped and sanitized:

- task: `ap_001_idle_counter` only
- profile: `neutral_baseline`
- model: configured/default `qwen36-27b`
- max tokens: unchanged `4096`
- thinking disabled: use fixed runner path
- samples: at most 3 additional samples
- no benchmark-wide run
- no model comparison
- no prompt/profile/task changes
- no committed raw model output, generated RTL, logs, VCDs, or compiled artifacts

If performed, commit only sanitized CSV/JSONL/Markdown rows and failure categories.

## Report Requirements

The Markdown report must include:

- scope and non-goals
- source evidence used
- post-fix `ap_001` evidence summary
- public expected-behavior summary from task/reference/testbench
- failure classification taxonomy
- classification of observed `ap_001` failures, if possible
- whether a task, evaluator, extractor, synthesis, endpoint, or prompt/profile defect is supported
- benchmark asset decision
- fine-tuning readiness decision
- exactly one recommended next bounded direction
- activity proxy caveat
- public synthetic / no private / not fine-tuning caveat

## Allowed Sanitized Failure Categories

Use these categories when applicable:

- `state_update_timing_mismatch`
- `enable_or_idle_behavior_mismatch`
- `reset_behavior_mismatch`
- `counter_width_or_wrap_mismatch`
- `output_latency_mismatch`
- `incomplete_or_malformed_candidate`
- `testbench_expectation_not_met`
- `benchmark_asset_issue_not_supported`
- `extractor_issue_not_supported`
- `tool_or_synthesis_issue_not_supported`
- `insufficient_sanitized_evidence`

## Decision Rules

Recommend exactly one:

- `ap001_targeted_3sample_retry_or_analysis_spec` if more sanitized `ap_001` samples are needed to classify the pattern.
- `post_fix_expanded_10task_baseline_spec` if the correctness issue is understood enough and should not block a fixed-run 10-task baseline.
- `ap001_prompt_or_task_clarity_spec` only if public task/prompt ambiguity is strongly supported without changing reference behavior.
- `benchmark_bugfix_spec` only if a concrete benchmark/testbench/reference bug is found.
- `no_immediate_followup_beyond_documentation` if no further bounded work is justified.

Do not recommend fine-tuning, private evaluation, model comparison, broad benchmark rerun before this classification, max-token increase, or additional prompt-profile experiments.

- add or modify benchmark tasks
- edit benchmark task prompt files
- change reference RTL or testbench behavior

- raise the global max-token default
- create training data

- add private benchmarks
- claim measured power
- claim final silicon PPA
- change area/activity scoring formula
- change evaluator policy

## Definition of Done

Done when:

1. `reports/v0.7_rfid_apbench_ap001_correctness_investigation.md` exists.
2. It classifies or clearly states the limits of classifying the `ap_001` simulation failures.
3. It keeps response reliability separate from functional correctness.
4. It states whether benchmark asset issues are supported.
5. It keeps benchmark assets unchanged unless a concrete bug is found.
6. It states fine-tuning is not recommended from this evidence alone.
7. It recommends exactly one next bounded direction.
8. `python -m pytest` passes.
9. No raw/private/model-output/training artifacts are committed.
