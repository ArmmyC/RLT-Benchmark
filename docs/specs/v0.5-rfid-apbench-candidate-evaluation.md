# Feature Spec: v0.5 RFID-APBench Candidate Evaluation

## Goal

Extend RFID-APBench from reference validation to generated/candidate RTL evaluation using public synthetic candidate fixtures first.

This stage should prove the benchmark can compare a candidate RTL implementation against the validated reference on correctness, area, and VCD toggle-count activity before any model job is run.

## Inputs

Read first:

- `docs/specs/v0.5-rfid-apbench-toolchain-ci-validation.md`
- `reports/v0.5_rfid_apbench_reference_validation.md`
- `reports/v0.5_area_activity_scoring_design.md`
- `src/rtlbench/area_activity_scoring.py`
- `src/rtlbench/vcd_activity.py`
- `scripts/validate_rfid_apbench_references.py`

## Required Scope

Implement a candidate-evaluation path that:

1. Loads RFID-APBench tasks through the adapter.
2. Accepts candidate RTL files from a public synthetic fixture directory or explicit CLI path.
3. Compiles and simulates each candidate against the task testbench.
4. Counts candidate VCD activity using the same workload/window as the reference.
5. Synthesizes candidate RTL with the same generic Yosys flow used for references.
6. Loads committed `expected/reference_metrics.yaml` for reference area/activity.
7. Computes `area_activity` scores with `src/rtlbench/area_activity_scoring.py`.
8. Emits sanitized JSONL/CSV/Markdown candidate reports.
9. Keeps raw VCDs, compiled artifacts, logs, and synthesis scratch files under `.tmp/` only.

## Required Files

Create or update as appropriate:

- `scripts/evaluate_rfid_apbench_candidates.py`
- `benchmarks/rfid_apbench/candidates/README.md`
- `benchmarks/rfid_apbench/candidates/reference_copy/` or another public synthetic candidate fixture set
- `reports/v0.5_rfid_apbench_candidate_smoke.md`
- `reports/v0.5_rfid_apbench_candidate_smoke.csv`
- `reports/v0.5_rfid_apbench_candidate_smoke.jsonl`
- tests for pure helper functions and sanitized output behavior

Candidate fixtures must be public synthetic RTL only. They may copy or lightly vary the committed reference RTL for smoke testing. They must not be model outputs.

## CLI Shape

The evaluator should support a command similar to:

```bash
python scripts/evaluate_rfid_apbench_candidates.py \
  --benchmark-root benchmarks/rfid_apbench \
  --candidate-root benchmarks/rfid_apbench/candidates/reference_copy \
  --work-dir .tmp/rfid_apbench_candidate_smoke \
  --output-md reports/v0.5_rfid_apbench_candidate_smoke.md \
  --output-csv reports/v0.5_rfid_apbench_candidate_smoke.csv \
  --output-jsonl reports/v0.5_rfid_apbench_candidate_smoke.jsonl
```

## Candidate Mapping

Support one candidate file per task, for example:

```text
benchmarks/rfid_apbench/candidates/reference_copy/<task_id>.sv
```

A missing candidate should be recorded as an invalid sample with a clear failure category, not crash the whole run.

## Scoring Requirements

For each candidate row, record:

- task_id
- candidate_id
- final_pass
- extraction_pass or candidate_file_available
- compile_pass
- correctness_pass
- synth_pass
- timing_status
- reference_area
- generated_area
- area_unit
- reference_activity
- generated_activity
- activity_metric
- area_score
- activity_score
- score
- score_status
- failure_category
- toolchain_id
- workload_id

Invalid samples must have null per-sample score and contribute zero in aggregate summaries.

## Report Requirements

Markdown report must include:

- scope and non-goals
- tool availability summary
- candidate fixture source
- aggregate gate pass counts
- valid and all-sample scores
- per-task candidate table
- caveat that activity is VCD toggle-count proxy, not measured power
- caveat that this is public synthetic candidate validation, not model performance

## Tests

Run:

```bash
python -m pytest
```

Add tests for:

- candidate mapping and missing candidate behavior
- sanitized row construction
- score calculation using reference metrics and candidate metrics
- report helpers if introduced

Tool-dependent integration may live in CI or be skipped locally when tools are unavailable.

## Forbidden

Do not commit:

- private SiliconCraft RTL
- private task text
- raw prompts
- raw model responses
- model-generated RTL
- model outputs
- raw benchmark run outputs
- VCD files
- simulator logs
- synthesis logs
- compiled simulation artifacts
- tool scratch files
- secrets
- model weights
- training datasets
- LoRA, QLoRA, or DoRA adapters
- fine-tuning scripts

## Non-Goals

Do not:

- run LLM/model evaluations
- compare models or prompt profiles
- create training data
- create a fine-tuning repository
- claim measured power
- claim final silicon PPA
- integrate private benchmarks

## Definition of Done

Done when:

1. Candidate evaluator script exists.
2. Public synthetic candidate fixtures exist.
3. Candidate evaluator uses reference metrics and computes `area_activity` scores.
4. Sanitized Markdown/CSV/JSONL smoke reports exist.
5. Missing/invalid candidates are reported cleanly.
6. `python -m pytest` passes.
7. No raw/private/model/training artifacts are committed.
8. Report clearly states this is candidate-fixture validation, not model performance.
