# Feature Spec: v0.4 Stage K Conservative Area Profile

## 1. Goal

Create and test a safer low-area prompt profile after the first matched `low_area_low_power` retest failed to improve paired-valid tasks and reduced correctness coverage.

Stage J showed that the existing `low_area_low_power` profile matched neutral on the two paired-valid tasks, improved zero tasks, and had one fewer valid correctness-gated PPA record than `neutral_baseline`. The next step is not a larger benchmark. The next step is a small prompt-iteration experiment focused on preserving correctness while allowing conservative area-oriented choices.

This remains a public-flow generic-cell proxy experiment. It is not final silicon PPA and has no measured power data.

## 2. Completed Evidence

Stage J artifacts:

- `artifacts/v0.4/ppa_matched/rtlopt_low_area_low_power_ppa.jsonl`
- `reports/v0.4_ppa_low_area_summary.md`
- `reports/v0.4_ppa_low_area_summary.csv`
- `reports/v0.4_ppa_neutral_vs_low_area_matched.md`
- `reports/v0.4_ppa_neutral_vs_low_area_matched.csv`

Stage J result:

- neutral valid PPA records: 3/5
- low-area valid PPA records: 2/5
- paired-valid tasks: 2
- improved paired tasks: 0
- matched paired tasks: 2
- regressed paired tasks: 0
- low-area lost validity on `add_sub`

Interpretation:

- The existing `low_area_low_power` prompt is not good enough to scale up.
- The next prompt should prioritize correctness and exact interface preservation more strongly.
- Area reduction should be conservative and only applied when it is structurally safe.

## 3. Next Stage

### Stage K: Conservative area prompt profile

Status: next.

Add a new prompt profile, then run it on the same five-task public RTL-OPT slice and compare against both:

- fixed `neutral_baseline`
- existing `low_area_low_power`

Suggested profile name:

- `conservative_area`

## 4. Prompt Design Requirements

The new profile should instruct the model to:

- preserve module name and port interface exactly
- prioritize functional correctness and synthesizability above optimization
- avoid changing latency, protocol behavior, reset behavior, or observable timing unless the task explicitly allows it
- prefer simple combinational simplifications, constant folding, common-subexpression sharing, mux simplification, and removing redundant logic
- avoid aggressive serialization, FSM rewrites, resource sharing, or register removal when it could change behavior
- treat timing as a constraint, not a score target
- not claim exact PPA improvement without synthesis reports
- return RTL code only

Do not modify the existing `low_area_low_power` profile because Stage J artifacts must remain reproducible.

## 7. Comparison Rules

The report must include:

- task IDs
- per-task validity for all three profiles
- per-task equivalence status for all three profiles
- generated generic cells where valid
- area-only score where valid
- valid/invalid counts for each profile
- equivalence pass count for each profile
- paired comparison against neutral for tasks valid under both profiles
- whether `conservative_area` improves, matches, or regresses versus neutral
- whether `conservative_area` improves validity versus `low_area_low_power`
- caveat that `generic_cells` is a proxy, not final silicon area
- caveat that power is unavailable

Do not compute score deltas for invalid or unmatched tasks.

## 8. Success Criteria

Stage K succeeds if:

- new profile is added without changing existing profile text
- Yosys preflight passes
- conservative-area run completes on the same five tasks
- sanitized PPA JSONL exists
- summary Markdown/CSV exists
- three-way comparison Markdown/CSV exists
- no raw/private fields are committed
- report gives a clear recommendation: keep, revise, or reject `conservative_area`

If no paired-valid tasks exist, the stage is still complete if the report clearly explains the blocker.

## 9. Commit Rules

Allowed:

- `configs/prompt_profiles.yaml`
- sanitized PPA JSONL artifacts under `artifacts/v0.4/ppa_matched/`
- Markdown/CSV reports
- docs/spec updates only if needed
- small report/export helper fixes only if needed

Forbidden:

- `outputs/`
- `raw_responses/`
- `extracted_rtl/`
- `logs/`
- `error_logs/`
- generated RTL code from model runs
- model responses
- prompts containing private tasks
- API keys or secrets
- caches, virtual environments, model weights

## 10. Non-goals

Do not:

- run full RTL-OPT
- run VerilogEval, RTLLM, or ProtocolLLM
- run `strict_rtl_only`
- run new models
- integrate SiliconCraft private benchmarks
- change v0.1/v0.2 frozen artifacts
- change v0.3 reports
- claim measured power improvement
- claim final silicon PPA quality

## 11. Definition of Done

Done when:

1. `conservative_area` prompt profile exists.
2. Preflight result is documented.
3. Conservative-area matched run is complete, or preflight blocks it with exact failure.
4. Sanitized conservative-area PPA JSONL exists if the run executes.
5. Conservative-area summary exists if the run executes.
6. Three-way comparison report exists.
7. Report states whether `conservative_area` should be kept, revised, or rejected.
8. `python -m pytest` passes.
9. No raw outputs or model responses are committed.
