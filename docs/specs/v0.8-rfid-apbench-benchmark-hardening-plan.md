# Feature Spec: v0.8 RFID-APBench Benchmark Hardening Plan

## Goal

Plan the v0.8 RFID-APBench hardening cycle after the v0.7 public/synthetic post-fix baseline closeout.

The v0.7 release closeout concludes that v0.7 is ready to tag and release as a public/synthetic RFID-APBench post-fix baseline. The next cycle should harden benchmark documentation, reproducibility checks, report hygiene, and release automation around the closed v0.7 evidence.

This is planning only. It is not a model evaluation, model comparison, prompt-profile experiment, private benchmark effort, training-data creation, fine-tuning, or benchmark behavior change.

## Inputs

Read first:

- `reports/v0.7_rfid_apbench_release_closeout.md`
- `reports/v0.7_rfid_apbench_post_fix_expanded_10task_baseline.md`
- `reports/v0.7_rfid_apbench_post_fix_failure_analysis.md`
- `reports/v0.7_rfid_apbench_ap001_post_clarity_targeted_validation.md`
- `reports/v0.7_rfid_apbench_ap001_prompt_clarity.md`
- `benchmarks/rfid_apbench/manifest.yaml`
- `benchmarks/rfid_apbench/README.md`
- `README.md`
- `configs/models.yaml`
- `configs/prompt_profiles.yaml`
- `scripts/run_rfid_apbench_3sample_baseline.py`
- `scripts/evaluate_rfid_apbench_candidates.py`
- current tests under `tests/`

## Required Scope

Create a v0.8 planning document that defines hardening work items for:

1. Benchmark documentation: public/synthetic scope, task inventory, scoring mode, activity-proxy caveat, and known limitations.
2. Reproducibility: endpoint/toolchain requirements, `.env`/environment handling, deterministic settings, run commands, expected output locations, and ignored raw-artifact policy.
3. Report hygiene: stable report filenames, schema expectations, report-to-report lineage, sanitized artifact rules, and release closeout pointers.
4. Release automation: suggested tag process, pre-tag checklist, optional release checklist script or CI assertions.
5. Validation guardrails: exactly-three-sample baseline behavior, targeted validation exceptions, task-filter behavior, model preset `extra_body` forwarding, and no-raw-output guarantees.
6. Known limitation tracking: `ap_006_wakeup_edge_filter` candidate limitation and how it should be carried forward without blocking v0.7 release.
7. Future work boundaries: what belongs in RTLBench versus future separate fine-tuning repositories.

The plan must separate immediate v0.8 hardening tasks from future/non-goal work.

## Required Files

Create:

- `benchmarks/rfid_apbench/README.md`
- `README.md`

Do not run a model evaluation in this spec.

## Report Requirements

The Markdown report must include:

- scope and non-goals
- source evidence used
- v0.7 release state summary
- hardening workstream list
- immediate v0.8 task recommendations
- release/tag checklist proposal
- reproducibility checklist proposal
- report hygiene checklist proposal
- benchmark asset decision
- fine-tuning readiness decision
- exactly one recommended next bounded direction
- activity proxy caveat
- public synthetic / no private / not fine-tuning caveat

## Decision Rules

Recommend exactly one:

- `v0.8_release_checklist_spec` if the next step should implement a concrete pre-tag/release checklist.
- `v0.8_report_hygiene_spec` if report schema/lineage cleanup should happen first.
- `v0.8_reproducibility_docs_spec` if user-facing run/reproduce docs are the highest priority.
- `v0.8_ci_release_assertions_spec` if automated checks should happen first.
- `no_immediate_followup_beyond_documentation` if the planning report is sufficient.

Do not recommend fine-tuning, private evaluation, model comparison, global max-token changes, prompt-profile experiments, or training-data creation from this planning step.

## Tests

Run:

```bash
python -m pytest
```

Add tests only if helper code is introduced. Do not require a live model endpoint in unit tests.

## Forbidden

Do not commit:

- private SiliconCraft RTL
- private task text
- raw prompts beyond public benchmark prompt text
- raw model responses
- response bodies
- generated model RTL
- model outputs outside sanitized reports
- raw benchmark output directories
- VCD files
- simulator logs
- synthesis logs
- compiled artifacts
- tool scratch files
- secrets or endpoint credentials
- model weights
- training datasets
- LoRA, QLoRA, or DoRA adapters
- fine-tuning scripts

## Non-Goals

Do not:

- run another model evaluation
- add or modify benchmark tasks
- edit reference RTL or testbench behavior
- change module interfaces
- compare models
- change prompt profiles
- raise global max-token default
- create training data
- fine-tune a model
- add private benchmarks
- claim measured power
- claim final silicon PPA
- change workload, metrics, area/activity scoring, or evaluator policy

## Definition of Done

Done when:

1. `reports/v0.8_rfid_apbench_benchmark_hardening_plan.md` exists.
2. It summarizes v0.7 release state and known limitations.
3. It defines concrete v0.8 hardening workstreams.
4. It proposes release, reproducibility, and report-hygiene checklists.
5. It states benchmark assets remain unchanged.
6. It states fine-tuning is not recommended from this evidence alone.
7. It recommends exactly one next bounded direction.
8. `python -m pytest` passes.
9. No raw/private/model-output/training artifacts are committed.
