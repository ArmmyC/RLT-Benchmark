# Feature Spec: v0.6 RFID-APBench 3-Sample Baseline Failure Analysis

## Goal

Analyze the completed v0.6 RFID-APBench 3-sample baseline results after the local endpoint became available and all 15 samples completed generation.

The baseline produced:

- 15/15 generation completed
- 10/15 extraction passed
- 10/15 candidate files available
- 0/15 compile passed
- 0/15 correctness passed
- 0/15 synthesis passed
- 0 valid scores
- failure distribution: 10 compile failures, 4 empty responses, 1 extraction failure

This stage should classify the failures at a high level and decide whether the next action should be benchmark-asset change, extractor/evaluator bugfix, endpoint/model-output investigation, or prompt/model evaluation planning.

This is benchmark analysis only. It is not prompt tuning, fine-tuning, private evaluation, or model comparison.

## Required Scope

Create a sanitized analysis pass that:

1. Summarizes the v0.6 3-sample baseline outcome.
2. Confirms all five tasks and exactly three samples per task were attempted.
3. Compares the v0.6 result against the v0.5 one-sample smoke.
4. Classifies the 10 compile failures into high-level categories using only sanitized information.
5. Classifies the 4 empty responses and 1 extraction failure.
6. Identifies whether the failures suggest model-output quality, extraction damage, benchmark ambiguity, evaluator/tool issue, or endpoint instability.
7. Recommends exactly one next action.
8. Does not modify benchmark tasks, prompts, references, testbenches, extractor, evaluator, scoring policy, or model settings.

Do not commit raw run directories, raw model responses, generated RTL, VCD files, simulator logs, synthesis logs, compiled artifacts, or tool scratch files.

## Analysis Requirements

The Markdown report must include:

- scope and non-goals
- model, endpoint/tool status, task count, sample count, run id
- aggregate gate summary
- per-task failure summary
- comparison to v0.5 one-sample smoke
- high-level compile-failure taxonomy
- empty-response/extraction-failure summary
- likely root-cause category
- recommended next action and rationale
- clear statement on whether benchmark assets should remain unchanged
- activity proxy caveat
- public synthetic / not private / not fine-tuning caveat

## Allowed High-Level Failure Categories

Use sanitized categories such as:

- empty_response
- incomplete_rtl
- malformed_module_boundary
- missing_port_or_wrong_port_list
- illegal_or_non_synthesizable_construct
- syntax_parse_error
- duplicate_or_extra_module_text
- natural_language_leakage_into_code
- interface_or_top_module_mismatch
- extraction_failure
- compile_failure_insufficient_detail
- possible_extractor_damage
- likely_model_output_quality
- possible_endpoint_instability
- insufficient_evidence

## Allowed Local Inspection

You may inspect ignored local/output artifacts to classify failures at a high level, but committed text must stay sanitized.

Allowed committed descriptions:

- "missing or mismatched port list"
- "natural-language text remained in extracted RTL"
- "module boundary incomplete"
- "syntax parse failure before simulation"
- "non-synthesizable or illegal procedural construct"
- "candidate appears structurally incomplete"
- "insufficient evidence from sanitized artifacts"

Forbidden committed content:

- raw generated RTL
- raw model response text
- raw prompt text beyond public file references
- simulator log excerpts
- synthesis log excerpts
- VCD snippets
- absolute output paths exposing raw artifacts
- private text or private RTL

## Decision Rules

Recommend:

- `keep_benchmark_unchanged` if public references/testbenches remain validated and failures appear model-output related.
- `extractor_bugfix_spec` only if extracted RTL is likely damaged/truncated by extraction despite usable model output.
- `evaluator_tool_bugfix_spec` only if compile failures are caused by the evaluator/tool invocation rather than generated candidates.
- `prompt_spec_clarification_spec` only if public task prompts are ambiguous in a way that likely caused repeated malformed outputs.
- `endpoint_stability_spec` only if empty responses or connection artifacts dominate and prevent fair generation.
- `bounded_prompt_profile_experiment_spec` only if benchmark assets are stable and failures are primarily model-output formatting/compile quality.

Do not recommend fine-tuning from this analysis.

3. The report states whether benchmark assets remain unchanged or need a scoped follow-up.

5. `python -m pytest` passes.
6. No raw/private/model-output/training artifacts are committed.
