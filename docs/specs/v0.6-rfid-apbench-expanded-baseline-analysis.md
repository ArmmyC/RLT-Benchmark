# Feature Spec: v0.6 RFID-APBench Expanded Baseline Analysis

## Goal

Analyze the expanded 10-task by 3-sample RFID-APBench model baseline and decide the next benchmark-planning step.

The fresh expanded run `20260617T164603Z` produced:

- 30/30 generation completed
- 23/30 extraction passed
- 23/30 candidate files available
- 23/30 compile passed
- 23/30 correctness passed
- 22/30 synthesis passed
- 22/30 valid scores
- mean valid score: 1.009141
- all-sample zero-filled mean: 0.740037
- failure distribution: 22 passed, 6 empty responses, 1 extraction failure, 1 synthesis failure

This analysis should summarize the expanded result, classify invalid rows, identify stable and weak tasks, and recommend exactly one next action.

This is benchmark analysis only. It is not model comparison, prompt tuning, fine-tuning, private evaluation, or training-data creation.

Read first:

## Required Scope

Create a sanitized analysis report that:

1. Summarizes the expanded 10-task baseline outcome.
2. Confirms all 10 manifest tasks and exactly three samples per task were attempted.
3. Compares against the five-task post-tool-health baseline without making a model-comparison claim across different task sets.
4. Summarizes per-task stability and score behavior.
5. Classifies the six empty responses.
6. Classifies the one extraction failure.
7. Classifies the one synthesis failure.
8. Identifies likely root-cause categories using only sanitized information.
9. States whether benchmark assets should remain unchanged.
10. Recommends exactly one next action.

## Required Files

Create or update as appropriate:

- `reports/v0.6_rfid_apbench_expanded_baseline_analysis.md`
- optional `reports/v0.6_rfid_apbench_expanded_baseline_analysis.csv`
- optional pure helper script/tests only if helper code is introduced

Do not commit raw run directories, raw model responses, generated RTL, VCD files, simulator logs, synthesis logs, compiled artifacts, or tool scratch files.

## Analysis Requirements

The Markdown report must include:

- scope and non-goals
- run id, model, prompt profile, endpoint/tool status
- aggregate gate summary
- score summary
- per-task stability table
- per-sample invalid-row summary
- comparison to the five-task post-tool-health baseline
- root-cause assessment
- benchmark asset decision
- exactly one recommended next action
- activity proxy caveat
- public synthetic / not private / not fine-tuning caveat

## Allowed High-Level Failure Categories

Use sanitized categories such as:

- empty_response
- model_output_empty
- extraction_failure
- likely_incomplete_or_malformed_response
- synthesis_failure
- likely_candidate_synthesis_issue
- benchmark_asset_issue_not_supported
- tool_health_issue_not_observed
- endpoint_instability_not_observed
- insufficient_evidence

## Allowed Local Inspection

You may inspect ignored local/output artifacts to classify failures at a high level, but committed text must stay sanitized.

Allowed committed descriptions:

- "empty response"
- "no complete required top module extracted"
- "candidate synthesis failed under healthy tools"
- "failure appears model-output related"
- "insufficient evidence from sanitized artifacts"

Forbidden committed content:

- raw generated RTL
- raw model response text
- raw prompt text beyond public file references
- simulator log excerpts
- synthesis log excerpts
- VCD snippets
- absolute output paths exposing raw artifacts
- private text or private RTL

## Decision Rules

Recommend exactly one:

- `v0.6_expanded_baseline_closeout_spec` if the expanded baseline is sufficient to close the v0.6 benchmark-scaling milestone.
- `targeted_task_investigation_spec` if a specific task appears benchmark-ambiguous or unstable beyond model-output behavior.
- `bounded_prompt_profile_experiment_spec` only if invalid rows are dominated by response-format/empty-response problems and the project should test whether a prompt profile improves completion quality.
- `runner_output_path_cleanup_spec` if report-output path risk is the main blocker before further runs.
- `extractor_bugfix_spec` only if extraction damage is supported.
- `evaluator_bugfix_spec` only if healthy-tool evaluation appears incorrectly classified.
- `endpoint_stability_spec` only if request failures or transport errors dominate.

Do not recommend fine-tuning from this analysis.

## Definition of Done

Done when:

1. A sanitized expanded-baseline analysis Markdown report exists.
2. The 22 valid rows and 8 invalid rows are summarized clearly.
3. Per-task stability and score behavior are explained.
4. Benchmark assets are either kept unchanged or a scoped issue is justified.
5. Exactly one next action is recommended.
6. `python -m pytest` passes.
7. No raw/private/model-output/training artifacts are committed.
