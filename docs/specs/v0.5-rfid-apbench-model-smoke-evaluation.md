# Feature Spec: v0.5 RFID-APBench Model Smoke Evaluation

## Goal

Run the first bounded RFID-APBench model-output smoke evaluation after reference validation and public candidate-fixture evaluation are passing.

This stage should evaluate one model on the five public/synthetic RFID-APBench tasks with one sample per task, then score extracted candidate RTL using the existing candidate evaluator and `area_activity` scoring.

This is benchmark evaluation only. It is not fine-tuning.

## Inputs

Read first:

- `docs/specs/v0.5-rfid-apbench-candidate-evaluation.md`
- `reports/v0.5_rfid_apbench_reference_validation.md`
- `reports/v0.5_rfid_apbench_candidate_smoke.md`
- `scripts/evaluate_rfid_apbench_candidates.py`
- `src/rtlbench/area_activity_scoring.py`

## Required Scope

Implement a bounded model-smoke path that:

1. Runs or documents a 5-task, 1-sample-per-task RFID-APBench model generation smoke.
2. Uses public/synthetic RFID-APBench prompts only.
3. Extracts generated RTL into a work/output directory, not a committed candidate fixture directory.
4. Converts extracted RTL into a candidate-root layout compatible with `scripts/evaluate_rfid_apbench_candidates.py`.
5. Evaluates generated candidates using compile, simulation, VCD activity, synthesis, reference metrics, and `area_activity` scoring.
6. Emits sanitized Markdown/CSV/JSONL reports.
7. Clearly separates raw run outputs from committed sanitized artifacts.

## Required Files

Create or update as appropriate:

- `scripts/run_rfid_apbench_model_smoke.py` or a small documented orchestration script
- `reports/v0.5_rfid_apbench_model_smoke.md`
- `reports/v0.5_rfid_apbench_model_smoke.csv`
- `reports/v0.5_rfid_apbench_model_smoke.jsonl`
- optional docs update explaining the model-smoke workflow
- tests for pure helper behavior if new helpers are introduced

## Model Settings

Use a small, bounded smoke only:

- benchmark: `rfid_apbench`
- task count: 5
- samples per task: 1
- model: current configured/default model, preferably `qwen36-27b` if available
- prompt profile: `neutral_baseline` or the current default RTLBench prompt behavior
- temperature: low or deterministic if supported

If no model endpoint is available, do not fake model outputs. Instead, write a clear blocker report and keep the workflow ready.

## Output Policy

Raw model responses, extracted RTL, VCD files, simulation work dirs, logs, and synthesis scratch files must stay in ignored output/work directories.

Committed reports may include only sanitized fields such as:

- task_id
- model
- prompt_profile
- candidate/source id
- gate outcomes
- reference area/activity
- generated area/activity
- area_score
- activity_score
- score
- score_status
- failure_category
- toolchain_id
- workload_id

Do not commit raw prompts, raw responses, generated RTL, VCD contents, logs, private data, model weights, or training datasets.

## Report Requirements

Markdown report must include:

- scope and non-goals
- model and endpoint availability status without secrets
- task count and sample count
- generation/extraction status
- compile/simulation/activity/synthesis/area pass counts
- valid score count
- mean valid score
- mean all-sample score with invalid rows as zero
- per-task sanitized table
- caveat that activity is VCD toggle-count proxy, not measured power
- caveat that this is public synthetic benchmark smoke, not fine-tuning or private SiliconCraft evaluation

## Tests

Run:

```bash
python -m pytest
```

Add tests for pure helper behavior where practical, especially:

- candidate-root preparation from extracted RTL paths
- blocker behavior when model endpoint is unavailable
- sanitized report row construction

Do not require a live model endpoint in unit tests.

## Forbidden

Do not commit:

- private SiliconCraft RTL
- private task text
- raw prompts
- raw model responses
- generated RTL
- model outputs
- raw benchmark run outputs
- VCD files
- simulator logs
- synthesis logs
- compiled simulation artifacts
- tool scratch files
- secrets or endpoint credentials
- model weights
- training datasets
- LoRA, QLoRA, or DoRA adapters
- fine-tuning scripts

## Non-Goals

Do not:

- run more than the bounded 5-task smoke
- compare multiple models
- tune prompts
- create training data
- create a fine-tuning repository
- claim measured power
- claim final silicon PPA
- integrate private benchmarks

## Definition of Done

Done when:

1. A bounded RFID-APBench model-smoke workflow exists.
2. It either runs a real 5-task/1-sample model smoke or writes a clear no-endpoint blocker report.
3. Generated candidates are evaluated through the existing candidate evaluator/scoring flow.
4. Sanitized Markdown/CSV/JSONL reports exist.
5. `python -m pytest` passes.
6. No raw/private/model-output/training artifacts are committed.
7. The report clearly states this is public synthetic smoke evaluation, not fine-tuning or private evaluation.
