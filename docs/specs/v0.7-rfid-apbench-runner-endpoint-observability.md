# Bugfix/Feature Spec: v0.7 RFID-APBench Runner and Endpoint Observability

## Goal

Add sanitized request/response observability to the RFID-APBench model-runner path so future empty-response rows can be distinguished from timeout, transport failure, response-schema failure, empty server content, zero-token completion, or client normalization behavior.

The targeted empty-response investigation recommends `runner_or_endpoint_observability_spec` because the current committed fields can show that a request followed the success path and produced empty text, but cannot distinguish the internal accepted-request cause.

This is observability only. It is not prompt tuning, model comparison, benchmark modification, private evaluation, fine-tuning, or training-data creation.

## Inputs

Read first:

- `reports/v0.7_rfid_apbench_targeted_empty_response_investigation.md`
- `reports/v0.7_rfid_apbench_model_output_robustness.md`
- `reports/v0.6_rfid_apbench_expanded_3sample_baseline.md`
- `reports/v0.6_rfid_apbench_expanded_3sample_baseline.csv`
- `scripts/run_rfid_apbench_3sample_baseline.py`
- `src/rtlbench/client.py`
- `tests/test_rfid_3sample_baseline.py`

## Required Scope

Update the OpenAI-compatible model-runner/reporting path so each generation row can record sanitized request and response metadata without storing prompt text, response text, response bodies, logs, or secrets.

Add or propagate sanitized fields where available:

- `request_outcome`
- `request_attempt_count`
- `latency_seconds`
- `response_choice_count`
- `response_content_present`
- `response_character_count`
- `finish_reason`
- `prompt_tokens`
- `completion_tokens`
- `total_tokens`
- `http_status_class`
- `response_parse_status`

If a field is unavailable from the client/endpoint, record an empty value or `unavailable`, but keep the schema stable.

## Required Behavior

The runner/reporting should distinguish at least:

- successful request with non-empty text
- successful request with empty text
- timeout
- HTTP error class
- transport/request failure
- response parse/schema failure
- endpoint unavailable or missing config
- tool-health blocker

For successful responses, preserve only metadata such as counts, booleans, latency, and finish reason. Never store raw generated content in committed reports.

## Required Files

Update as appropriate:

- `src/rtlbench/client.py`
- `scripts/run_rfid_apbench_3sample_baseline.py`
- `tests/test_rfid_3sample_baseline.py`
- optional additional tests for client metadata handling

Create a sanitized documentation/report note:

- `reports/v0.7_rfid_apbench_runner_endpoint_observability.md`

Do not rerun the full benchmark in this spec. Do not create new model-output reports beyond observability dry-run/unit-test artifacts unless they contain only sanitized blocker or synthetic mocked rows.

## Report Requirements

The observability report must include:

- scope and non-goals
- source investigation summary
- fields added and their meaning
- how empty response will be distinguished after this change
- what remains unavailable or endpoint-dependent
- artifact/privacy policy
- benchmark asset decision
- fine-tuning readiness decision
- recommended next bounded step

## Tests

Run:

```bash
python -m pytest
```

Tests should cover without requiring a live model endpoint:

- successful mocked non-empty response metadata
- successful mocked empty response metadata
- timeout/request failure metadata
- HTTP error class metadata if practical
- response parse/schema failure metadata if practical
- CSV/JSONL report rows include the new sanitized fields
- Markdown output does not include raw response text
- existing 3-sample and 10-task behavior remains intact

## Forbidden

Do not commit:

- private SiliconCraft RTL
- private task text
- raw prompts beyond public benchmark prompts
- raw model responses
- response bodies
- generated model RTL
- model outputs outside sanitized reports
- raw benchmark output directories
- VCD files
- simulator logs
- synthesis logs
- compiled simulation artifacts
- tool scratch files
- secrets or endpoint credentials
- model weights
- training datasets
- LoRA, QLoRA, or DoRA adapters
- fine-tuning scripts

## Non-Goals

Do not:

- add or modify benchmark tasks
- change prompts or prompt profiles
- rerun the full benchmark
- run a targeted empty-response rerun in this step
- compare models
- tune prompts
- create training data
- fine-tune a model
- add private benchmarks
- claim measured power
- claim final silicon PPA
- change area/activity scoring formula
- change evaluator policy unless needed only for report schema propagation

## Definition of Done

Done when:

1. Generation rows can carry the new sanitized request/response metadata fields.
2. CSV and JSONL outputs include the stable observability schema.
3. Markdown reporting summarizes observability without exposing raw content.
4. Tests cover mocked non-empty, empty, and failure cases without a live endpoint.
5. `reports/v0.7_rfid_apbench_runner_endpoint_observability.md` exists.
6. `python -m pytest` passes.
7. No raw/private/model-output/training artifacts are committed.
