# Feature Spec: v0.7 RFID-APBench Bounded Prompt-Profile Experiment

## Goal

Run a small, bounded prompt-profile experiment to test whether prompt/profile-level output-completion guidance reduces the repeated empty-content behavior observed in targeted RFID-APBench runs.

The targeted observability rerun recommends `bounded_prompt_profile_experiment_spec` because empty behavior remained repeatable and the new metadata supports successful parsed empty-content outcomes rather than endpoint, transport, parse, or schema failures. In particular, `ap_010_retry_timeout_fsm` produced 3/3 `success_empty` rows with null content, finish reason `length`, and completion-token limit metadata.

This is a prompt-profile evaluation experiment only. It is not benchmark modification, model comparison, private evaluation, fine-tuning, or training-data creation.

## Required Scope

Run a bounded experiment that is limited to:

- tasks: `ap_001_idle_counter`, `ap_010_retry_timeout_fsm`
- model: configured/default model, expected `qwen36-27b` when available
- samples: exactly 3 samples per task per profile
- baseline/control profile: existing `neutral_baseline`
- challenger profile: one explicitly named completion-reliability profile
- generation settings: unchanged deterministic defaults unless the profile system already controls them
- benchmark assets: unchanged
- extractor/evaluator/scoring: unchanged
- committed outputs: sanitized reports only

The challenger may be a new additive prompt profile in `configs/prompt_profiles.yaml` if no suitable existing profile exists. Do not edit any benchmark task prompt files.

## Challenger Profile Constraints

The challenger profile should focus only on completion reliability and output format. It may instruct the model to:

- always return a complete SystemVerilog module for the requested top module
- avoid prose and markdown fences
- finish the module with `endmodule`
- avoid hidden reasoning or lengthy explanation
- keep code concise enough to fit within the token budget

It must not include task-specific answers, private examples, private RTL, or training data.

## Report Requirements

The Markdown report must include:

- scope and non-goals
- profiles compared
- targeted task list
- run ids if multiple runs are used
- model and generation settings
- endpoint/tool health status
- total attempted samples per profile
- request outcome distribution per profile
- empty-response rate per profile and task
- extraction/compile/correctness/synthesis/score counts per profile
- observability metadata summary for empty rows
- interpretation of whether the challenger reduced empty/null-content rows
- caution that correctness and scoring remain gated outcomes
- benchmark asset decision
- fine-tuning readiness decision
- exactly one recommended next step
- activity proxy caveat
- public synthetic / not private / not fine-tuning caveat

## Decision Rules

Recommend exactly one:

- `targeted_profile_validation_spec` if the challenger reduces empty/null-content rows without introducing obvious new extraction or syntax failure patterns.
- `profile_rejected_closeout_spec` if the challenger does not improve empty-response behavior or worsens gate outcomes.
- `runner_or_endpoint_bugfix_spec` if observability shows endpoint/client behavior still prevents interpreting profile effects.
- `no_immediate_followup_beyond_documentation` if results are inconclusive and no further bounded experiment is justified.

Do not recommend fine-tuning, private evaluation, model comparison, benchmark modification, or training-data creation from this experiment.

## Tests

Run:

```bash
python -m pytest
```

Add tests only if helper code is introduced. Do not require a live model endpoint in unit tests.

## Forbidden

- private SiliconCraft RTL
- private task text

- raw model responses
- response bodies
- generated model RTL
- model outputs outside sanitized reports
- raw benchmark output directories
- VCD files
- simulator logs
- synthesis logs
- compiled simulation artifacts
- tool scratch files
- secrets or endpoint credentials

- training datasets
- LoRA, QLoRA, or DoRA adapters
- fine-tuning scripts

## Non-Goals

Do not:

- add or modify benchmark tasks
- edit benchmark task prompt files
- run a benchmark-wide baseline
- compare different models
- create training data
- fine-tune a model
- add private benchmarks
- claim measured power
- claim final silicon PPA
- change area/activity scoring formula
- change evaluator policy unless needed only for profile-selection/report plumbing

## Definition of Done

Done when:

1. Sanitized Markdown/CSV/JSONL experiment reports exist.
2. The experiment covers exactly the two targeted tasks and exactly 3 samples per task per profile.
3. The reports compare neutral/control and one challenger profile using request outcomes and gate counts.
4. Empty/null-content behavior is interpreted using sanitized observability metadata.
5. Benchmark task assets remain unchanged.
6. Fine-tuning is not recommended from this evidence alone.
7. Exactly one next bounded step is recommended.
8. `python -m pytest` passes.
9. No raw/private/model-output/training artifacts are committed.
