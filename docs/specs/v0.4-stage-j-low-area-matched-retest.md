# Feature Spec: v0.4 Stage J Matched Low-Area Retest

## 1. Goal

Run the first matched public RTL-OPT prompt-profile retest now that the neutral control has valid correctness-gated generic-cell PPA records.

Stage I fixed the PPA export generic-cell fallback. The fixed neutral control now has valid `area_only` records using `area_unit=generic_cells` and `toolchain_id=yosys-generic-cell-count-proxy`. This is enough to schedule a matched `neutral_baseline` versus `low_area_low_power` experiment on the same small public RTL-OPT slice.

This is still not final silicon PPA. It is a public-flow generic-cell proxy experiment.

Required preconditions are complete:

- Yosys techmap preflight passed in Stage H.
- The fixed neutral control used the same five-task RTL-OPT slice.
- The fixed neutral control produced 3 valid PPA records out of 5.
- The fixed neutral control uses `area_only` scoring.
- Power metrics remain unavailable.
- `low_area_low_power` was blocked until now and can now be tested in a matched retest.

Reference artifacts:

## 3. Next Stage

### Stage J: Matched `neutral_baseline` vs `low_area_low_power` retest

Status: next.

Run only the matched low-area prompt retest on the same public RTL-OPT slice used by the fixed neutral control.

Run:

- benchmark: `rtlopt`
- model: `qwen36-27b`
- prompt profiles: `neutral_baseline`, `low_area_low_power`
- task slice: exactly the same 5 tasks used by the fixed control
- samples per task: 1
- evaluator: `yosys_equiv`
- scoring: v0.4 sanitized PPA export, `area_only`, generic-cell proxy when mapped area is unavailable

Do not run full benchmarks.

## 4. Required Behavior

Before model calls:

1. Run `python scripts/check_yosys_techmap.py --yosys yosys`.
2. If preflight fails, stop and do not run model jobs.
3. Confirm the same task slice as the fixed neutral control.

For the experiment:

1. Reuse the existing fixed neutral control artifact where possible.
2. Run `low_area_low_power` only on the same five tasks.
3. Export sanitized PPA records for `low_area_low_power`.
4. Summarize `low_area_low_power` records.
5. Produce a matched comparison report against the fixed `neutral_baseline` records.

## 6. Matched Comparison Rules

The comparison report must include:

- task IDs for the matched slice
- per-task neutral status
- per-task low-area status
- valid/invalid counts for both profiles
- equivalence pass counts for both profiles
- generic cell counts where valid
- area-only optimization scores where valid
- paired score delta for tasks valid under both profiles
- count of tasks where low-area improves, matches, or regresses versus neutral
- clear caveat that `generic_cells` is a proxy, not final silicon area
- clear caveat that power is unavailable

Do not compare unmatched tasks.

If a task is invalid for either profile, include it in status counts but do not compute paired optimization delta for that task.

## 7. Success Criteria

Stage J succeeds if:

- Yosys preflight passes.
- `low_area_low_power` completes on the same five tasks.
- Sanitized PPA JSONL exists for `low_area_low_power`.
- Summary Markdown/CSV exists.
- Matched neutral-vs-low-area report exists.
- No raw/private fields are committed.
- The report states whether `low_area_low_power` improved, matched, or regressed on valid paired tasks.

If no paired-valid tasks exist, the stage is still complete if the report clearly explains the blocker.

## 8. Commit Rules

Allowed:

- sanitized PPA JSONL artifacts under `artifacts/v0.4/ppa_matched/`
- matched summary Markdown/CSV reports
- docs/spec updates only if needed
- small exporter/report helper fixes only if needed

Forbidden:

- `outputs/`
- `raw_responses/`
- `extracted_rtl/`
- `logs/`
- `error_logs/`
- generated RTL code from model runs
- model responses
- prompts containing private tasks
- API keys or secrets
- caches, virtual environments, model weights

## 9. Non-goals

Do not:

- run full RTL-OPT
- run VerilogEval, RTLLM, or ProtocolLLM
- run `strict_rtl_only`
- run new models
- integrate SiliconCraft private benchmarks
- change v0.1/v0.2 frozen artifacts
- change v0.3 reports
- claim measured power improvement
- claim final silicon PPA quality

## 10. Definition of Done

Done when:

1. Preflight result is documented.
2. `low_area_low_power` matched run is complete, or preflight blocks it with exact failure.
3. Sanitized low-area PPA JSONL exists if the run executes.
4. Low-area Markdown/CSV summary exists if the run executes.
5. Matched neutral-vs-low-area report exists.
6. Report includes paired valid-task comparison.
7. Report states whether low-area improved, matched, or regressed on valid paired tasks.
8. `python -m pytest` passes.
9. No raw outputs or model responses are committed.
