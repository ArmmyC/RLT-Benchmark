# Feature Spec: v0.7 RFID-APBench ap_001 Prompt Clarity

## Goal

Clarify the public `ap_001_idle_counter` task prompt so the terminal-count timing contract is explicit while preserving the existing reference RTL, testbench behavior, interface, workload, metrics, evaluator policy, and scoring.

The `ap_001` correctness investigation classified the repeated post-fix model failure as a `state_update_timing_mismatch`: generated candidates advanced `count` from `8'h0e` to `8'h0f` while keeping `expired` low, then asserted `expired` one active clock later. The public reference and testbench expect `expired` to assert on the same active clock edge that changes `count` from `8'h0e` to `8'h0f`.

This is a public prompt/task clarity update only. It is not a benchmark behavior change, model comparison, prompt-profile experiment, private evaluation, fine-tuning, or training-data creation.

## Required Scope

Update only public documentation/task prompt text needed to make the existing behavior unambiguous.

The update should:

1. State explicitly that when `active` is high and `count` is `8'h0e`, the next rising clock edge must update `count` to `8'h0f` and assert `expired` in the same sequential update.
2. State that after expiration, `count` remains `8'h0f` and `expired` remains high until reset or synchronous clear.
3. Preserve existing reset, clear, active-low/active-high, hold, saturation, and interface semantics.
4. Preserve the existing top module name and port list.
5. Preserve reference RTL behavior.
6. Preserve testbench checks and expected outputs.
7. Preserve activity workload, constraints, metrics, scoring, evaluator behavior, and manifest identity.
8. Add or update tests only if existing tests assert prompt contents or benchmark checksums.
9. Create a sanitized report documenting the clarity change and why it is behavior-preserving.
