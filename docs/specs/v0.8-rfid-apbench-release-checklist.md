## Goal

Create a concrete, reviewable release checklist for RFID-APBench public/synthetic benchmark milestones.

The v0.8 benchmark hardening plan recommends `v0.8_release_checklist_spec` as the next bounded step because v0.7 is release-ready and needs an explicit pre-tag checklist before broader documentation, report-hygiene automation, or CI assertions.

This is documentation/checklist hardening only. It is not a model evaluation, benchmark modification, prompt-profile experiment, model comparison, private evaluation, fine-tuning, or training-data creation.

## Required Scope

Create a release checklist artifact for RFID-APBench that covers:

1. Release evidence requirements: closeout report, baseline report, CSV/JSONL row artifacts when applicable, failure-analysis/limitation report when applicable.
2. Benchmark inventory checks: manifest exists, task count is expected, task IDs are listed, public/synthetic status is stated.
3. Configuration checks: model name, prompt profile, deterministic settings, max tokens, model preset `extra_body`, endpoint/tool status, and no-secret reporting.
4. Validation guardrails: exactly 3 samples per task for baselines, targeted validation exceptions remain explicit, task filters reject unknown/duplicate IDs.
5. Report hygiene checks: source evidence, lineage, request metadata, gate counts, score summaries, limitations, caveats, and asset/fine-tuning decisions.
6. Artifact policy checks: no raw responses, generated RTL, logs, VCDs, compiled artifacts, scratch dirs, credentials, model weights, datasets, adapters, or fine-tuning scripts are tracked.
7. Git/tag checks: working tree expectations, suggested tag command, and clear note that tags are not created automatically.
8. Claim-safety checks: no measured-power, final-silicon-PPA, private-evaluation, model-superiority, or fine-tuning-readiness claims.

## Report Requirements

The Markdown report must include:

- source evidence used
- summary of the checklist artifact created
- release checklist sections covered
- v0.7 tag/readiness note
- how this checklist should be used before tagging
- benchmark asset decision
- fine-tuning readiness decision
- exactly one recommended next bounded direction
- activity proxy caveat
- public synthetic / no private / not fine-tuning caveat

## Checklist Artifact Requirements

`docs/release/rfid_apbench_release_checklist.md` must be practical and copy-pastable. It should include checkbox sections for:

- release evidence
- benchmark inventory
- model/config/run settings
- endpoint/tool status
- validation guardrails
- report hygiene
- artifact and privacy policy
- git/tag readiness
- claim safety
- known limitations
- post-release next-step planning

It should include the v0.7 suggested tag as an example:

```bash
git tag v0.7-rfid-apbench-post-fix-baseline
git push origin v0.7-rfid-apbench-post-fix-baseline
```

Do not create or push the tag.

## Decision Rules

Recommend exactly one:

- `v0.8_reproducibility_docs_spec` if the checklist is complete and the next step should document how to reproduce the v0.7 baseline.
- `v0.8_report_hygiene_spec` if report schema/lineage cleanup should happen before reproduction docs.
- `v0.8_ci_release_assertions_spec` if the checklist should be automated next.
- `no_immediate_followup_beyond_documentation` if the checklist is sufficient for now.

Do not recommend fine-tuning, private evaluation, model comparison, global max-token changes, prompt-profile experiments, benchmark modification, broad rerun, or training-data creation from this checklist step.

## Tests

Run:

```bash
python -m pytest
```

Add tests only if helper code is introduced. Do not require a live model endpoint in unit tests.

## Forbidden

Do not commit:

- private SiliconCraft RTL
- private task text
- raw prompts beyond public benchmark prompt text
- raw model responses
- response bodies
- generated model RTL
- model outputs outside sanitized reports
- raw benchmark output directories
- VCD files
- simulator logs
- synthesis logs
- compiled artifacts
- tool scratch files
- secrets or endpoint credentials
- model weights
- training datasets
- LoRA, QLoRA, or DoRA adapters
- fine-tuning scripts

## Non-Goals

Do not:

- run another model evaluation
- add or modify benchmark tasks
- edit prompt text
- edit reference RTL or testbench behavior
- change module interfaces
- compare models
- change prompt profiles
- raise global max-token default
- create training data
- fine-tune a model
- add private benchmarks
- claim measured power
- claim final silicon PPA
- change workload, metrics, area/activity scoring, or evaluator policy
- create or push git tags

## Definition of Done

Done when:

1. `docs/release/rfid_apbench_release_checklist.md` exists.
2. `reports/v0.8_rfid_apbench_release_checklist.md` exists.
3. The checklist covers release evidence, inventory, config, guardrails, report hygiene, artifacts/privacy, git/tag readiness, claim safety, known limitations, and next planning.
4. The report summarizes how to use the checklist before tagging.
5. Benchmark assets remain unchanged.
6. Fine-tuning is not recommended from this evidence alone.
7. Exactly one next bounded direction is recommended.
8. `python -m pytest` passes.
9. No raw/private/model-output/training artifacts are committed.
