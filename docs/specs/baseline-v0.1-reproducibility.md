# Feature Spec: Baseline v0.1 Reproducibility Package

## 1. Goal

Build a reproducible Baseline v0.1 reporting package for the current public RTL benchmark results in `RLT-Benchmark`.

The repository already contains a reusable RTL benchmark harness, benchmark adapters, model presets, output manifests, and a manually curated five-model comparison report. This feature turns those existing results into a structured, machine-readable, regenerable baseline package.

The goal is to make the current five-model public RTL benchmark baseline easy to audit, regenerate, compare, and present without adding more models or changing benchmark methodology.

The feature must add:

1. A committed run registry at `runs/index.yaml`.
2. A generated comparison report pipeline.
3. A cross-model failure analysis pipeline.
4. A static dashboard generated from committed baseline data.
5. Documentation that explains how Baseline v0.1 should be maintained.

The current Baseline v0.1 model set is:

- `qwen36-27b`
- `qwen36-35b-a3b`
- `qwen3-coder-30b-a3b-instruct`
- `qwen25-coder-32b`
- `deepseek-coder-v2-lite-instruct`

The current benchmark families are:

- VerilogEval v2 functional simulation
- RTLLM 2.0 functional simulation
- ProtocolLLM public lint-only
- RTL-OPT lint, generic synthesis, and equivalence

The main reason for this feature is to stop relying on hand-maintained Markdown tables as the only comparison artifact. Markdown reports should remain useful for humans, but the baseline must also have YAML, JSON, CSV, and generated HTML artifacts.

## 2. Non-goals

This task must not build or change the following:

1. Do not add new model benchmarks.
2. Do not rerun any benchmark unless explicitly requested outside this spec.
3. Do not implement automatic Slurm model swapping.
4. Do not execute SSH commands against LANTA.
5. Do not add Open WebUI, LiteLLM, or chat UI features to this repository.
6. Do not start the private RFID benchmark.
7. Do not add low-power or PPA-specific prompt experiments.
8. Do not implement repair-loop benchmarking.
9. Do not replace the existing benchmark runner.
10. Do not commit `outputs/`, raw model responses, extracted RTL files, large logs, model weights, virtual environments, or caches.
11. Do not claim new benchmark results unless they are backed by existing committed reports, registered summaries, or accessible `summary.json` / `results.jsonl` files.
12. Do not mix lint-only results with functional simulation results as if they measure the same thing.
13. Do not change the existing model serving workflow from `Lanta-LLM-Hosting`.

## 3. Assumptions

1. `outputs/` is intentionally gitignored and may not exist on every machine.
2. Some historical runs may exist only on LANTA, not in the local checkout.
3. Existing committed reports and output manifests are acceptable sources for Baseline v0.1 registration, as long as the registry clearly marks those values as manually registered from committed reports.
4. The generated report scripts must work even when the raw output directories are unavailable locally.
5. If a registered run has an accessible `summary.json`, the script should prefer it over manually registered summary values.
6. If `results.jsonl` is unavailable, cross-model failure analysis should skip per-task analysis for that run and clearly report that per-task data is missing.
7. The repository currently uses Python and PyYAML. New scripts should use the standard library plus existing dependencies unless a dependency is clearly necessary.
8. The static dashboard can be plain HTML, CSS, and JavaScript generated by Python. It should not require a server.
9. The feature is primarily for reproducibility and presentation, not for new research claims.
10. LANTA model swapping is owned by the separate `Lanta-LLM-Hosting` repository. This repository should only reference that workflow in documentation.

## 4. User stories

- As a benchmark maintainer, I want a committed run registry, so that I can identify which output folders and settings produced Baseline v0.1.
- As a benchmark maintainer, I want comparison reports generated from structured data, so that I do not accidentally hand-edit incorrect numbers.
- As a researcher, I want the report to separate functional simulation, lint-only, synthesis-only, and equivalence results, so that I can interpret the results honestly.
- As a researcher, I want cross-model failure analysis, so that I can understand which tasks are hard for all models and which tasks are uniquely solved by one model.
- As a reviewer, I want a static dashboard, so that I can inspect benchmark results without running Python or opening multiple Markdown files.
- As a future contributor, I want clear documentation for Baseline v0.1, so that I can update the registry and regenerate reports safely.
- As a LANTA user, I want the benchmark repo to avoid owning model swap logic, so that serving and benchmarking remain separate.
- As a project owner, I want all generated artifacts to be small and committed, so that the public repo remains lightweight and useful.

## 5. UX / UI requirements

This feature has two user interfaces:

1. Command-line scripts.
2. A generated static dashboard.

### 5.1 CLI UX requirements

The CLI scripts must be usable from the repository root.

Required commands:

```bash
python scripts/generate_comparison_report.py --registry runs/index.yaml --baseline baseline_v0_1
python scripts/analyze_cross_model_failures.py --registry runs/index.yaml --baseline baseline_v0_1
python scripts/build_dashboard.py --registry runs/index.yaml --baseline baseline_v0_1
```

Each script must:

1. Print a short summary of what it generated.
2. Print output file paths.
3. Exit with nonzero status on invalid registry schema, missing required fields, or unrecoverable parsing errors.
4. Exit with zero status if some optional raw artifacts are unavailable but fallback manual summaries are valid.
5. Clearly distinguish warnings from errors.

Example successful output:

```text
Loaded baseline: baseline_v0_1
Registered runs: 35
Generated reports/baseline_v0.1_public_rtl_benchmarks.md
Generated reports/baseline_v0.1_public_rtl_benchmarks.json
Generated reports/baseline_v0.1_public_rtl_benchmarks.csv
Warnings: 4 runs used manual_summary fallback because summary.json was unavailable.
```

Example error output:

```text
ERROR: runs/index.yaml baseline_v0_1 run 3 is missing required field: model
```

### 5.2 Static dashboard UX requirements

The dashboard must be generated at:

```text
dashboard/index.html
```

The dashboard must be viewable by opening the file directly in a browser. It must not require a server.

Dashboard sections:

1. Header
   - Title: `RTLBench Baseline v0.1`
   - Subtitle explaining that this is a five-model public RTL benchmark baseline.
   - Generation timestamp.

2. Baseline summary cards
   - Number of models.
   - Number of benchmark families.
   - Best model for VerilogEval pass@1.
   - Best model for VerilogEval pass@5.
   - Best model for RTLLM pass@1.
   - Best model for RTL-OPT equivalence.

3. Model comparison table
   - Rows: benchmark/mode/metric.
   - Columns: models.
   - Values: rates or counts.
   - Lint-only and synthesis-only rows must be visibly labeled.

4. RTL-OPT section
   - Equivalence pass rate.
   - Equiv-passing tasks with fewer generic cells.
   - Average generic cell ratio among equiv-passing tasks.
   - Text warning that generic Yosys cells are not final silicon PPA.

5. Failure analysis section
   - Failure category counts by model if available.
   - Per-task matrix if `results.jsonl` data is available.
   - Empty state if per-task data is unavailable.

6. Artifact links section
   - Links to generated Markdown report.
   - Links to JSON and CSV outputs.
   - Links to committed output manifests if registered.

### 5.3 Empty states

The dashboard must handle missing data gracefully.

If no per-task `results.jsonl` data is available, show:

```text
Per-task failure data is not available for this run. The summary-level baseline is still shown from registered summaries.
```

If no RTL-OPT metric exists for a model, show `N/A` instead of `0`.

If a metric is lint-only, show a label such as:

```text
lint-only, not functional correctness
```

### 5.4 Loading states

The generated dashboard is static. No runtime loading state is required.

If JavaScript is used, all data should be embedded in the HTML or loaded from local `dashboard/data/*.json` files. If local JSON loading is used, the HTML must include a fallback message explaining that some browsers block local file fetches when opened through `file://`.

Preferred implementation: embed the needed JSON directly into `index.html` to avoid local fetch restrictions.

### 5.5 Error states

The dashboard should not display raw Python tracebacks. Dashboard generation should fail before writing incomplete HTML when required registry data is invalid.

If optional data is missing, the dashboard should render with warnings.

### 5.6 Responsive behavior

The dashboard must be readable on:

1. Desktop browser width, 1200px and above.
2. Tablet width, around 768px.
3. Narrow width, around 390px.

Tables may horizontally scroll on narrow screens. Cards should stack vertically on narrow screens.

## 7. Technical requirements

### 7.1 Architecture

The feature should use a small set of reusable modules rather than putting all logic in scripts.

Preferred structure:

```text
src/rtlbench/registry.py
src/rtlbench/comparison.py
src/rtlbench/failure_matrix.py
src/rtlbench/dashboard.py
scripts/generate_comparison_report.py
scripts/analyze_cross_model_failures.py
scripts/build_dashboard.py
```

Responsibilities:

- `registry.py`
  - Load `runs/index.yaml`.
  - Validate schema.
  - Resolve run paths.
  - Load summary data from `summary.json` or manual fallback.

- `comparison.py`
  - Convert registered runs into comparison tables.
  - Render Markdown.
  - Render JSON structure.
  - Render CSV rows.

- `failure_matrix.py`
  - Load `results.jsonl` files when available.
  - Build per-task and per-model matrix rows.
  - Render Markdown and CSV.

- `dashboard.py`
  - Render static HTML from generated comparison and failure data.

- `scripts/*.py`
  - Thin command-line wrappers only.

This structure keeps the logic testable.

### 7.2 Data flow

Primary data flow:

```text
runs/index.yaml
    ↓
registry loader
    ↓
summary loader
    ↓
comparison builder
    ↓
Markdown / JSON / CSV reports
    ↓
dashboard builder
    ↓
dashboard/index.html
```

Failure data flow:

```text
runs/index.yaml
    ↓
registered results.jsonl paths
    ↓
failure matrix builder
    ↓
failure_matrix.md / failure_matrix.csv
    ↓
dashboard failure section
```

### 7.3 Validation rules

The registry loader must validate:

1. The requested baseline key exists.
2. `runs` is a list.
3. Each run has all required fields.
4. `samples_per_task` is a positive integer.
5. `temperature` is numeric.
6. `top_p` is numeric.
7. `max_tokens` is a positive integer.
8. `evaluation_kind` is one of:
   - `functional_simulation`
   - `lint_only`
   - `synthesis_only`
   - `equivalence`
   - `optimization_summary`
9. `result_source` is one of the allowed values listed in Functional requirements.
10. If `summary_path` is missing or inaccessible, `manual_summary` exists.
11. Manual summary values are numeric where numeric metrics are expected.
12. No secret-like keys are present, including:
   - `api_key`
   - `token`
   - `password`
   - `secret`

### 7.4 Permissions and security

1. Scripts must only read local files and write generated report/dashboard files.
2. Scripts must not call external network services.
3. Scripts must not run SSH.
4. Scripts must not execute Slurm commands.
5. Scripts must not read environment variables containing secrets except if future code needs local path overrides. This feature does not need secret environment variables.
6. Generated files must not include API keys or passwords.
7. If the registry includes endpoint values, they should be local or non-secret endpoint strings only, for example `http://127.0.0.1:8000/v1`.

### 7.5 Dependency policy

Do not introduce new runtime dependencies unless necessary.

Allowed existing dependencies:

- Python standard library
- PyYAML, already used by the project

Do not add:

- pandas
- plotly
- streamlit
- flask
- fastapi
- rich
- jinja2

If templating is needed, use simple string rendering or the standard library.

### 7.6 Formatting policy

Generated Markdown tables must be deterministic and stable.

Floating-point values should be formatted to four decimal places for pass rates and ratios unless the existing report uses a different required precision.

Counts should be formatted as integers or `x / y` strings.

Missing values should be rendered as `N/A`.

### 7.7 Existing behavior preservation

Existing commands such as the base `rtlbench` CLI must continue to work.

Do not change core benchmark execution unless required for shared helper imports.

Do not change evaluator semantics.

Do not change existing model presets unless a test reveals a clear schema issue.

## 9. Data model

There is no database migration for this feature.

The data model is YAML plus generated JSON/CSV files.

### 9.1 `runs/index.yaml` schema

Top-level type: mapping.

Example:

```yaml
baseline_v0_1:
  name: Baseline v0.1 Public RTL Benchmarks
  description: Five-model public RTL benchmark baseline.
  created_utc: "2026-06-15T00:00:00Z"
  status: frozen
  models:
    - qwen36-27b
    - qwen36-35b-a3b
    - qwen3-coder-30b-a3b-instruct
    - qwen25-coder-32b
    - deepseek-coder-v2-lite-instruct
  benchmarks:
    - verilogeval
    - rtllm2
    - protocollm
    - rtlopt
  runs:
    - id: verilogeval_pass1_qwen36_27b
      model: qwen36-27b
      served_model_name: qwen36-27b
      benchmark: verilogeval
      mode: pass1
      evaluation_kind: functional_simulation
      samples_per_task: 1
      temperature: 0.2
      top_p: 0.95
      max_tokens: 2048
      evaluator_type: icarus
      endpoint: http://127.0.0.1:8000/v1
      verified_by_models_endpoint: true
      result_source: manual_summary
      output_path: /project/.../outputs/verilogeval/qwen36-27b/...
      manifest_path: reports/qwen36_27b_output_manifest.md
      manual_summary:
        tasks: 156
        samples: 156
        syntax_pass_rate: 0.8397
        functional_pass_rate: 0.6154
        pass_at_k:
          pass@1: 0.6154
        failure_categories:
          passed: 96
          simulation_failure: 35
          compile_failure: 17
          code_extraction_failure: 8
```

### 9.2 Run fields

Required fields:

| Field | Type | Required | Description |
|---|---:|---:|---|
| `id` | string | yes | Unique run identifier inside the baseline. |
| `model` | string | yes | Normalized model key used in reports. |
| `served_model_name` | string | yes | Name returned by the model endpoint or configured vLLM alias. |
| `benchmark` | string | yes | Benchmark family. |
| `mode` | string | yes | Experiment mode, for example `pass1`, `pass5`, `lint`, `equiv`. |
| `evaluation_kind` | string | yes | Semantic category of evaluation. |
| `samples_per_task` | integer | yes | Samples generated per task. |
| `temperature` | float | yes | Generation temperature. |
| `top_p` | float | yes | Generation top-p. |
| `max_tokens` | integer | yes | Generation max token limit. |
| `evaluator_type` | string | yes | Evaluator type used by the harness. |
| `result_source` | string | yes | Source of result values. |

Optional fields:

| Field | Type | Description |
|---|---:|---|
| `endpoint` | string | Non-secret base URL used during run. |
| `verified_by_models_endpoint` | boolean | Whether `/v1/models` was checked. |
| `output_path` | string | Original output directory. May be remote. |
| `summary_path` | string | Path to `summary.json` if available. |
| `results_path` | string | Path to `results.jsonl` if available. |
| `manifest_path` | string | Committed manifest path. |
| `report_path` | string | Committed report path. |
| `manual_summary` | mapping | Fallback summary values. |
| `rtlopt_metrics` | mapping | Optional RTL-OPT-specific metrics. |
| `notes` | string | Human notes. |

### 9.3 Manual summary fields

Manual summary should support these fields:

```yaml
manual_summary:
  tasks: 156
  samples: 156
  syntax_pass_rate: 0.8397
  functional_pass_rate: 0.6154
  pass_at_k:
    pass@1: 0.6154

    passed: 96
    simulation_failure: 35
  evaluation_metrics:

      mean: 10.0
```

RTL-OPT runs may include:

```yaml
rtlopt_metrics:

```

The generated JSON should include:


No database indexes or migrations are required.

## 10. API contract

This feature does not add HTTP API endpoints or server actions.

It adds command-line contracts.

### 10.1 Generate comparison report

- Name: Generate comparison report
- Method: CLI command
- Path: `scripts/generate_comparison_report.py`
- Request body: Not applicable
- Command:

```bash
python scripts/generate_comparison_report.py \
  --registry runs/index.yaml \
  --baseline baseline_v0_1
```

- Optional flags:

```text
--output-md reports/baseline_v0.1_public_rtl_benchmarks.md
--output-json reports/baseline_v0.1_public_rtl_benchmarks.json
--output-csv reports/baseline_v0.1_public_rtl_benchmarks.csv
--strict
```

- Response body: Not applicable
- Successful stdout:

```text
Generated reports/baseline_v0.1_public_rtl_benchmarks.md
Generated reports/baseline_v0.1_public_rtl_benchmarks.json
Generated reports/baseline_v0.1_public_rtl_benchmarks.csv
```

- Error cases:
  - Registry file missing.
  - Baseline key missing.
  - Required run field missing.
  - Invalid numeric value.
  - Missing summary with no manual fallback.
  - Output path unwritable.

### 10.2 Analyze cross-model failures

- Name: Analyze cross-model failures
- Method: CLI command
- Path: `scripts/analyze_cross_model_failures.py`
- Request body: Not applicable
- Command:

```bash
python scripts/analyze_cross_model_failures.py \
  --registry runs/index.yaml \
  --baseline baseline_v0_1
```

- Optional flags:

```text
--output-md reports/baseline_v0.1_failure_matrix.md
--output-csv reports/baseline_v0.1_failure_matrix.csv
--benchmark verilogeval
--mode pass1
```

- Response body: Not applicable
- Successful stdout:

```text
Generated reports/baseline_v0.1_failure_matrix.md
```

### 10.3 Build static dashboard

- Name: Build static dashboard
- Method: CLI command

## 11. Edge cases

1. `runs/index.yaml` does not exist.
   - Create it for the initial implementation if possible, otherwise fail with a clear instruction.

2. A run has `summary_path`, but the file is unavailable on the current machine.
   - Use `manual_summary` if provided.
   - Emit a warning.

3. A run has neither accessible `summary_path` nor `manual_summary`.
   - Treat as an error for comparison report generation.

4. A run has `results_path`, but the file is unavailable.
   - Skip per-task failure analysis for that run.
   - Emit a warning.

5. A run has accessible `results.jsonl`, but one line is malformed JSON.
   - Fail in strict mode.
   - In non-strict mode, skip the malformed line and warn.

6. A metric is missing for one model.
   - Render `N/A`.
   - Do not treat missing as zero.

7. A model has different generation settings for the same benchmark/mode.
   - Mark the matched settings audit as failed for that row.
   - Do not make a best-model claim for that row unless the report explicitly notes the mismatch.

8. ProtocolLLM has only lint results.
   - Keep it in a separate lint-only section.

9. RTL-OPT generic synthesis passes but equivalence fails.
   - Report synthesis separately from equivalence.
   - Do not count synthesis pass as behavior-preserving success.

10. Multiple registered runs exist for the same model/benchmark/mode.
    - Require either unique `id`s and a field such as `selected: true`, or fail with ambiguity.
    - The initial registry should avoid duplicate selected runs.

11. Historical qwen36-27b data lacks modern `run_metadata.json`.
    - Allow manual summary values and note that the run predates the newer metadata format.

12. Dashboard opened on mobile.
    - Tables should scroll horizontally.
    - Summary cards should stack.

13. Dashboard opened through `file://`.
    - It should still work if data is embedded.

14. User runs scripts from a subdirectory.
    - Either support `--repo-root` or document that commands must be run from repo root.
    - Preferred: resolve paths relative to current working directory and show clear errors.

15. Existing reports contain values not represented in registry.
    - Registry should be updated rather than reading arbitrary Markdown tables at runtime.
