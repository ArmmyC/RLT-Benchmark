# Feature Spec: v0.6 RFID-APBench Expanded 10-Task 3-Sample Model Baseline

## Goal

Run the first model baseline on the expanded 10-task RFID-APBench benchmark.

The benchmark now contains exactly 10 public/synthetic RFID/AP-style tasks. Reference validation passes for all 10 tasks, the `reference_copy` candidate fixture passes for all 10 tasks, and the 3-sample baseline runner now accepts the expanded manifest-driven task count while still enforcing exactly 3 samples per task.

This stage should produce a sanitized 10-task by 3-sample model-evaluation baseline using the existing tool-health-aware evaluator and existing area/activity scoring.

This is model evaluation only. It is not model comparison, prompt tuning, fine-tuning, private evaluation, or training-data creation.

## Required Scope

Run a fresh expanded model baseline that:

1. Uses all current RFID-APBench manifest tasks, expected 10 tasks.
2. Runs exactly 3 samples per task, expected 30 attempted samples.
3. Uses unchanged prompts, references, testbenches, activity workloads, reference metrics, extractor policy, evaluator policy, and scoring formula.
4. Uses the configured/default model, expected `qwen36-27b` when available.
5. Uses `neutral_baseline` or the existing default prompt behavior.
6. Uses low/deterministic generation settings where supported.
7. Requires healthy Icarus, `vvp`, and Yosys tool health checks.
8. Evaluates extracted candidates through compile, simulation correctness, VCD activity, synthesis, area/activity metric availability, and scoring.
9. Emits sanitized Markdown/CSV/JSONL reports.
10. Does not commit raw prompts, raw responses, generated RTL, VCDs, logs, compiled artifacts, synthesis scratch, private data, or training artifacts.

## Required Files

Update or create as appropriate:

- `reports/v0.6_rfid_apbench_expanded_3sample_baseline.md`
- `reports/v0.6_rfid_apbench_expanded_3sample_baseline.csv`
- `reports/v0.6_rfid_apbench_expanded_3sample_baseline.jsonl`

Do not overwrite the earlier five-task baseline reports. Keep the expanded 10-task run in separate files.

## Report Requirements

The Markdown report must include:

- scope and non-goals
- run id
- model and prompt profile
- endpoint status without secrets
- tool health status for Icarus, `vvp`, and Yosys
- task count and samples per task
- total sample count
- generation/extraction/candidate availability counts
- compile/correctness/synthesis counts
- area/activity availability counts
- valid score count
- mean valid score
- mean all-sample score with invalid rows as zero
- per-task aggregate summary
- per-sample sanitized summary
- failure distribution
- comparison against the five-task post-tool-health baseline
- statement that activity is a VCD toggle-count proxy, not measured power
- public synthetic / not private / not fine-tuning caveat

## Expected Behavior

If endpoint and tools are healthy:

- attempt all 30 samples
- evaluate extracted candidates through the full tool-health-aware gate stack
- distinguish model-output failures from tool-health failures

If the endpoint fails:

- produce sanitized `request_failed` or `endpoint_unavailable` rows as appropriate
- do not fake generation

If tool health fails:

- produce sanitized `tool_health_failed` or `tool_startup_failure` rows
- do not classify those as candidate `compile_failure`

## Tests

Run:

```bash
python -m pytest
```

Do not require a live model endpoint in unit tests.

## Forbidden

Do not commit:

- private SiliconCraft RTL
- private task text
- raw prompts beyond public benchmark prompts
- raw model responses
- generated model RTL
- model outputs outside sanitized reports
- raw benchmark output directories
- VCD files
- simulator logs
- synthesis logs
- compiled simulation artifacts
- tool scratch files
- secrets or endpoint credentials
- model weights
- training datasets
- LoRA, QLoRA, or DoRA adapters
- fine-tuning scripts

## Non-Goals

Do not:

- add or modify benchmark tasks
- compare models
- tune prompts
- create training data
- fine-tune a model
- add private benchmarks
- claim measured power
- claim final silicon PPA
- change area/activity scoring formula
- change evaluator policy except for clear validation bugs

## Definition of Done

Done when:

1. A fresh expanded 10-task by 3-sample model baseline is recorded in sanitized Markdown/CSV/JSONL.
2. All current manifest tasks have exactly three attempted samples.
3. Reports distinguish generation, extraction, compile, simulation correctness, synthesis, area/activity metrics, scoring, and any tool-health failures.
4. Valid and all-sample scores are reported.
5. The earlier five-task baseline reports are preserved.
6. `python -m pytest` passes.
7. No raw/private/model-output/training artifacts are committed.
