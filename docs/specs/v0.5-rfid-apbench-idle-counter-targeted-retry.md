# Feature Spec: v0.5 RFID-APBench Idle Counter Targeted Retry

## Goal

Run a bounded targeted retry for the only failed task from the first real RFID-APBench model smoke: `ap_001_idle_counter`.

The first real smoke generated all five tasks, but `ap_001_idle_counter` failed extraction while the other four tasks extracted, compiled, simulated, synthesized, and scored successfully. This retry should determine whether the idle-counter failure was a one-off empty/incomplete response or a repeatable extraction-fragile task for the same model/settings pair.

This is benchmark evaluation only. It is not prompt tuning and not fine-tuning.

## Inputs

Read first:

- `reports/v0.5_rfid_apbench_model_smoke_analysis.md`
- `reports/v0.5_rfid_apbench_model_smoke_analysis.csv`
- `reports/v0.5_rfid_apbench_model_smoke.md`
- `scripts/run_rfid_apbench_model_smoke.py`
- `scripts/evaluate_rfid_apbench_candidates.py`
- `src/rtlbench/extraction.py`

## Required Scope

Implement or run a targeted retry path that:

1. Runs only `ap_001_idle_counter`.
2. Uses the same model/settings as the first real smoke unless the current endpoint requires equivalent config names.
3. Uses 3 samples for this one task.
4. Keeps public/synthetic prompts only.
5. Keeps raw model responses, extracted RTL, generated candidate RTL, VCDs, logs, compiled artifacts, and synthesis scratch under ignored output/work directories.
6. Evaluates any extracted candidates through the existing candidate evaluator/scoring path.
7. Emits sanitized Markdown/CSV/JSONL retry reports.
8. Does not modify the prompt, task spec, extractor, or evaluator unless a concrete bug is found and documented.

## Required Files

Create or update as appropriate:

- `reports/v0.5_rfid_apbench_idle_counter_retry.md`
- `reports/v0.5_rfid_apbench_idle_counter_retry.csv`
- `reports/v0.5_rfid_apbench_idle_counter_retry.jsonl`
- optional small orchestration script or CLI option to support `--task-id` and `--samples`
- optional tests for pure helper behavior if new helper code is introduced

Do not commit raw run directories or generated RTL.

## Retry Settings

Use:

- task: `ap_001_idle_counter`
- samples: 3
- model: same configured model as first smoke, expected `qwen36-27b`
- prompt profile: `neutral_baseline`
- temperature: same as first smoke, preferably deterministic/low
- top-p: same as first smoke
- max tokens: same as first smoke

If endpoint or EDA tools are unavailable, write a clear sanitized blocker report instead of faking outputs.

## Report Requirements

Markdown report must include:

- scope and non-goals
- model/endpoint availability without secrets
- task id and sample count
- generation/extraction counts
- compile/correctness/synthesis counts
- area/activity metric availability counts
- valid score count
- per-sample sanitized table
- conclusion: one-off response issue vs repeatable extraction fragility vs tool/endpoint blocker
- recommended next action
- caveat that activity is VCD toggle-count proxy, not measured power
- caveat that this is public synthetic smoke, not private evaluation and not fine-tuning

CSV/JSONL rows should include sanitized per-sample fields:

- task_id
- sample_id
- model
- prompt_profile
- generation_status
- extraction_status
- candidate_file_available
- compile_pass
- correctness_pass
- synth_pass
- reference_area
- generated_area
- reference_activity
- generated_activity
- area_score
- activity_score
- score
- score_status
- failure_category
- notes

## Acceptance Criteria

If at least one sample extracts and reaches evaluation, summarize the best valid retry score and compare it to the original invalid sample.

If all three samples fail extraction, classify `ap_001_idle_counter` as repeatably extraction-fragile for this model/settings pair and recommend a separate scoped prompt/extractor investigation.

If endpoint or tools are unavailable, classify as an operational blocker and do not change benchmark conclusions.

## Forbidden

Do not commit:

- private SiliconCraft RTL
- private task text
- raw prompts
- raw model responses
- generated RTL
- model outputs
- raw benchmark run outputs
- VCD files
- simulator logs
- synthesis logs
- compiled simulation artifacts
- tool scratch files
- secrets or endpoint credentials
- model weights
- training datasets
- LoRA, QLoRA, or DoRA adapters
- fine-tuning scripts

## Non-Goals

Do not:

- run more than 3 samples for this task
- rerun all five tasks unless endpoint/tool failure invalidates the targeted retry
- compare multiple models
- tune prompts broadly
- change the benchmark task text without a separate spec
- fine-tune a model
- create training data
- claim measured power
- claim final silicon PPA
- integrate private benchmarks

## Definition of Done

Done when:

1. A sanitized targeted-retry Markdown report exists.
2. Sanitized CSV/JSONL retry reports exist.
3. The retry outcome is classified as one-off extraction issue, repeatable extraction fragility, or operational blocker.
4. Any valid retry candidate is evaluated through the existing area/activity scoring path.
5. `python -m pytest` passes.
6. No raw/private/model-output/training artifacts are committed.
