# Feature Spec: v0.4 Stage M Conservative Area v2

## 1. Goal

Revise the `conservative_area` prompt after the 20-task validation showed it is safer than `low_area_low_power` but still weaker than `neutral_baseline`.

Stage L showed:

- `neutral_baseline`: 14 valid PPA records out of 20
- `conservative_area`: 12 valid PPA records out of 20
- paired-valid tasks: 11
- conservative improved: 0
- conservative matched: 10
- conservative regressed: 1
- conservative gained validity on `calculation`
- conservative lost validity on `adder`, `divider_4bit`, and `fsm`

The next step is prompt revision, not scale-up. Create a stricter correctness-first area profile and retest on the same 20-task slice.

This remains a public-flow generic-cell proxy experiment. It is not final silicon PPA and has no measured power data.

## 2. Next Stage

### Stage M: `conservative_area_v2` prompt revision and 20-task retest

Status: next.

Add a new prompt profile:

- `conservative_area_v2`

Do not modify existing prompt profiles. Keep all previous artifacts reproducible.

## 3. Prompt Design Requirements

The new profile should be stricter than `conservative_area`.

It should instruct the model to:

- preserve module name, port list, signal widths, signedness, and interface behavior exactly
- prioritize correctness, synthesizability, and equivalence above area
- implement the most direct correct RTL first
- apply only local, obviously behavior-preserving simplifications
- avoid changing latency, reset behavior, enable behavior, FSM transitions, arithmetic semantics, division semantics, or protocol behavior
- avoid custom minimized arithmetic/control rewrites unless the task explicitly asks for them
- avoid aggressive serialization, resource sharing, register removal, or FSM restructuring
- avoid reducing bit widths unless the specification explicitly proves it is safe
- use standard Verilog/SystemVerilog operators for arithmetic/comparison where that is the clearest correct implementation
- optimize only when the full input/output truth table or sequential behavior is unchanged
- if optimization conflicts with clarity or correctness, choose the straightforward correct version
- return RTL code only
- do not claim exact PPA improvement without synthesis reports

1. Run `python scripts/check_yosys_techmap.py --yosys yosys`.
2. If preflight fails, stop and do not run model jobs.
3. Confirm the exact same 20-task slice used in Stage L.

Run only:

- benchmark: `rtlopt`

- task slice: exactly the same 20 Stage L tasks

Do not rerun `neutral_baseline` unless the existing Stage L neutral artifact is missing or unusable.

Do not rerun `low_area_low_power`.

## 6. Comparison Rules

Compare `conservative_area_v2` against the existing Stage L `neutral_baseline` artifact.

The matched comparison report must include:

- exact 20-task list
- valid/invalid counts for neutral and v2
- equivalence pass counts for neutral and v2
- extraction/compile/synthesis pass counts
- per-task status for both profiles
- per-task generic cell counts where available
- per-task area-only scores where valid
- paired score delta for tasks valid under both profiles
- counts of v2 improved, matched, and regressed tasks versus neutral
- count of tasks where v2 gained or lost validity versus neutral
- explicit status for Stage L weak points: `adder`, `divider_4bit`, `fsm`, `decoder_6bit`, and `calculation`
- mean valid score and mean all-sample score for both profiles
- recommendation: reject, revise again, keep as challenger, or promote for a larger run

Do not compute deltas for invalid or unmatched tasks.

## 7. Required Caveats

Reports must state:

- `generic_cells` is a public-flow area proxy, not final silicon area
- power metrics are unavailable
- timing is not being optimized or claimed
- this is still a small public validation, not a SiliconCraft private benchmark

## 8. Success Criteria

Stage M succeeds if:

- `conservative_area_v2` profile is added without changing previous profiles
- Yosys preflight passes
- v2 completes the same 20-task slice, or blocker is documented
- sanitized JSONL artifact exists if the run executes
- summary Markdown/CSV exists if the run executes
- matched comparison Markdown/CSV exists
- report recommends the next action based on validity and paired score deltas

A promising v2 should meet at least one of these:

- validity is equal to or better than neutral, with no paired score regression
- validity is equal to neutral and at least one paired-valid task improves
- if validity is below neutral, the report should recommend rejection or another revision, not scale-up

## 9. Commit Rules

Allowed:

- `configs/prompt_profiles.yaml`
- sanitized JSONL artifact under `artifacts/v0.4/ppa_validation20/`
- Markdown/CSV reports
- docs/spec updates only if needed
- small report/export helper fixes only if needed

Forbidden:

- `outputs/`
- `raw_responses/`
- `extracted_rtl/`
- `logs/`
- `error_logs/`
- generated RTL code from model runs
- model responses
- prompts containing private tasks
- API keys or secrets
- caches, virtual environments, model weights

## 10. Non-goals

Do not:

- run full RTL-OPT
- run `low_area_low_power`
- run `strict_rtl_only`
- run VerilogEval, RTLLM, or ProtocolLLM
- run new models
- integrate SiliconCraft private benchmarks
- change v0.1/v0.2 frozen artifacts
- change v0.3 reports
- claim measured power improvement
- claim final silicon PPA quality

## 11. Definition of Done

Done when:

1. `conservative_area_v2` profile exists.
2. Preflight result is documented.
3. Exact Stage L 20-task list is documented.
4. v2 completes the matched run, or preflight/tooling blocker is clearly documented.
5. Sanitized PPA JSONL artifact exists if run executes.
6. Summary Markdown/CSV report exists if run executes.
7. Matched comparison Markdown/CSV report exists.
8. Report states whether v2 should be rejected, revised again, kept as challenger, or promoted to a larger run.
9. `python -m pytest` passes.
10. No raw outputs or model responses are committed.
