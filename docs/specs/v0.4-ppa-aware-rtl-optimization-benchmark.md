# Feature Spec: v0.4 PPA-Aware RTL Optimization Benchmark

## 1. Goal

Add sanitized v0.4 PPA result artifact and summary reporting support without running new model jobs.

The v0.4 design docs and pure scoring core are complete. The next layer should make scored records easy to save, validate, load, aggregate, and summarize while preserving the no-raw-output safety boundary.

Do not integrate with real benchmark run directories yet. This stage should use synthetic/unit-test records only.

## 2. Completed Inputs

Design docs:

- `docs/experiments/v0.4_ppa_aware_rtl_optimization.md`
- `reports/v0.4_ppa_scoring_design.md`

Scoring core:

- `src/rtlbench/ppa_scoring.py`
- `tests/test_ppa_scoring.py`

Core scoring decisions:

- required gates: extraction, compile, synthesis, equivalence, timing when required
- invalid per-sample records use `score_status = invalid` and `optimization_score = null`
- aggregate all-sample views map invalid samples to zero
- full score uses area and power ratios when both are available
- area-only mode is explicit when power is unavailable
- sanitized records must exclude raw RTL, prompts, model responses, logs, and private task text

## 3. Next Stage

### Stage C: Sanitized PPA artifact and summary reporting

Status: next.

Implement a reporting layer that operates only on sanitized v0.4 PPA score records. It must not read raw model outputs, raw RTL, prompts, extracted RTL, logs, or real run directories.

Suggested module:

- `src/rtlbench/ppa_reporting.py`

Suggested scripts:

- `scripts/summarize_v0.4_ppa_results.py`

Optional script name for later, if useful as a stub only:

- `scripts/export_v0.4_ppa_results.py`

If an export script is added in this stage, it must operate only on already-sanitized records or synthetic test fixtures. Do not parse real benchmark outputs yet.

## 4. Required Reporting Behavior

The reporting layer should support:

- loading sanitized JSONL score records
- rejecting records with forbidden raw/private fields
- validating schema version and required fields
- writing sanitized CSV summaries
- rendering Markdown summaries
- grouping by benchmark, model, prompt profile, and scoring mode
- counting total, valid, and invalid samples
- counting gate pass rates
- counting failure categories
- computing valid-only mean/median optimization score
- computing all-sample mean score with invalid samples mapped to zero
- separating `area_power` and `area_only` modes
- reporting task-level wins/ties/regressions when a control profile is provided

## 5. Suggested Report Columns

Per group summary CSV should include at least:

- benchmark
- model
- prompt_profile
- scoring_mode
- samples
- valid_scores
- invalid_scores
- extraction_pass_count
- compile_pass_count
- synth_pass_count
- equiv_pass_count
- timing_pass_count
- mean_valid_score
- median_valid_score
- mean_all_sample_score
- failure_categories

If task-level comparison is implemented, include:

- control_profile
- task_wins
- task_ties
- task_regressions

## 6. Required Tests

Add tests for:

- loading valid sanitized JSONL
- rejecting forbidden fields such as raw RTL, prompt, model response, logs, and API keys
- missing required fields
- schema version mismatch
- area-only and full-PPA rows kept separate
- valid-only mean/median
- all-sample mean with invalid mapped to zero
- failure category counts
- Markdown rendering
- CSV rendering
- optional control-profile wins/ties/regressions if implemented

Suggested test file:

- `tests/test_ppa_reporting.py`

Synthetic fixtures are allowed only if they contain sanitized records and no raw/private content.

## 7. Commit Rules

Allowed:

- PPA reporting module
- PPA summary script
- tests and sanitized test fixtures
- README/docs usage notes
- spec updates

Forbidden:

- `outputs/`
- `raw_responses/`
- `extracted_rtl/`
- `logs/`
- `error_logs/`
- generated RTL code
- model responses
- prompts containing private tasks
- API keys or secrets
- caches, virtual environments, model weights

## 8. Non-goals

Do not:

- run v0.4 model jobs
- run full benchmarks
- parse real benchmark output directories
- integrate with RTL-OPT run outputs yet
- add new models
- change v0.1/v0.2 frozen artifacts
- change v0.3 reports
- claim PPA improvement
- implement SiliconCraft private benchmarks yet

## 9. Definition of Done for Next Codex Task

The next task is done when:

1. Sanitized PPA reporting module exists.
2. Summary script exists or existing tooling supports sanitized v0.4 PPA summaries.
3. Tests cover loading, validation, aggregation, Markdown, and CSV rendering.
4. Forbidden raw/private fields are rejected.
5. Area-only and full area/power modes are reported separately.
6. Invalid samples are included in all-sample scoring as zero.
7. `python -m pytest` passes.
8. No benchmark runs or raw outputs are committed.
9. The final summary explains what remains before integrating real RTL-OPT outputs.
