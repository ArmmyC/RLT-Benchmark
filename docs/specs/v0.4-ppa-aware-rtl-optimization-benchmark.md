## 1. Goal

Fix and preflight the Yosys techmap data-file issue found during the first v0.4 RTL-OPT PPA control validation.

The v0.4 pipeline has reached the point where scoring, reporting, export, and sanitized control artifacts exist. The current blocker is not `low_area_low_power` and not the PPA exporter. ## 3. Diagnosis Result

The first five-task `neutral_baseline` control validation produced:

- samples: 5
- valid scores: 0
- invalid scores: 5
- extraction pass count: 5
- compile pass count: 0
- synthesis pass count: 0
- equivalence pass count: 0
- scoring mode: `area_only`
- sanitized failure categories: `compile_failure=5`
- raw run failure categories: `synthesis_failure=5`

The diagnosis report concluded that the likely root cause is a local Yosys/OSS CAD Suite data-directory issue:

- the evaluator emitted a bare `techmap` command
- invoked Yosys looked for `share/techmap.v`
- the local install had `share/yosys/techmap.v`
- a direct probe succeeded when `techmap` was given the absolute `share/yosys/techmap.v` path

Therefore the generated RTL was not meaningfully tested for synthesis/equivalence. `low_area_low_power` remains blocked.

## 4. Next Stage

### Stage G: Yosys techmap preflight and evaluator fix

Status: next.

Add a small, tested Yosys data-file resolution/preflight layer and update the Yosys evaluator so the generic synthesis script can use an explicit `techmap.v` path when needed.

This task should not run new model jobs. It should make the evaluator robust enough that a later Stage H can rerun the same five-task neutral control.

## 5. Required Behavior

Implement a helper that can resolve a usable Yosys `techmap.v` file in a portable way.

Recommended resolution order:

1. explicit environment variable, for example `RTLBench_YOSYS_TECHMAP` or `YOSYS_TECHMAP`
2. `YOSYS_DATDIR/techmap.v` if `YOSYS_DATDIR` is set
3. common OSS CAD Suite layouts near the Yosys executable, including `../share/yosys/techmap.v` and `../share/techmap.v`
4. fallback to bare `techmap` if no explicit file is found

Evaluator behavior:

- if a techmap file is found, emit `techmap -map <absolute path>; opt`
- if no file is found, keep the current bare `techmap; opt` behavior
- paths in generated Yosys scripts should be absolute and shell-safe enough for spaces
- log/report should make it clear whether explicit techmap or fallback bare techmap was used

Add a preflight script that checks whether the configured Yosys can run a minimal `techmap` flow before model calls.

Suggested script:

- `scripts/check_yosys_techmap.py`

The preflight should:

- accept `--yosys` path, default `yosys`
- write a tiny synthetic Verilog module in a temporary directory
- generate a small Yosys script using the same techmap resolution helper
- run Yosys with timeout
- print PASS/FAIL and the chosen techmap mode
- not require any benchmark checkout or model endpoint

## 6. Required Tests

Add tests for:

- explicit env var techmap path resolution
- `YOSYS_DATDIR` resolution
- OSS CAD Suite-style `share/yosys/techmap.v` resolution near the executable
- fallback to bare `techmap` when no file exists
- generated Yosys script uses `techmap -map` when a file is resolved
- generated Yosys script preserves bare `techmap` fallback when no file is resolved
- preflight script argument parsing or core command construction without requiring a real Yosys install

Suggested test file:

- `tests/test_yosys_techmap.py`

## 7. Commit Rules

Allowed:

- evaluator/helper changes
- Yosys preflight script
- tests
- docs/spec updates only if needed

Forbidden:

- `outputs/`
- `raw_responses/`
- `extracted_rtl/`
- `logs/`
- `error_logs/`
- generated RTL code from model runs
- model responses
- prompts containing private tasks

- caches, virtual environments, model weights

Do not rerun the five-task neutral PPA control in this stage unless the user explicitly asks. The default next step after this fix is a separate Stage H rerun.
