# Feature Spec: v0.4 PPA-Aware RTL Optimization Benchmark

## 1. Goal

Design the next benchmark layer for evaluating area/power-oriented RTL generation while preserving functional correctness.

v0.3 showed that prompt changes alone were not enough to justify changing the default prompt profile. The `low_area_low_power` prompt remains an interesting hypothesis, but it should not be promoted without an equivalence-aware PPA scoring flow.

v0.4 should therefore focus on benchmark and scoring infrastructure, not on new model comparison.

## 2. Background

Completed v0.3 decision:

- keep `neutral_baseline` as default for `qwen36-27b`
- keep `strict_rtl_only` as optional challenger
- hold back `low_area_low_power`
- hold back `no_explanation_code_only`

Reason:

- smoke: `neutral_baseline` and `strict_rtl_only` tied at 7/9
- larger validation: `neutral_baseline` and `strict_rtl_only` tied at 31/40
- `low_area_low_power` did not show enough evidence without PPA-aware scoring

## 3. Research Question

Can an LLM generate RTL that is functionally equivalent to the reference while improving area and/or power under controlled synthesis assumptions?

The primary objective is area and power. Timing is a constraint/gate, not the reward.

## 4. Scope

Stage 1 should create the v0.4 design and scaffolding only. Do not run new model jobs yet.

Target benchmark family:

- RTL-OPT first, because it already supports equivalence-style checks
- later SiliconCraft/RFID private tasks, after public flow is validated

Target model:

- `qwen36-27b` only for initial v0.4 infrastructure validation

Prompt profiles to keep available:

- `neutral_baseline`
- `strict_rtl_only`
- `low_area_low_power`

Do not use `no_explanation_code_only` unless a later extraction-specific reason appears.

## 5. Required Scoring Design

A candidate RTL sample should receive PPA credit only if it passes all required gates:

1. extraction succeeds
2. syntax/compile succeeds
3. synthesis succeeds
4. equivalence succeeds
5. timing constraint check succeeds when timing data is available

If any required gate fails, the optimization score should be zero or marked invalid.

Recommended first scoring formula:

```text
score = 0.5 * area_score + 0.5 * power_score
area_score = min(reference_area / generated_area, 2.0)
power_score = min(reference_power / generated_power, 2.0)
```

Timing should be reported as pass/fail or slack, not directly rewarded in the first version.

If power data is unavailable in the public flow, v0.4 may start with area-only scoring plus explicit `power_unavailable` status. Do not fake power results.

## 6. Stage A: v0.4 PPA Design Report

Status: next.

Create:

- `docs/experiments/v0.4_ppa_aware_rtl_optimization.md`
- `reports/v0.4_ppa_scoring_design.md`

The design report should define:

- gates before optimization scoring
- metrics and units
- area score
- power score
- timing treatment
- invalid/failure handling
- public RTL-OPT limitations
- how this differs from v0.3 prompt experiments
- why `low_area_low_power` should be retested only after this scoring exists

## 7. Stage B: Scaffolding Plan

The design report should propose, but not necessarily implement, scripts such as:

- `scripts/export_v0.4_ppa_results.py`
- `scripts/summarize_v0.4_ppa_results.py`

Suggested sanitized fields:

- benchmark
- task_id
- model
- prompt_profile
- final_pass
- extraction_pass
- compile_pass
- synth_pass
- equiv_pass
- timing_pass
- reference_area
- generated_area
- reference_power
- generated_power
- area_score
- power_score
- optimization_score
- failure_category
- source_run_id

Do not include raw RTL, raw model responses, logs, prompts, API keys, or private task text.

## 8. Commit Rules

Allowed:

- v0.4 design docs
- report docs
- README links
- tests only if adding actual helper code

Forbidden:

- `outputs/`
- `raw_responses/`
- `extracted_rtl/`
- `logs/`
- `error_logs/`
- generated RTL code
- model responses
- prompts containing private tasks
- API keys or secrets
- caches, virtual environments, model weights

## 9. Non-goals

Do not:

- run v0.4 model jobs
- run full benchmarks
- add new models
- change v0.1/v0.2 frozen artifacts
- change v0.3 reports
- claim PPA improvement without synthesis/equivalence evidence
- implement SiliconCraft private benchmarks yet

## 10. Definition of Done for Next Codex Task

The next task is done when:

1. v0.4 experiment design doc exists.
2. v0.4 PPA scoring design report exists.
3. README or docs index links the v0.4 design.
4. The report clearly defines scoring gates and invalid handling.
5. The report states what can and cannot be measured with the current public flow.
6. The report explains how `low_area_low_power` will be retested later.
7. `python -m pytest` passes.
8. No benchmark runs or raw outputs are committed.
