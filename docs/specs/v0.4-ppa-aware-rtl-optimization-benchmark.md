## 4. Next Stage

### Stage F: Control compile-failure diagnosis

Status: next.

Create a diagnostic report that explains why the 5-task neutral control produced compile failures and recommends the smallest safe fix or rerun plan.

The report may inspect local, uncommitted run outputs and logs, but committed artifacts must remain sanitized. Do not commit raw responses, extracted RTL, logs, prompts, generated RTL, or private data.

## 5. Required Diagnostic Questions

Answer:

1. Were failures caused by model-generated RTL, extraction, wrong evaluator/config, RTL-OPT benchmark setup, or export mapping?
2. Did the `results.jsonl` rows contain enough structured information to distinguish compile vs synthesis vs equivalence failure?
3. Were generated-area metrics missing because synthesis failed, because the wrong evaluator ran, or because export selected the wrong metric source?
4. Is the fix a code/config issue or simply weak model output on this task subset?
5. What is the safest next control rerun configuration?

## 7. Commit Rules

Allowed:

- sanitized diagnostic Markdown/CSV reports
- docs/spec updates only if needed
- small helper/test fixes only if needed

Forbidden:

- `outputs/`
- `raw_responses/`
- `extracted_rtl/`
- `logs/`
- `error_logs/`
- generated RTL code
- model responses
- prompts containing private tasks
- API keys or secrets
- caches, virtual environments, model weights

## 8. Non-goals

Do not:

- run `low_area_low_power`
- run `strict_rtl_only`
- run full benchmarks
- add new models
- commit raw logs or generated RTL
- integrate SiliconCraft private benchmarks
- change v0.1/v0.2 frozen artifacts
- change v0.3 reports
- claim power improvement
- claim final silicon PPA quality

A tiny neutral control rerun is allowed only if needed to verify a config/helper fix, and only sanitized v0.4 artifacts/reports may be committed.

## 9. Definition of Done for Next Codex Task

The next task is done when:

1. Control diagnosis report exists.
2. It explains the 0/5 compile-pass result.
3. It identifies whether the issue is model output, extraction, evaluator/config, benchmark setup, or export mapping.
4. It recommends the next safe control rerun or fix.
5. It explicitly blocks `low_area_low_power` until neutral control produces valid PPA records.
6. `python -m pytest` passes.
7. No raw outputs are committed.
