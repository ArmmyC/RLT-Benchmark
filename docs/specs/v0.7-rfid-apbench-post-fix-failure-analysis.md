# Feature Spec: v0.7 RFID-APBench Post-Fix Failure Analysis

## Goal

Classify the remaining invalid pattern in the v0.7 post-fix expanded 10-task RFID-APBench baseline.

The post-fix expanded baseline ran successfully across 10 tasks × 3 samples. Response reliability improved strongly: 30/30 rows were non-empty, parsed HTTP `2xx`, and stopped normally. 27/30 rows were valid. The only remaining invalid rows were concentrated in `ap_006_wakeup_edge_filter`: two simulation failures and one compile/synthesis failure.

This spec should analyze the `ap_006` failures using sanitized evidence only. It is not a benchmark modification, model comparison, prompt-profile experiment, private evaluation, fine-tuning, max-token experiment, or training-data creation.

## Required Scope

Create a sanitized failure-analysis report focused on `ap_006_wakeup_edge_filter` only.

The report should:

1. Summarize the v0.7 post-fix expanded baseline results.
2. Confirm that response-boundary reliability is no longer the issue: no empty/null-content/token-limit rows.
3. Summarize the `ap_006` row outcomes: non-empty, extraction passed, one compile/synthesis failure, two simulation failures, zero valid rows.
4. Inspect the public `ap_006` task specification, prompt, reference, and testbench to summarize expected behavior at a high level.
5. Classify the observed `ap_006` failures using sanitized categories, without committing raw generated RTL, raw responses, simulator logs, synthesis logs, VCDs, or compiled artifacts.
6. Determine whether evidence supports a task/prompt clarity issue, candidate implementation issue, extractor issue, tool issue, evaluator issue, or benchmark asset bug.
7. Keep benchmark assets unchanged unless a concrete public benchmark bug is found.
8. Keep fine-tuning out of scope and not recommended.
9. Recommend exactly one next bounded direction.

## Optional Local Inspection

Ephemeral local inspection of ignored generated candidates, simulator logs, or synthesis logs is allowed only to classify the failure. Do not commit raw contents, paths, hashes, code snippets, waveforms, logs, VCD contents, or generated RTL. Report only sanitized categories and high-level behavior.

No live model rerun is required for this spec. If a bounded diagnostic is performed anyway, it must be limited to `ap_006_wakeup_edge_filter`, `neutral_baseline`, `qwen36-27b`, `max_tokens=4096`, fixed thinking-disabled runner path, at most 3 samples, and sanitized outputs only.

## Decision Rules

Recommend exactly one:

- `ap006_prompt_or_task_clarity_spec` if public task wording ambiguity is supported and can be clarified without changing reference behavior.
- `ap006_targeted_validation_spec` if more sanitized `ap_006` samples are needed after classification.
- `v0.7_rfid_apbench_release_closeout_spec` if the failure is understood enough and should not block release closeout.
- `benchmark_bugfix_spec` only if a concrete reference/testbench/task asset bug is found.
- `runner_bugfix_spec` only if a runner/reporting/evaluation defect is found.
- `no_immediate_followup_beyond_documentation` if no further bounded work is justified.

Do not recommend fine-tuning, private evaluation, model comparison, max-token increase, prompt-profile experiments, broad rerun, or training-data creation from this analysis alone.
