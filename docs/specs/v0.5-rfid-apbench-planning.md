# Feature Spec: v0.5 RFID-APBench Planning

## Goal

Plan a public/synthetic RFID-style area/activity benchmark for RTLBench. This is benchmark-only and must not add private SiliconCraft RTL or fine-tuning code.

## Current Status

This spec is still incomplete in the repository.

The unrelated Windows clean-portability documentation that was accidentally added to this repo has been removed. The next Codex task must implement only this v0.5 RFID-APBench planning package unless the user explicitly changes direction.

## Required Deliverables

Create exactly these v0.5 planning deliverables:

- `docs/experiments/v0.5_rfid_apbench_planning.md`
- `reports/v0.5_area_activity_scoring_design.md`
- `docs/specs/rfid_apbench_task_schema.md`
- `docs/specs/fine_tuning_repo_boundary.md`

Do not create unrelated portability, Makefile, clean-flow, dashboard, model-serving, or fine-tuning specs as part of this task.

## Scope

Define `RFID-APBench`, a public/synthetic benchmark for RFID/NFC-style RTL tasks where AP means area plus activity. Activity is a VCD toggle-count proxy, not measured silicon power.

This follows the project direction that RTLBench should be the measurement foundation before any RTL-specialist fine-tuning work begins.

## Requirements

The planning documents must define:

1. Public/synthetic task folder schema.
2. `area_activity` scoring mode.
3. VCD toggle-count measurement requirements.
4. Sanitized output fields for area/activity records.
5. Five initial synthetic RFID/AP task proposals.
6. Pass/fail gates for correctness, synthesis, area metric availability, and activity metric availability.
7. Boundary between RTLBench and a future fine-tuning repository.
8. Decision note that fine-tuning starts only after RFID-APBench can score 5 to 10 tasks end-to-end.

## Area/Activity Scoring

Use conceptual scoring:

```text
area_score = min(reference_area / generated_area, 2.0)
activity_score = min(reference_activity / generated_activity, 2.0)
score = 0.5 * area_score + 0.5 * activity_score
```

Invalid samples count as zero in aggregate comparisons.

Required gates:

1. extraction
2. compile
3. simulation or equivalence
4. synthesis
5. timing if required
6. area metric availability
7. activity metric availability

## VCD Toggle-Count Planning Requirements

The planning docs must describe:

- how a task testbench emits VCD
- how `workload.json` identifies stimulus scenarios
- how reset and initialization windows are excluded or marked
- whether all signals or selected public signals are counted
- how reference and generated RTL are compared under the same workload
- how missing VCD, malformed VCD, zero activity, or unavailable toggle data are reported
- why this is an activity proxy, not measured power

## Initial Synthetic Task Proposals

Document five public/synthetic tasks:

- `ap_001_idle_counter`
- `ap_002_command_decoder`
- `ap_003_register_bank_unnecessary_writes`
- `ap_004_crc_serial_parallel_tradeoff`
- `ap_005_fsm_controller_idle_activity`

For each, include behavior, optimization opportunity, correctness gate, area/activity expectations, timing requirement, and RFID/NFC relevance.

## Sanitized Artifact Fields

The planning docs must define sanitized fields for area/activity records, including at least:

```yaml
schema_version: v0.5
benchmark: rfid_apbench
task_id: string
model: string
prompt_profile: string
source_run_id: string
final_pass: bool
extraction_pass: bool
compile_pass: bool
sim_or_equiv_pass: bool
synth_pass: bool
timing_status: not_required | unavailable | pass | fail
reference_area: number | null
generated_area: number | null
area_unit: generic_cells | mapped_area | null
reference_activity: number | null
generated_activity: number | null
activity_unit: rtl_vcd_toggle_count | null
activity_source: string | null
area_score: number | null
activity_score: number | null
optimization_score: number | null
scoring_mode: area_activity
score_status: valid | invalid
failure_category: string
toolchain_id: string
```

The schema must explicitly forbid raw prompts, raw RTL, raw model responses, error logs, secrets, private task text, training datasets, and model adapters.

## Repository Boundary

RTLBench owns:

- benchmarks and task definitions
- adapters
- evaluation gates
- area/activity scoring
- sanitized artifacts
- reports

The future fine-tuning repo owns:

- dataset construction from public/sanitized artifacts
- QLoRA/DoRA scripts
- adapter checkpoints
- model routing experiments
- post-training evaluation calls into RTLBench

Do not put training code or private data into RTLBench.

## Forbidden

Do not commit:

- private RTL
- private task text
- raw prompts
- raw model responses
- generated RTL
- benchmark outputs
- logs or error logs
- secrets
- model weights
- training datasets
- LoRA/QLoRA/DoRA adapters

## Definition of Done

Done when all required planning docs exist, `area_activity` scoring is specified, VCD toggle flow is specified, sanitized fields are specified, five synthetic tasks are proposed, private RTL is explicitly out of scope, fine-tuning is gated on a benchmark MVP, and no raw/private/training artifacts are committed.
