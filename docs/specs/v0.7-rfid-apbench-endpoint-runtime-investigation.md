# Feature Spec: v0.7 RFID-APBench Endpoint Runtime Investigation

## Goal

Investigate the endpoint/runtime behavior behind repeated parsed-success, null-content, token-limit responses observed in the RFID-APBench empty-response thread.

The v0.7 prompt-profile rejected closeout recommends `endpoint_runtime_investigation_spec` because both `neutral_baseline` and `completion_reliability` repeatedly produced successful HTTP `2xx`, parsed responses with one choice, null content, zero response characters, `finish_reason=length`, and 4096 completion tokens. The bounded prompt-profile challenger did not reduce this behavior.

This investigation should determine whether the issue is caused by runtime configuration, tokenizer/chat-template behavior, hidden reasoning/thinking mode, max-token limits, OpenAI-compatible response normalization, or another serving boundary condition.

This is endpoint/runtime diagnosis only. It is not benchmark modification, model comparison, broad rerun, prompt-profile tuning, private evaluation, fine-tuning, or training-data creation.

- `reports/v0.7_rfid_apbench_bounded_prompt_profile_experiment.csv`

## Required Scope

Create a sanitized endpoint/runtime investigation report that:

1. Summarizes the observed null-content token-limit behavior.
2. Identifies endpoint/runtime hypotheses that could produce parsed `2xx` null content with `finish_reason=length` and full completion tokens.
3. Checks current model configuration for relevant flags such as chat-template thinking controls, max token limits, extra body, and endpoint compatibility assumptions.
4. Determines whether raising `max_tokens` should be tested as a bounded diagnostic, not adopted globally.
5. Determines whether a raw-response inspection is needed locally, while keeping committed output sanitized.
6. Recommends exactly one next bounded step.
7. Keeps benchmark assets, task prompts, evaluator, scoring, and fine-tuning out of scope.

## Optional Bounded Diagnostics

A live diagnostic is allowed only if it is narrowly scoped and sanitized:

- tasks: at most `ap_010_retry_timeout_fsm` and optionally `ap_001_idle_counter`
- samples: at most 1 sample per task per setting
- profiles: `neutral_baseline` only, unless existing evidence requires otherwise
- max tokens: compare current 4096 against one larger value, for example 8192, only if endpoint/model supports it
- no benchmark-wide run
- no model comparison
- no committed raw response bodies or generated RTL

If a diagnostic is performed, commit only sanitized findings such as request outcome, finish reason, content presence, character count, token counts, latency, and whether the endpoint accepted the requested max-token value.

## Report Requirements

The Markdown report must include:

- scope and non-goals
- source evidence used
- observed null-content token-limit pattern
- endpoint/model configuration summary without secrets
- hypothesis table
- whether `max_tokens` increase is technically possible and safe to test
- whether thinking/chat-template configuration needs verification
- whether raw local inspection is needed and how it remains uncommitted
- benchmark asset decision
- fine-tuning readiness decision
- exactly one recommended next step
- activity proxy caveat
- public synthetic / no private / not fine-tuning caveat

## Decision Rules

Recommend exactly one:

- `bounded_max_tokens_diagnostic_spec` if evidence supports testing whether a higher output budget changes null-content behavior.
- `thinking_mode_configuration_bugfix_spec` if current configuration appears to allow hidden reasoning/thinking token consumption despite intended disabling.
- `client_response_capture_observability_spec` if the client still lacks enough sanitized metadata to diagnose null-content responses.
- `serving_stack_issue_spec` if the problem appears to be endpoint/server behavior outside RTLBench code.
- `no_immediate_followup_beyond_documentation` if there is no safe bounded diagnostic.

Do not recommend fine-tuning, private evaluation, benchmark modification, broad model comparison, or training-data creation.

## Definition of Done

Done when:

1. `reports/v0.7_rfid_apbench_endpoint_runtime_investigation.md` exists.
2. It explains the parsed-success null-content token-limit pattern.
3. It reviews endpoint/model configuration without secrets.
4. It decides whether a bounded max-token or runtime diagnostic is justified.
5. It keeps benchmark assets unchanged.
6. It states fine-tuning is not recommended from this evidence alone.
7. It recommends exactly one next bounded step.
8. `python -m pytest` passes.
9. No raw/private/model-output/training artifacts are committed.
