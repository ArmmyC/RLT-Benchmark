# v0.7 RFID-APBench Runner And Endpoint Observability

## Scope

This change adds sanitized request and response observability to the OpenAI-compatible client and RFID-APBench 3-sample runner. It addresses the v0.7 targeted investigation finding that existing rows could identify a successful request followed by empty text but could not distinguish timeout, retry behavior, HTTP failure class, response parsing, response shape, null versus empty content, token usage, or client normalization metadata.

No model baseline or targeted empty-response rerun was performed.

## Non-Goals

- No benchmark task, prompt, prompt profile, reference, testbench, workload, metric, extractor, evaluator, or scoring formula is changed.
- No live endpoint is required by the tests.
- No model comparison, prompt tuning, private evaluation, training-data creation, fine-tuning, or adapter work is performed.
- No raw prompt, response text, response body, generated RTL, simulator or synthesis log, VCD, compiled artifact, secret, model weight, or dataset is committed.

## Source Investigation

The targeted empty-response investigation found six expanded-run empty rows concentrated in `ap_001_idle_counter` and `ap_010_retry_timeout_fsm`. All six had an available endpoint, completed generation status, failed extraction, and `empty_response`, but the committed schema did not preserve enough accepted-request metadata to attribute the empty text.

The client already measured latency and received endpoint usage data, but the 3-sample runner did not propagate those values. The client also normalized null content to an empty string and discarded response choice count and finish reason. Request exceptions were retried and collapsed into a generic runtime error.

## Stable Observability Schema

The following fields are now present in every sanitized CSV and JSONL row. Unavailable numeric or boolean values serialize as empty CSV cells and JSON `null`; categorical fields use `unavailable` or `not_attempted` where appropriate.

| field | meaning |
| --- | --- |
| `request_outcome` | Sanitized success, timeout, HTTP error, transport error, response parse/schema failure, endpoint blocker, tool blocker, or unavailable outcome. |
| `request_attempt_count` | Number of client attempts including retries; zero means no generation request was attempted. |
| `latency_seconds` | Elapsed client time across attempts, when measured. |
| `response_choice_count` | Number of response choices when a choices collection is available. |
| `response_content_present` | Whether the first choice carried a non-null content value. |
| `response_character_count` | Character count after safe null-to-empty normalization; response text is not retained in reports. |
| `finish_reason` | Endpoint-provided finish reason when it is a string. |
| `prompt_tokens` | Endpoint-provided prompt token count when available. |
| `completion_tokens` | Endpoint-provided completion token count when available. |
| `total_tokens` | Endpoint-provided total token count when available. |
| `http_status_class` | Sanitized status class such as `2xx`, `4xx`, or `5xx`; response bodies are excluded. |
| `response_parse_status` | `parsed`, `json_error`, `schema_error`, `not_attempted`, or `unavailable`. |

The schema is additive. Existing 10-task manifest loading, exactly-three-samples enforcement, extraction, evaluation, and scoring behavior remain unchanged.

## Request Outcome Semantics

| condition | `request_outcome` | supporting metadata |
| --- | --- | --- |
| successful non-empty text | `success_nonempty` | Text contains at least one non-whitespace character; choice, content, finish, usage, latency, status class, and parse metadata are recorded where available. |
| successful empty text | `success_empty` | Text is empty or whitespace-only; character count and content presence distinguish zero-length, whitespace-only, empty-string, and null-content cases. |
| timeout | `timeout` | Attempt count and elapsed latency are retained; parse status is `not_attempted`. |
| HTTP error | `http_error` | HTTP status class is retained without status body or endpoint secrets. |
| transport failure | `transport_error` | Attempt count and latency are retained without raw exception or network details. |
| invalid JSON response | `response_parse_failure` | HTTP success class is retained and parse status is `json_error`. |
| invalid response shape | `response_schema_failure` | Available choice/content metadata is retained and parse status is `schema_error`. |
| endpoint unavailable or failed required preflight | `endpoint_unavailable` | Attempt count is zero and parse status is `not_attempted`. |
| missing or unhealthy EDA tool | `tool_health_blocker` | Attempt count is zero and parse status is `not_attempted`. |

Existing `failure_category` values remain compatible. The finer request distinction is carried by `request_outcome`, while downstream extraction and evaluation categories continue to describe candidate processing.

## Empty-Response Distinction

Future empty rows can now be interpreted without raw response content:

- `success_empty`, character count zero, content present, and completion tokens zero supports an endpoint-returned empty string or zero-token completion.
- `success_empty`, character count zero, and content not present identifies null content normalized safely by the client.
- `success_empty` with a positive character count identifies whitespace-only content.
- `response_schema_failure` with zero choices distinguishes a missing completion choice from a parsed empty content value.
- `response_parse_failure` distinguishes invalid JSON from a parsed response.
- `timeout`, `http_error`, and `transport_error` no longer collapse into an indistinguishable client runtime error.
- attempt count and latency expose retry behavior without retaining request or response content.

The Markdown report now includes a request-outcome distribution and metadata-availability counts. It does not display generated text, response bodies, prompts, authorization values, or raw diagnostics.

## Endpoint-Dependent And Unavailable Data

- Token counts and finish reason remain empty when the endpoint omits them.
- HTTP status class is intentionally coarse and does not retain response bodies.
- Response character count does not reveal content and cannot diagnose semantic quality.
- Even complete metadata cannot prove an internal model cause versus a serving implementation cause; it narrows the observable boundary.
- Unexpected exceptions outside the structured client categories remain a sanitized `request_failure` with unavailable detail.
- Historical v0.6 reports are preserved and are not backfilled with metadata that was not recorded at run time.

## Tests

Mocked tests cover:

- non-empty success metadata
- empty success with zero completion tokens
- timeout with retry count
- HTTP error class without response-body leakage
- transport failure
- invalid JSON response
- invalid response schema with zero choices
- propagation of empty-success and request-failure metadata into generation rows
- endpoint and tool blocker outcomes
- stable CSV and JSONL observability fields
- Markdown observability summary without raw response content
- existing 10-task and exactly-3-sample behavior

No test contacts a live model endpoint.

## Artifact And Privacy Policy

Only sanitized counts, booleans, categorical outcomes, coarse status class, latency, token usage, and finish reason may enter committed reports. Raw prompts, generated content, response bodies, authorization data, provider diagnostics, and artifact locations remain excluded. Raw generation and evaluation artifacts remain confined to ignored work and output directories during actual runs.

## Benchmark Asset Decision

Keep benchmark assets unchanged. This change improves request and response observability only; it provides no evidence for changing tasks, prompts, references, testbenches, workloads, extraction, evaluation, metrics, or scoring.

## Fine-Tuning Readiness

Fine-tuning remains out of scope and is not recommended from this observability work. Metadata plumbing is not training evidence and does not create a dataset or causal basis for model training.

## Recommended Next Step

Recommend exactly one bounded next step: `targeted_empty_response_observability_rerun_spec`.

That specification should permit only the two previously identified tasks, unchanged prompts and settings, and at most three samples per task. Its purpose should be to exercise the new sanitized fields and distinguish empty content, zero-token completion, response-shape failure, timeout, and retry behavior without a benchmark-wide run.

## Activity Proxy Caveat

Activity is a VCD toggle-count proxy from the declared public workload. It is not measured silicon power, signoff power, final silicon PPA, or production QoR.

## Public Synthetic Caveat

This work concerns only public/synthetic RFID-APBench infrastructure. It contains no private data or private RTL, is not private evaluation, and is not fine-tuning or training-readiness evidence.
