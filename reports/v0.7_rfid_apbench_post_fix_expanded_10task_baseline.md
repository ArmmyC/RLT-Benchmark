# v0.7 RFID-APBench Post-Fix Expanded 10-Task Baseline

## Scope And Non-Goals

This report runs the current public/synthetic RFID-APBench manifest after the v0.7 fixes: the RFID runner forwards the configured Qwen thinking-disable `extra_body`, and the `ap_001_idle_counter` public prompt includes the clarified same-cycle terminal-count/`expired` contract.

The run covers exactly 10 manifest tasks with exactly 3 samples per task, for 30 attempted samples total. It uses configured/default `qwen36-27b`, `neutral_baseline`, deterministic generation settings, `max_tokens=4096`, current benchmark assets, and unchanged extraction, evaluation, metrics, and scoring.

This is not a model comparison, private evaluation, prompt-profile experiment, benchmark modification, training-data creation, fine-tuning run, max-token experiment, or final silicon PPA claim.

Only sanitized reports are committed. Raw prompts beyond public task prompt text, raw response bodies, generated RTL, VCDs, simulator logs, synthesis logs, compiled artifacts, tool scratch, secrets, model weights, datasets, adapters, and fine-tuning scripts are excluded.

## Interpretation

The v0.7 fixes improved response reliability across the expanded public manifest: the earlier empty/null-content token-limit signature did not recur in any of the 30 rows, and every row produced extractable RTL. The clarified `ap_001` state also held in the expanded run, with `ap_001_idle_counter` improving from 0/3 valid in the earlier expanded evidence to 3/3 valid here.

Overall correctness/scoring improved at the all-sample level: valid rows increased from 22/30 to 27/30, functional simulation passes increased from 23/30 to 27/30, and the all-sample zero-filled score increased from 0.740037 to 0.855912. The mean valid score decreased from 1.009141 to 0.951014 because the set of valid rows changed and the run state differs; that value should not be read as a model comparison.

The remaining issue is narrow and interpretable: all invalid rows are in `ap_006_wakeup_edge_filter`, with two simulation failures and one compile failure. That suggests a post-fix failure-analysis follow-up, not an endpoint/tool blocker, runner bugfix, prompt-profile experiment, private evaluation, or fine-tuning step.

## Benchmark Asset Decision

Keep benchmark assets unchanged. The current public manifest loaded exactly 10 tasks, the clarified `ap_001` prompt was included through the manifest, and no evidence indicates a reference RTL, testbench, interface, workload, metrics, evaluator, or scoring bug. The `ap_006` failures should be classified from sanitized candidate behavior before any benchmark asset change is considered.

## Fine-Tuning Readiness Decision

Fine-tuning is not recommended. This public/synthetic 30-row baseline is an evaluation artifact, not a training dataset. It does not provide sufficient volume, diversity, governance basis, private-task evidence, or model-weight/adaptation justification.

## Recommended Next Bounded Direction

Recommend exactly one bounded direction: `post_fix_failure_analysis_spec`.

Rationale: the fixed expanded baseline ran successfully and response reliability is improved, but `ap_006_wakeup_edge_filter` has a concentrated 0/3 valid pattern that should be classified before release closeout.
