# v0.7 RFID-APBench Targeted 3-Sample Thinking-Disabled Validation

## Scope

This targeted validation confirms whether the fixed public/synthetic RFID-APBench runner reliably avoids the prior null-content token-limit symptom when `qwen36-27b` requests include `chat_template_kwargs.enable_thinking=false`.

The run is limited to exactly three samples each for:

- `ap_001_idle_counter`
- `ap_010_retry_timeout_fsm`

It uses `neutral_baseline`, unchanged deterministic generation settings, the existing 4096-token limit, and unchanged benchmark assets, extraction, evaluation, metrics, and scoring. It is not a benchmark-wide baseline, model comparison, prompt-profile experiment, private evaluation, fine-tuning step, or training-data creation effort.

## Thinking-Disable Confirmation

The run used the fixed RFID path. The effective `qwen36-27b` model resolved its existing preset and forwarded the following non-secret `extra_body` through `generate_samples()` and `OpenAICompatibleClient.generate()`:

```yaml
chat_template_kwargs:
  enable_thinking: false
```

Mocked payload tests prove this mapping reaches outgoing JSON while preserving the selected system prompt and `max_tokens=4096`. This live validation used that same code path; no raw request body was retained or committed.
