# v0.7 RFID-APBench Bounded Thinking-Disabled Validation

## Scope

This tiny post-bugfix validation tests whether the prior null-content token-limit symptom changes when the fixed public/synthetic RFID-APBench runner forwards `chat_template_kwargs.enable_thinking=false` for `qwen36-27b`.

The validation is limited to one sample each for:

- `ap_001_idle_counter`
- `ap_010_retry_timeout_fsm`

It uses `neutral_baseline`, unchanged deterministic settings, the existing 4096-token limit, and unchanged benchmark assets, extraction, evaluation, metrics, and scoring. It is not a benchmark-wide run, model comparison, prompt-profile experiment, private evaluation, fine-tuning step, or training-data creation effort.

## Thinking-Disable Confirmation

The validation used the fixed RFID runner path. The effective model `qwen36-27b` resolved the existing `configs/models.yaml` preset, and `generate_samples()` forwarded this non-secret `extra_body` through `OpenAICompatibleClient.generate()`:

```yaml
chat_template_kwargs:
  enable_thinking: false
```

Mocked payload tests prove this mapping reaches the outgoing JSON request while preserving the selected system prompt and `max_tokens=4096`. The live validation used that same code path; no raw request body was retained or committed.

## Pre-Bugfix Comparison

Pre-bugfix targeted evidence repeatedly showed parsed HTTP `2xx` responses with one choice, null content, zero response characters, `finish_reason=length`, and exactly 4096 completion tokens:

- The targeted observability rerun had 1/3 empty on `ap_001_idle_counter` and 3/3 empty on `ap_010_retry_timeout_fsm`.
- The fresh `neutral_baseline` control in the bounded prompt-profile experiment had 2/3 empty on `ap_001_idle_counter` and 3/3 empty on `ap_010_retry_timeout_fsm`.

Post-fix, both one-sample rows are non-empty, have `finish_reason=stop`, and consume 348 and 899 completion tokens. The observable null-content token-limit symptom changed on both targeted tasks in this bounded validation.

This two-row result supports the thinking-mode configuration fix as relevant to output completion behavior, but it is too small to establish a stable rate or broader model-performance claim. The correctness outcomes also differ by task and must not be conflated with response reliability.

## Benchmark Asset Decision

Keep benchmark tasks, task prompts, references, testbenches, workloads, extractor, evaluator, metrics, and scoring unchanged. The validation exercised only request configuration propagation, and it produced extractable candidates for both unchanged tasks.

## Fine-Tuning Readiness

Fine-tuning remains out of scope and is not recommended. Two public/synthetic validation rows are not a training dataset and provide no governance, coverage, or causal basis for model-weight or adapter changes.

## Recommended Next Step

Recommend exactly one bounded next step: `targeted_3sample_thinking_disabled_validation_spec`.

The one-sample validation improved the targeted null-content symptom on both tasks, so a three-sample-per-task confirmation is justified before any max-token diagnostic or broader work. It should retain `neutral_baseline`, `enable_thinking=false`, 4096 tokens, deterministic settings, the same two tasks, sanitized outputs, and unchanged benchmark/evaluation policy. It must not become a benchmark-wide run or model comparison.
