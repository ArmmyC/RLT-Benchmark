# v0.8 RFID-APBench Benchmark Hardening Plan

## Scope And Non-Goals

This report plans the v0.8 hardening cycle for the public/synthetic RFID-APBench benchmark after the v0.7 post-fix baseline closeout. It defines documentation, reproducibility, report hygiene, release automation, validation guardrail, limitation-tracking, and repository-boundary workstreams.

This is planning only. No model evaluation, model comparison, prompt-profile experiment, benchmark task edit, reference RTL edit, testbench edit, interface change, workload change, scoring change, evaluator policy change, max-token increase, private benchmark work, training-data creation, fine-tuning, adapter work, or tag creation was performed.

## Known Limitations

The remaining v0.7 invalid rows are localized to `ap_006_wakeup_edge_filter`:

| sample pattern | classification |
| --- | --- |
| one compile-invalid row | `syntax_or_compile_construct_error`; `incomplete_or_malformed_candidate` |
| two simulation-invalid rows | `edge_filter_semantics_mismatch`; `wakeup_pulse_width_mismatch`; `state_update_timing_mismatch` |

The failure analysis did not support endpoint, token-limit, extractor, EDA tool, evaluator, scoring, prompt-profile, or benchmark-asset defects. v0.8 should carry `ap_006` forward as a known candidate-implementation limitation in the v0.7 evidence, not as a v0.7 release blocker and not as justification for prompt edits or benchmark behavior changes.

## Hardening Workstreams

### 1. Benchmark Documentation

Immediate v0.8 tasks:

- Add or update RFID-APBench documentation pointers so `benchmarks/rfid_apbench/README.md` and top-level docs clearly state the public/synthetic scope, 10-task inventory, `area_activity` scoring mode, activity-proxy definition, and artifact policy.
- Document the approved `ap_001_idle_counter` clarity update as the only v0.7 benchmark asset change.
- Document the `ap_006_wakeup_edge_filter` limitation as a run-result limitation, not a benchmark defect.

Non-goals:

- Do not add tasks, modify existing task behavior, edit reference RTL, edit testbench behavior, change interfaces, or alter workloads.

### 2. Reproducibility

Immediate v0.8 tasks:

- Publish a user-facing reproduction path for the v0.7 post-fix baseline: required environment variables, deterministic settings, tool requirements, command shape, output destinations, and sanitized report expectations.
- Clarify `.env` and environment-variable handling for `QWEN_BASE_URL`, `QWEN_API_KEY`, `QWEN_MODEL`, `BENCHMARK_ROOT`, and `OUTPUT_ROOT`, while reiterating that secrets and absolute private paths must not be committed.
- Record expected toolchain health checks for Icarus compile, `vvp`, and Yosys synthesis.
- State that raw responses, extracted/generated RTL, VCDs, logs, compiled artifacts, and scratch directories remain under ignored output/work locations.

Non-goals:

- Do not require a live model endpoint in unit tests.
- Do not make the public baseline dependent on private infrastructure paths.

### 3. Report Hygiene

Immediate v0.8 tasks:

- Define stable report naming for RFID-APBench milestones, including Markdown, CSV, and JSONL when a run produces tabular rows.
- Define a minimal sanitized row schema expectation for request metadata, content/token metadata, extraction gates, evaluation gates, score fields, failure categories, tool status, and caveat text.
- Require source-evidence sections and report-to-report lineage in closeout documents.
- Add a checklist for excluding raw prompts beyond public benchmark prompt text, raw response bodies, generated RTL, logs, VCDs, compiled artifacts, endpoint credentials, and training artifacts.

Non-goals:

- Do not retroactively rewrite all historical reports unless a separate bounded report-hygiene spec requests it.

### 4. Release Automation

Immediate v0.8 tasks:

- Create a pre-tag release checklist that verifies the intended report set exists, tests pass, benchmark/config assets are unchanged except approved changes, and prohibited artifacts are absent from the tracked diff.
- Consider a lightweight script or CI assertion for release-readiness checks: report existence, manifest task count, sample-count guardrail tests, sanitized-output tests, and no-root-`activity.vcd` checks.
- Document the tag process for the closed v0.7 baseline without creating tags automatically.

Non-goals:

- Do not push tags, publish releases, or add broad CI that requires a live model endpoint without explicit follow-up.

### 5. Validation Guardrails

Immediate v0.8 tasks:

- Preserve the RFID baseline rule that benchmark baselines use exactly 3 samples per task.
- Keep the targeted validation exception explicit and bounded; the existing runner permits exactly 1 sample only with `--allow-single-sample-validation`.
- Preserve manifest task filtering behavior through explicit `--task-id` values and reject unknown or duplicate task IDs.
- Preserve model preset `extra_body` forwarding from `configs/models.yaml` into the OpenAI-compatible client request path.
- Preserve sanitized outputs as the commit boundary; raw response and generated RTL directories remain ignored artifacts.

Non-goals:

- Do not change prompt profiles, introduce `completion_reliability`, raise global max tokens, or alter deterministic defaults.

### 6. Known Limitation Tracking

Immediate v0.8 tasks:

- Add a known-limitation section in release-facing documentation that names `ap_006_wakeup_edge_filter` and links to the v0.7 failure-analysis report.
- Track the limitation as candidate behavior under one run, not as a benchmark asset defect.
- Require any future `ap_006` follow-up to be bounded and to preserve the unchanged prompt/reference/testbench/workload/evaluator unless new evidence proves an asset defect.

Non-goals:

- Do not perform another `ap_006` evaluation as part of this planning report.
- Do not modify `ap_006` prompt, reference RTL, or testbench based on the current evidence.

### 7. Repository Boundaries

Immediate v0.8 tasks:

- Keep RFID-APBench benchmark assets, sanitized reports, harness code, and tests in RTLBench.
- Keep serving orchestration, model hosting, private SiliconCraft RTL, private benchmarks, datasets, adapters, model weights, and fine-tuning scripts out of this repository.
- Reiterate that raw outputs belong under ignored output/work directories and should not become committed evidence.

Non-goals:

- Do not create training repositories, private benchmark directories, model adapters, or fine-tuning scripts from this evidence.

## Release And Tag Checklist Proposal

Before tagging a public RFID-APBench baseline:

- Confirm the intended release closeout report exists and states release readiness.
- Confirm the tag name is recorded; for v0.7, use `v0.7-rfid-apbench-post-fix-baseline`.
- Confirm `python -m pytest` passes.
- Confirm `git status --short --ignored` shows no unexpected tracked raw artifacts.
- Confirm `git diff -- benchmarks/rfid_apbench configs` is empty except for explicitly approved benchmark/config changes.
- Confirm no root `activity.vcd` or other VCD file is tracked.
- Confirm reports contain source evidence, endpoint status without secrets, deterministic settings, model, prompt profile, max tokens, gate counts, score summaries, limitations, and caveats.
- Confirm the release notes do not claim measured power, final silicon PPA, private evaluation, model superiority, or fine-tuning readiness.
- Confirm raw model responses, generated RTL, raw logs, VCDs, compiled artifacts, scratch directories, secrets, model weights, datasets, adapters, and fine-tuning scripts are not staged.

Suggested manual tag commands after the checklist passes:

```powershell
git tag v0.7-rfid-apbench-post-fix-baseline
git push origin v0.7-rfid-apbench-post-fix-baseline
```

The checklist should be implemented before it becomes release automation.

## Reproducibility Checklist Proposal

For a reproducible RFID-APBench baseline run:

- Record repository commit or release tag.
- Record benchmark manifest version and task count.
- Record model preset and served model name.
- Record endpoint base URL without credentials.
- Record `QWEN_BASE_URL`, `QWEN_MODEL`, `BENCHMARK_ROOT`, and `OUTPUT_ROOT` usage without printing secrets.
- Record prompt profile and system prompt name; for v0.7, use `neutral_baseline`.
- Record temperature `0.0`, top-p `1.0`, max tokens `4096`, and samples per task `3`.
- Confirm `configs/models.yaml` forwards model preset `extra_body` where required.
- Confirm Icarus compile, `vvp`, and Yosys synthesis health.
- Use a timestamped output root and a separate temporary work directory.
- Commit only sanitized Markdown, CSV, and JSONL reports.
- Preserve raw responses, generated/extracted RTL, logs, VCDs, compiled artifacts, and tool scratch only under ignored paths.

## Report Hygiene Checklist Proposal

For each committed RFID-APBench report:

- Use a stable versioned filename under `reports/`.
- Include scope and non-goals.
- Include source evidence and report lineage.
- Include model, prompt profile, generation settings, endpoint/tool status without secrets, and benchmark manifest scope.
- Include request outcome distribution and response/content/token metadata when available.
- Include extraction, compile, functional simulation, synthesis, area, activity, and valid-score gates.
- Separate functional correctness from compile-only or synthesis-only results.
- Include per-task summaries for multi-task runs and per-sample metadata for targeted validations.
- Include failure categories and known limitations.
- Include valid-score and all-sample zero-filled score summaries when scores exist.
- Include benchmark asset and fine-tuning readiness decisions.
- Include activity-proxy and public synthetic/no-private/not-fine-tuning caveats.
- Exclude raw response bodies, generated RTL, logs, VCD snippets, compiled artifacts, secrets, private paths, model weights, datasets, adapters, and fine-tuning scripts.

## Fine-Tuning Readiness Decision

Fine-tuning remains out of scope and is not recommended. The v0.7 evidence is public/synthetic benchmark evaluation and release-hardening evidence, not a governed training dataset or sufficient basis for model-weight changes, LoRA/QLoRA/DoRA adapters, or private-task adaptation.

## Recommended Next Bounded Direction

Recommend exactly one bounded next direction: `v0.8_release_checklist_spec`.

Rationale: v0.7 is already release-ready, and the most useful immediate hardening step is to turn the manual release/tag checklist into an explicit, reviewable artifact before adding broader documentation, report-hygiene automation, or CI assertions.

## Activity Proxy Caveat

Activity is a VCD toggle-count proxy from the declared public workload. It is not measured silicon power, signoff power, final silicon PPA, or production QoR.

## Public Synthetic Caveat

This plan covers only public/synthetic RFID-APBench evidence. It contains no private data or private RTL, is not private evaluation, does not create training data, and is not fine-tuning or training-readiness evidence.
