# v0.4 PPA Control Diagnosis

This diagnosis covers the first five-task public RTL-OPT `neutral_baseline` control run for `qwen36-27b`. It uses sanitized committed score records plus local run metadata and evaluator logs. No raw prompts, model responses, generated RTL, extracted RTL paths, or error-log paths are included here.

## Summary

The 0/5 compile-pass result was most likely caused by a local Yosys/OSS CAD Suite data-directory issue, not by the sanitized export layer and not by proven weak model RTL on this subset.

All five samples reached extraction and entered the configured `yosys_equiv` evaluator. The evaluator first runs generic Yosys synthesis before equivalence. In every sample, Yosys failed while loading its standard `techmap.v` support file. The local Windows OSS CAD Suite layout contains `share/yosys/techmap.v`, but the invoked Yosys binary looked for `share/techmap.v`. A direct toolchain probe on original public RTL-OPT input reproduced the failure with plain `techmap`; the same probe succeeded when `techmap` was given the absolute `share/yosys/techmap.v` path.

Therefore the generated RTL was not meaningfully tested for synthesis or equivalence in this run. `low_area_low_power` remains blocked.

## Diagnostic Answers

1. The failures were most likely caused by evaluator/toolchain configuration on Windows OSS CAD Suite. Extraction passed for all samples, RTL-OPT benchmark setup was sufficient to load tasks and reference metrics, and export mapping operated on the structured fields it received. Model output quality is not proven either way because Yosys failed before generated designs could be synthesized.

2. `results.jsonl` contained enough structured information to identify the evaluator stage at a coarse level: extraction succeeded, `failure_category` was `synthesis_failure`, `compile_pass` and `final_pass` were false, and generated metrics were absent. It did not contain a separate `synth_pass` field; the v0.4 exporter inferred PPA gates and mapped `compile_pass=false` to sanitized `compile_failure`. That creates a naming mismatch between raw run failure (`synthesis_failure`) and sanitized first PPA gate failure (`compile_failure`), but the artifact still correctly marks all records invalid.

3. Generated-area metrics were missing because Yosys generic synthesis failed before writing generated statistics. The wrong evaluator did not run: the config snapshot used `yosys_equiv`, which intentionally invokes generic Yosys synthesis first. The exporter did not choose the wrong metric source; it preserved reference area where available and left generated area null because no generated metric existed.

4. The immediate fix is a code/config/toolchain issue, not a conclusion about weak model output. The evaluator currently emits a bare `techmap` command, relying on Yosys' built-in data-directory lookup. In this local OSS CAD Suite install, that lookup points to `share/techmap.v`, while the package provides `share/yosys/techmap.v`. A small evaluator helper or run configuration should make Yosys' data path explicit, or the rerun should use a known-good Linux Yosys environment.

5. The safest next control rerun is the same `neutral_baseline` control slice only, with no prompt-profile comparison:

- model: `qwen36-27b`
- benchmark: `rtlopt`
- prompt profile: `neutral_baseline`
- limit: 5 tasks
- samples per task: 1
- evaluator: `yosys_equiv`
- same public RTL-OPT checkout
- same local OpenAI-compatible endpoint
- a Yosys preflight that runs `techmap` successfully before model calls
- either use a known-good Linux Yosys on Lanta or patch the Yosys evaluator to call `techmap -map <oss-cad-suite>/share/yosys/techmap.v` when that file is present

Do not run `low_area_low_power` until a neutral control rerun produces valid PPA records. This run has zero valid optimization scores and cannot serve as a comparison baseline.
