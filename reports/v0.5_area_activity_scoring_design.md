# v0.5 Area/Activity Scoring Design

## Decision Summary

v0.5 defines `area_activity` scoring for RFID-APBench. The score rewards generated RTL only when it passes required gates and has measured area plus VCD toggle-count activity under matched public/synthetic benchmark conditions.

Activity means a simulation toggle-count proxy. It is not measured silicon power, estimated signoff power, or a final PPA result.

This report is a design artifact only. It contains no benchmark results.

## Scoring Mode

The scoring mode name is `area_activity`.

For valid samples:

```text
area_score = min(reference_area / generated_area, 2.0)
activity_score = min(reference_activity / generated_activity, 2.0)
score = 0.5 * area_score + 0.5 * activity_score
```

Interpretation:

- `1.0` means parity with the reference metric.
- greater than `1.0` means improvement.
- less than `1.0` means regression.
- each component is capped at `2.0`.
- the combined valid score is greater than `0` and at most `2.0`.

Reference and generated metrics must come from the same task, same interface contract, same synthesis flow, same activity workload, same VCD window, and same metric units.

## Gate Contract

Gates are evaluated before scoring. Failed required gates receive no area/activity credit.

| Order | Gate | Pass condition | Failure category |
| ---: | --- | --- | --- |
| 1 | Extraction | A candidate RTL unit is extracted unambiguously. | `code_extraction_failure` |
| 2 | Compile | RTL parses/elaborates with the declared interface. | `compile_failure` |
| 3 | Simulation or equivalence | The generated design satisfies the task correctness contract. | `simulation_failure` or `equiv_failure` |
| 4 | Synthesis | The declared synthesis flow completes. | `synthesis_failure` |
| 5 | Timing if required | Required timing constraint passes. | `timing_failure` |
| 6 | Area metric availability | Positive finite reference and generated area are available in matched units. | `area_metric_unavailable` |
| 7 | Activity metric availability | Positive finite reference and generated activity are available in matched units. | `activity_metric_unavailable` |

Per-sample artifacts should represent failed gates as `score_status: invalid` with `score: null`. Aggregate all-sample comparisons must also publish zero-filled means where invalid samples contribute zero.

## Metric Validity

Area values must be:

- positive
- finite
- from the declared synthesis flow
- from matched reference and generated runs
- expressed in the same unit or proxy label

Activity values must be:

- positive
- finite
- counted from VCD generated by the declared workload
- measured over the same window for reference and generated designs
- produced by the same toggle-count method

Missing, zero, negative, non-finite, mismatched, or ambiguous metrics invalidate the sample for `area_activity` scoring.

## VCD Toggle-Count Contract

The activity flow must be deterministic and reproducible:

1. compile the reference and generated RTL with the same testbench harness shape
2. apply the task-declared reset and stimulus workload
3. dump VCD over the declared measurement window
4. count toggles using the declared signal inclusion policy
5. compare reference and generated aggregate activity counts
6. export only sanitized aggregate values

The first metric may be `total_signal_toggles` or `width_weighted_signal_toggles`. The selected value must be stored in `activity_metric`. Reports must not compare rows that use different activity metrics as if they were equivalent.

Do not commit VCD files. They are raw benchmark artifacts and can expose generated RTL structure, signal names, and workload traces.

## Timing Policy

Timing is a gate, not a bonus.

- If timing is required, `timing_status` must be `pass` for a valid score.
- If timing is not required by the task, record `timing_status: not_required`.
- If timing should be measured but the tool cannot provide it, record `timing_status: unavailable` and invalidate only when the task declares timing as required.
- Positive slack does not increase the area/activity score.

## Aggregate Reporting

Reports should include:

- total samples
- task count
- gate pass counts
- invalid count by failure category
- valid-score count
- mean and median valid `area_score`
- mean and median valid `activity_score`
- mean and median valid `score`
- all-sample mean score with invalid samples mapped to zero
- task-level wins, ties, and regressions when comparing prompt profiles or models

Reports must lead with gate pass rates before optimization scores. A model or prompt profile with few valid samples must not look strong because failed samples were filtered out.

## Sanitized Area/Activity Record

Commit-safe records should use a schema like:

| Field | Type | Purpose |
| --- | --- | --- |
| `schema_version` | string | Version of this sanitized schema. |
| `benchmark` | string | `rfid_apbench`. |
| `task_id` | string | Public synthetic task id. |
| `model` | string | Model label. |
| `prompt_profile` | string | Prompt profile label. |
| `source_run_id` | string | Sanitized run id. |
| `scoring_mode` | string | `area_activity`. |
| `final_pass` | boolean | True only when all required gates and metric checks pass. |
| `extraction_pass` | boolean | Extraction gate result. |
| `compile_pass` | boolean | Compile/elaboration gate result. |
| `correctness_pass` | boolean | Simulation or equivalence gate result. |
| `correctness_method` | string/null | `simulation`, `equivalence`, or null before the gate. |
| `synth_pass` | boolean | Synthesis gate result. |
| `timing_status` | string | `pass`, `fail`, `not_required`, or `unavailable`. |
| `timing_pass` | boolean/null | Timing gate result when applicable. |
| `timing_constraint` | number/null | Declared timing constraint. |
| `timing_unit` | string/null | Timing unit, usually `ns`. |
| `reference_area` | number/null | Reference area. |
| `generated_area` | number/null | Generated area. |
| `area_unit` | string/null | Area unit or proxy label. |
| `area_metric_status` | string | `available` or `unavailable`. |
| `reference_activity` | number/null | Reference toggle-count activity. |
| `generated_activity` | number/null | Generated toggle-count activity. |
| `activity_metric` | string/null | Toggle-count method identifier. |
| `activity_metric_status` | string | `available` or `unavailable`. |
| `area_score` | number/null | Capped area ratio. |
| `activity_score` | number/null | Capped activity ratio. |
| `score` | number/null | Combined score. |
| `score_status` | string | `valid` or `invalid`. |
| `failure_category` | string | First decisive failure or `passed`. |
| `toolchain_id` | string | Sanitized synthesis/simulation configuration id. |
| `workload_id` | string | Sanitized activity workload id. |

Forbidden fields and payloads:

- private RTL
- private task text
- raw prompts
- raw model responses
- generated RTL
- VCD files or VCD snippets
- output paths that expose raw artifacts
- logs or error-log contents
- secrets
- model weights
- training datasets
- LoRA, QLoRA, or DoRA adapters

## Example Score Cases

| Case | Area score | Activity score | Combined score |
| --- | ---: | ---: | ---: |
| Reference parity | 1.0 | 1.0 | 1.0 |
| 20% smaller, activity parity | 1.25 | 1.0 | 1.125 |
| Area parity, 20% fewer toggles | 1.0 | 1.25 | 1.125 |
| 20% larger, 20% more toggles | 0.8333 | 0.8333 | 0.8333 |
| Failed correctness gate | null | null | null per sample, zero in all-sample mean |

## Reporting Language

Allowed:

- "VCD toggle-count activity improved under the declared workload."
- "The result is a public/synthetic area/activity proxy."
- "Invalid samples are zero-filled in aggregate score comparisons."

Not allowed:

- "Power improved" unless a separate trustworthy power flow exists.
- "Final silicon PPA improved."
- "Private RFID performance improved."
- "The model is better" unless compared on the same tasks, settings, gates, and scoring mode.

## Implementation Readiness

The scoring contract is ready for implementation after the task schema is frozen. Fine-tuning remains blocked until an RFID-APBench MVP scores 5 to 10 public/synthetic tasks end-to-end and emits sanitized reports with no private, raw, output, or training artifacts.
