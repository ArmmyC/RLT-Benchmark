# v0.7 RFID-APBench Endpoint Runtime Investigation

## Scope

This investigation examines the endpoint/runtime boundary behind repeated parsed-success, null-content, token-limit responses in the public/synthetic RFID-APBench runs. It reviews committed sanitized evidence, model configuration, request construction, client parsing, and sanitized local endpoint capability metadata.

No model generation or benchmark evaluation was run. No benchmark task, task prompt, reference, testbench, workload, extractor, evaluator, metric, scoring policy, or prompt profile was changed. Model comparison, private evaluation, training-data creation, fine-tuning, and adapter work remain out of scope.

## Observed Pattern

The bounded prompt-profile experiment produced the same result under `neutral_baseline` and `completion_reliability`:

| profile | empty rows | non-empty rows | empty rate | valid scores |
| --- | ---: | ---: | ---: | ---: |
| `neutral_baseline` | 5/6 | 1/6 | 83.3% | 0/6 |
| `completion_reliability` | 5/6 | 1/6 | 83.3% | 0/6 |

All ten empty rows shared the following sanitized metadata:

- HTTP class `2xx`
- parsed response schema
- one response choice
- `response_content_present=false`
- `response_character_count=0`
- `finish_reason=length`
- `completion_tokens=4096`
- one request attempt

The targeted observability rerun showed the same signature on four additional empty rows. This is not a timeout, HTTP error, transport failure, retry failure, invalid JSON response, missing choice, or client schema rejection. It is a successful parsed response whose first choice has null content after consuming the full requested completion budget.

## Endpoint And Model Configuration

| item | sanitized finding | implication |
| --- | --- | --- |
| configured model | `qwen36-27b` | Matches the model served by the local endpoint. |
| endpoint compatibility | OpenAI-compatible `/v1/models` and `/v1/chat/completions`; model listing available | Existing requests and response parsing are accepted by the endpoint. |
| serving runtime | vLLM `0.22.1` | The serving stack supports OpenAI-compatible chat requests; this report does not infer undocumented server internals. |
| advertised model limit | 131072 tokens | An 8192 completion request is below the advertised model limit, subject to prompt plus completion capacity and server policy. |
| intended thinking control | `configs/models.yaml` sets `chat_template_kwargs.enable_thinking=false` for `qwen36-27b` | Repository policy intends to disable hidden thinking for this model. |
| standard benchmark path | `src/rtlbench/config.py` loads model/config `extra_body`; `src/rtlbench/runner.py` forwards it to the client | Standard benchmark requests carry the thinking control. |
| RFID endpoint path | `load_endpoint_config()` reads only endpoint, credential, model, and timeout environment values | The RFID runner does not load the model preset or its `extra_body`. |
| RFID generation call | passes model, prompts, temperature, top-p, and max tokens only | `chat_template_kwargs.enable_thinking=false` is absent from RFID requests. |
| client support | `OpenAICompatibleClient.generate()` accepts `extra_body` and merges it into the JSON payload | No client capability change is needed merely to transmit the existing flag. |
| client content parsing | reads `choices[0].message.content` and safely normalizes null to empty text | The committed schema correctly describes null content, but it does not record whether an endpoint-specific reasoning field exists. |
| current max-token assumption | RFID default is 4096 and is enforced per invocation, not as a model capability claim | Reaching 4096 proves the requested budget was exhausted; it does not prove 4096 is the serving maximum. |

The material configuration mismatch is that RFID requests do not carry the thinking-disable flag already defined for this model and used by the standard benchmark path.

## Runtime Hypotheses

| hypothesis | evidence | assessment |
| --- | --- | --- |
| Hidden thinking consumes the full completion budget before visible content | Thinking is intended to be disabled, but the RFID runner omits the flag; empty rows end at exactly 4096 tokens with `finish_reason=length`. Repository comments already identify this Qwen failure mode. | `strongly_supported_configuration_risk` |
| An endpoint-specific reasoning field is populated while visible `content` remains null | The client observes only visible content; the signature is compatible with reasoning-only output terminated at the token limit. No raw body or sanitized reasoning-field-presence metadata was retained. | `plausible_but_unconfirmed` |
| The 4096 output budget is intrinsically too small for the requested RTL | The visible tasks are bounded, and non-empty outputs under the same limit used fewer than 4096 tokens. Raising the limit could merely permit more hidden reasoning. | `not_the_first_diagnostic` |
| The chat template or server ignores `enable_thinking=false` | Possible, but the flag was never sent by this runner, so endpoint compliance has not yet been tested on this path. | `premature_to_attribute` |
| Client null normalization creates the failure | Normalization preserves accurate content presence and character count; it does not cause the server to return null or consume 4096 tokens. | `not_supported_as_primary_cause` |
| Benchmark assets, extraction, or evaluation cause null content | Null content occurs before extraction and evaluation, while references and known-good fixtures pass. | `not_supported` |

## Max-Token Diagnostic Decision

Raising `max_tokens` from 4096 to 8192 is technically possible as a bounded diagnostic because the endpoint advertises a 131072-token model limit and the runner exposes a per-run `--max-tokens` option. A future test could safely remain within one sample on `ap_010_retry_timeout_fsm`, `neutral_baseline`, and sanitized metadata only. It must not become a global default without evidence.

However, an 8192-token diagnostic is not justified before fixing and verifying the missing thinking configuration. Running it now would confound output budget with hidden-thinking behavior, could double inference cost and latency, and might simply produce 8192 hidden tokens followed by null content. Preserve 4096 as the current default.

## Thinking And Chat-Template Decision

The RFID runner should first be specified to resolve the configured model preset and forward its non-secret `extra_body`, including `chat_template_kwargs.enable_thinking=false`, through the already-supported client parameter. Mocked tests should prove the payload contains the flag and that unrelated generation settings remain unchanged.

A subsequent validation should be separately bounded and sanitized. It should confirm whether the endpoint accepts the flag and whether the null-content token-limit signature changes, without changing task prompts, prompt profiles, evaluator behavior, or scoring.

## Raw-Response Inspection Decision

Raw-response inspection is not needed to select the next step. The code/configuration gap and current sanitized metadata already establish that intended thinking control was not sent.

If null-content responses persist after the flag is correctly propagated, a later local diagnostic may inspect only enough ephemeral response structure to determine whether an endpoint-specific reasoning field is present. Any body, generated content, prompt, or RTL must remain in ignored local storage and must never enter committed Markdown, CSV, JSONL, logs, fixtures, or tests. Prefer adding a sanitized boolean/count field over retaining raw content if further observability becomes necessary.

## Benchmark Asset Decision

Keep benchmark tasks, task prompts, references, testbenches, workloads, extractor, evaluator, metrics, and scoring unchanged. The observed condition occurs at request/template/runtime response generation before candidate evaluation, and known-good benchmark validation remains healthy.

## Fine-Tuning Readiness

Fine-tuning remains out of scope and is not recommended. A request-configuration mismatch is not training evidence, creates no governed training dataset, and should be resolved before drawing model-level conclusions.

## Recommended Next Step

Recommend exactly one bounded next step: `thinking_mode_configuration_bugfix_spec`.

That specification should make the RFID runner use the existing model preset `extra_body`, prove `enable_thinking=false` is transmitted with mocked tests, preserve all benchmark/evaluation behavior, and define only a narrowly bounded sanitized validation. It should not raise the global token default, run a benchmark-wide evaluation, compare models, tune task prompts, inspect or commit raw responses, or create training data.

## Public Synthetic Caveat
