# v0.8 RFID-APBench Reproducibility Docs

## Scope And Non-Goals

This report documents the v0.8 reproducibility guide created for the v0.7 RFID-APBench public/synthetic post-fix baseline. The guide explains how release owners and reviewers can reproduce or audit the baseline with placeholder-only environment examples, deterministic settings, command templates, output boundaries, verification steps, and troubleshooting notes.

This is documentation hardening only. No model evaluation, benchmark modification, prompt edit, reference RTL edit, testbench edit, model comparison, prompt-profile experiment, max-token change, private evaluation, training-data creation, fine-tuning, adapter work, or git tag creation was performed.

## Use With Release Checklist

Use `docs/release/rfid_apbench_reproducibility.md` before or during the release checklist in `docs/release/rfid_apbench_release_checklist.md`.

Recommended flow:

1. Set local endpoint environment variables using placeholder-only documentation as the template.
2. Confirm local tools and Python tests are healthy.
3. Run the full baseline or bounded targeted validation only when a release owner or approved spec calls for it.
4. Commit only sanitized Markdown/CSV/JSONL reports under `reports/`.
5. Use the release checklist to verify evidence, settings, report hygiene, artifact boundaries, claim safety, known limitations, and tag readiness before tagging.

The reproducibility guide does not create tags and does not replace the release checklist. It supplies the run and audit instructions that checklist users need.

## Benchmark Asset Decision

Benchmark assets remain unchanged by this reproducibility documentation step. No task prompt, reference RTL, testbench behavior, module interface, workload, metric, scoring rule, evaluator policy, manifest identity, prompt profile, or max-token default was changed.

The only approved v0.7 benchmark asset change remains the previously completed `ap_001_idle_counter` public prompt clarity update. The `ap_006_wakeup_edge_filter` issue remains a candidate implementation limitation in the v0.7 evidence, not a benchmark asset defect.

## Fine-Tuning Readiness Decision

Fine-tuning remains out of scope and is not recommended. The reproducibility guide and v0.7 evidence are public/synthetic evaluation and release-hardening artifacts, not a governed training dataset or sufficient basis for model-weight changes, LoRA/QLoRA/DoRA adapters, or private-task adaptation.

## Recommended Next Bounded Direction

Recommend exactly one bounded next direction: `v0.8_report_hygiene_spec`.

Rationale: the release checklist and reproducibility guide now cover pre-tag gates and user-facing rerun instructions. The next useful hardening step is to define stable report schema, lineage, naming, and sanitized-field expectations before adding automated CI release assertions.

## Activity Proxy Caveat

Activity is a VCD toggle-count proxy from the declared public workload. It is not measured silicon power, signoff power, final silicon PPA, or production QoR.

## Public Synthetic Caveat

This documentation covers only public/synthetic RFID-APBench evidence. It contains no private data or private RTL, is not private evaluation, does not create training data, and is not fine-tuning or training-readiness evidence.
