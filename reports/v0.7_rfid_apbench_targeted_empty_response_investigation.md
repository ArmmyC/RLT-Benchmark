# v0.7 RFID-APBench Targeted Empty-Response Investigation

## Scope

This investigation focuses only on the two public/synthetic RFID-APBench tasks that produced all six empty responses in the expanded v0.6 baseline:

- `ap_001_idle_counter`
- `ap_010_retry_timeout_fsm`

It assesses whether the committed sanitized fields can distinguish an accepted request with empty text from request, endpoint, timeout, tool, evaluator, or benchmark failures. No model request or targeted rerun was performed.

## Non-Goals

- No benchmark task, prompt, prompt profile, reference, testbench, workload, metric, extractor, evaluator, or scoring formula is changed.
- No benchmark-wide baseline, prompt experiment, model comparison, private evaluation, fine-tuning, or training-data creation is performed.
- No raw prompt, model response, generated RTL, simulator or synthesis log, VCD, compiled artifact, tool scratch, private text, or private RTL is included.

## Existing Empty-Response Evidence

### Expanded Baseline

| task id | samples | generation completed | empty responses | extracted candidates | valid scores |
| --- | ---: | ---: | ---: | ---: | ---: |
| `ap_001_idle_counter` | 3 | 3 | 3 | 0 | 0 |
| `ap_010_retry_timeout_fsm` | 3 | 3 | 3 | 0 | 0 |

For all six rows:

- endpoint status was `available`
- generation status was `completed`
- extraction status was `failed`
- candidate availability was false
- failure category was `empty_response`
- the sanitized note was `empty response`

The six rows account for every expanded-run empty response. No other task returned empty text in that run.

### Earlier Evidence

The earlier five-task post-tool-health baseline produced two empty responses for `ap_001_idle_counter`; its third sample returned extractable output but failed functional simulation. Across those two completed three-sample baselines, the idle-counter task therefore produced five empty responses in six attempts. This is descriptive repeatability evidence, not a model comparison.

`ap_010_retry_timeout_fsm` first entered the benchmark in the expansion, so its available multi-sample evidence is the expanded run's three empty responses in three attempts.

## Current Sanitized Observability

| condition | distinguishable now? | committed evidence or limitation |
| --- | --- | --- |
| accepted request with empty text | Yes, as an observable runner outcome | `endpoint_status=available`, `generation_status=completed`, and `failure_category=empty_response` show that the client success path returned an empty text value. |
| request or transport failure | Yes at a high level | Exceptions are recorded as `generation_status=request_failed` and `failure_category=request_failed`; none occurred in the six targeted rows. |
| endpoint unavailable | Yes | Missing configuration or failed required preflight produces blocked rows with `endpoint_status=unavailable` and `failure_category=endpoint_unavailable`; neither condition occurred here. |
| timeout versus another request failure | No, not structurally | Client request exceptions are retried and ultimately collapsed into generic `request_failed`; there is no dedicated sanitized timeout category or retry count. |
| tool-health blocker | Yes at a high level | Startup health gates produce `tool_unavailable` or `tool_health_failed`, and the baseline report records healthy Icarus, `vvp`, and Yosys. |
| evaluator or candidate failure | Yes at a high level | Compile, correctness, synthesis, and activity outcomes have distinct fields and failure categories. Empty rows retain `empty_response` rather than being classified as downstream candidate failures. |

Current fields are sufficient to say that these six rows followed the request-success path and exposed empty text to the runner. They are also sufficient to reject endpoint unavailability, generic request failure, and tool/evaluator failure as the recorded category.

Current fields are not sufficient to determine whether the empty text represents a genuine zero-token model completion, an empty choice or content value returned by the server, response normalization inside the client, or another accepted-request serving/runtime condition. The correct causal classification is `endpoint_or_serving_cause_insufficiently_observed`.

## Missing Sanitized Fields

The following bounded metadata would materially improve attribution without retaining response text or secrets:

| proposed field | purpose |
| --- | --- |
| `request_outcome` | Structured success, timeout, HTTP failure, transport failure, response-schema failure, or endpoint blocker category. |
| `request_attempt_count` | Distinguish first-attempt success from success or failure after client retries. |
| `latency_seconds` | Preserve the latency already returned by the client for successful requests. |
| `response_choice_count` | Distinguish a normal single choice from a missing or empty choice collection. |
| `response_content_present` | Record whether the response carried a non-null content field before text normalization. |
| `response_character_count` | Confirm zero-length output without storing output content. |
| `finish_reason` | Preserve the sanitized completion termination reason when supplied. |
| `prompt_tokens`, `completion_tokens`, `total_tokens` | Preserve sanitized usage already available from compatible endpoints; zero completion tokens would be especially informative. |
| `http_status_class` | Record success or error class without endpoint secrets or response bodies. |
| `response_parse_status` | Distinguish valid response parsing from missing choices, content, or malformed JSON. |

These fields should remain sanitized and must not include raw response text, raw prompts, authorization values, response bodies, or diagnostic log excerpts.

## Targeted Rerun Decision

Do not perform the optional targeted rerun yet. The existing evidence already shows repeatability and task concentration: `ap_001_idle_counter` is empty in 3/3 expanded samples and 5/6 samples across two baselines, while `ap_010_retry_timeout_fsm` is empty in 3/3 expanded samples.

An additional six samples with unchanged observability could change the empirical empty-response counts but would still not distinguish zero-token completion from accepted-request serving/runtime behavior. A rerun should follow, not precede, a separately specified sanitized observability change.

This decision is not a prompt-profile conclusion. The evidence is insufficient to treat the empty rows as a prompt/completion-reliability problem until the accepted-request response metadata is better observed.

## Endpoint, Tool, Evaluator, And Benchmark Assessment

| concern | assessment |
| --- | --- |
| endpoint unavailability | Not observed. The endpoint was recorded as available and all six requests used the completed generation path. |
| request or transport instability | Not observed in committed categories. No targeted row is `request_failed`, but accepted-request serving behavior remains insufficiently observed. |
| timeout | Insufficiently distinguished. A timeout would be collapsed into generic request failure after retries; no structured timeout field exists. |
| tool-health issue | Not observed. The expanded baseline recorded healthy Icarus, `vvp`, and Yosys checks. |
| evaluator defect | Not supported. Empty rows are classified before candidate evaluation, while known-good fixtures pass the evaluator for both tasks. |
| benchmark asset problem | Not supported. Both public references and both `reference_copy` candidates pass the full gate stack. |

## Benchmark Asset Decision

Keep both benchmark tasks and all associated assets unchanged. The public references and known-good candidate fixtures validate the interfaces and behavior, and the empty rows occur before extraction or evaluation. There is no strong evidence for changing either prompt, reference, testbench, workload, extractor, evaluator, metric, or scoring rule.

## Fine-Tuning Readiness

Fine-tuning is out of scope and is not recommended from this evidence alone. Six empty outputs from two evaluation tasks do not provide a training dataset, causal diagnosis, governance basis, or evidence that model training would address the observed request-success ambiguity.

## Recommended Next Step

Recommend exactly one bounded next step: `runner_or_endpoint_observability_spec`.

That specification should add only sanitized request and response metadata needed to distinguish zero-token completion, empty server content, response-schema failure, timeout, retry behavior, and generic transport failure. It should define tests and a bounded follow-up path without changing prompts, benchmark assets, evaluator policy, models, or training scope.

## Activity Proxy Caveat

## Public Synthetic Caveat

This investigation covers only public/synthetic RFID-APBench evidence. It contains no private data or private RTL, is not private evaluation, and is not fine-tuning or training-readiness evidence.
