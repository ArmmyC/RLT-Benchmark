## v0.7 Timeline

1. The v0.6 expanded 3-sample baseline attempted 30 public/synthetic RFID-APBench rows and exposed a response-boundary problem: 6 empty-response rows, concentrated on `ap_001_idle_counter` and `ap_010_retry_timeout_fsm`.
2. Endpoint and runtime investigation found the RFID runner was not propagating the configured Qwen thinking-disabled request body. The selected model preset already contained `chat_template_kwargs.enable_thinking=false`, but that `extra_body` was not reaching the OpenAI-compatible request path.
3. The runner path was fixed so `generate_samples()` forwards the model preset `extra_body` through `OpenAICompatibleClient.generate()`. Targeted validation then reproduced visible non-empty responses across 6/6 rows with no null-content token-limit recurrence.
4. The post-fix evidence separated response reliability from candidate correctness. `ap_001_idle_counter` was no longer empty, but candidates still failed simulation.
5. The `ap_001_idle_counter` public prompt was clarified to explicitly state that when `active` is high and `count` is `8'h0e`, the next rising clock edge must update `count` to `8'h0f` and assert `expired` in the same sequential update.
6. The `ap_001_idle_counter` post-clarity targeted validation attempted exactly 3 samples and produced 3/3 successful, non-empty, extractable, compiling, functionally passing, synthesizing, scored rows.
7. The post-fix expanded 10-task baseline attempted 30 rows across the full manifest and produced 30/30 successful non-empty responses with 27/30 valid scores.
8. The remaining invalid rows were isolated to `ap_006_wakeup_edge_filter`. Failure analysis classified them as candidate implementation failures rather than endpoint, extractor, tool-health, evaluator, scoring, prompt-profile, token-limit, or benchmark-asset defects.

## `ap_001_idle_counter` Closeout

The approved `ap_001_idle_counter` prompt clarity update made the terminal-count timing requirement explicit without changing the reference RTL, testbench, interface, workload, metrics, evaluator, scoring, or manifest identity.

The post-clarity targeted validation attempted exactly 3 samples with `neutral_baseline`, `qwen36-27b`, deterministic defaults, max tokens unchanged at 4096, and the fixed thinking-disabled runner path. All 3 rows were successful non-empty requests, extractable, compilable, functionally correct, synthesizable, and scored. The result supports the interpretation that the prompt clarity improved `ap_001` candidate correctness under the fixed request path.

## `ap_006_wakeup_edge_filter` Limitation

The remaining v0.7 limitation is localized to `ap_006_wakeup_edge_filter` candidate behavior:

- Sample 2 is best classified as `syntax_or_compile_construct_error` with an incomplete or malformed candidate.
- Samples 1 and 3 are best classified as `edge_filter_semantics_mismatch`, `wakeup_pulse_width_mismatch`, and `state_update_timing_mismatch`.

The failure analysis found no evidence that the remaining failures are caused by endpoint availability, token limits, the thinking-disabled runner path, extraction, tool health, evaluator policy, scoring, the unchanged prompt profile, or benchmark assets. The failures should be treated as model-candidate correctness limitations in this run.

## Asset And Policy Decisions

Benchmark assets remain unchanged after the approved `ap_001_idle_counter` public prompt clarity update. No further task prompt, reference RTL, testbench, module interface, workload, metric, scoring, evaluator, or manifest change is justified by the v0.7 evidence.

`completion_reliability` remains rejected and unused. The post-fix evidence was produced with `neutral_baseline`, and the response-boundary fix came from request configuration rather than prompt-profile substitution.

A global max-token increase is not justified. Post-fix observed completion tokens remained well below the existing 4096-token limit, with the expanded baseline maximum at 899 completion tokens.

Fine-tuning is out of scope and not recommended. The evidence is public/synthetic benchmark validation and does not provide training-data governance, private-task coverage, model-weight-change justification, or adapter-readiness evidence.
