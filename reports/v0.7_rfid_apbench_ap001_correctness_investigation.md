# v0.7 RFID-APBench ap_001 Correctness Investigation

## Scope

This report investigates the persistent functional simulation failures for the public/synthetic `ap_001_idle_counter` task after the RFID runner's thinking-mode configuration fix resolved the empty-response boundary issue.

This is sanitized correctness analysis only. No model generation, benchmark-wide evaluation, prompt-profile experiment, model comparison, private evaluation, training-data creation, fine-tuning, or adapter work was performed. Benchmark tasks, prompts, references, testbenches, evaluator behavior, scoring, model settings, and token limits remain unchanged.

## Issue Boundary

The committed and local evidence supports the following exclusions:

- Not an empty-response issue: 3/3 responses were non-empty.
- Not an endpoint or token-limit issue: all rows were parsed HTTP success, `finish_reason=stop`, with 348 completion tokens.
- Not an incomplete or malformed candidate: extraction passed for all rows.
- Not a compile issue: Icarus compilation passed for all rows.
- Not a tool or synthesis issue: Yosys synthesis passed and area metrics were available for all rows.
- Not a benchmark asset issue: the committed reference and known-good fixture validation pass, and the public reference/testbench agree on the terminal transition.

The remaining boundary is generated-candidate functional behavior under simulation.

## Public Expected Behavior

At a high level, the public task requires:

- active-low asynchronous reset clears the 8-bit count and `expired`
- synchronous `clear` has priority over active counting and clears both outputs
- when `active` is low, count and `expired` hold their state
- when `active` is high, count increments toward `0x0f`
- on the clock edge that advances count from `0x0e` to `0x0f`, count saturates and `expired` becomes high
- after expiration, count remains at `0x0f` until reset or clear

The public reference implements the terminal count and `expired` assertion in the same sequential state update. The public testbench checks reset/idle behavior, the terminal result after 15 active count cycles, idle hold after expiration, and synchronous clear.

## Failure Classification

Ephemeral inspection found that the repeated candidate increments from `0x0e` to `0x0f` while keeping `expired` low, then asserts `expired` on a later active clock after count is already at `0x0f`. The simulator therefore fails the public terminal-count expectation.

| sanitized category | assessment | evidence boundary |
| --- | --- | --- |
| `state_update_timing_mismatch` | primary classification | Count reaches the terminal value one state update before `expired` asserts. |
| `output_latency_mismatch` | supporting classification | `expired` is delayed by one active clock relative to the reference contract. |
| `testbench_expectation_not_met` | observed result | The terminal count/expiration check fails before later hold and clear checks complete. |
| `enable_or_idle_behavior_mismatch` | not supported by inspected logic | The candidate explicitly holds state while inactive; the simulation stops earlier at the terminal check. |
| `reset_behavior_mismatch` | not supported | Reset logic matches the public active-low reset form, and the failure occurs later. |
| `counter_width_or_wrap_mismatch` | not supported | The candidate uses an 8-bit count and saturates rather than wrapping. |
| `incomplete_or_malformed_candidate` | not supported | Extraction and compilation pass. |
| `benchmark_asset_issue_not_supported` | supported exclusion | Reference, testbench, and known-good validations agree. |
| `extractor_issue_not_supported` | supported exclusion | The complete required module was extracted consistently. |
| `tool_or_synthesis_issue_not_supported` | supported exclusion | Compile and synthesis pass under healthy tools. |

The failure is sufficiently classified; `insufficient_sanitized_evidence` is not needed for the primary terminal-timing mismatch. This does not establish how frequently a nondeterministic sample would choose another implementation because the tested deterministic rows were identical.

## Task And Prompt Clarity Assessment

No concrete reference or testbench bug was found. Their expected behavior is mutually consistent and the benchmark assets should not be changed in this investigation.

The public prompt's terminal wording can plausibly be read as asserting `expired` only after count is already `0x0f`, which matches the repeated candidate's one-cycle-late implementation. A narrowly scoped clarity specification could state explicitly that `expired` must assert on the same active clock edge that changes count from `0x0e` to `0x0f`, without changing reference behavior, task intent, interfaces, workloads, metrics, evaluator policy, or scoring.

## Response Reliability Versus Correctness

The thinking-disable fix remains successful at the response boundary: all three rows are visible, complete, extractable outputs. The current failure is downstream functional correctness. These gates must remain separate; non-empty output, compilation, and synthesis do not imply simulation correctness.

## Benchmark Asset Decision

Keep all current benchmark assets unchanged. The evidence does not support a task, reference, testbench, extractor, evaluator, or synthesis bug. Any future prompt/task clarity work should first be specified and should preserve the current functional contract rather than silently modifying benchmark behavior.

## Fine-Tuning Readiness

Fine-tuning remains out of scope and is not recommended. One deterministic candidate pattern repeated three times is not a training dataset and provides no data-volume, diversity, governance, or model-weight justification.

## Recommended Next Direction

Recommend exactly one bounded next direction: `ap001_prompt_or_task_clarity_spec`.

That specification should document the simultaneous terminal count and `expired` update more explicitly while preserving the reference and testbench contract. It should not run a benchmark-wide baseline, compare models, change prompt profiles, raise token limits, create training data, or modify scoring/evaluator policy.

## Activity Proxy Caveat

Activity is a VCD toggle-count proxy from the declared public workload. It is not measured silicon power, signoff power, final silicon PPA, or production QoR. No candidate activity score is valid because the candidates failed functional simulation.

## Public Synthetic Caveat

This investigation covers only a public/synthetic RFID-APBench task. It contains no private data or private RTL, is not private evaluation, and is not fine-tuning or training-readiness evidence.
