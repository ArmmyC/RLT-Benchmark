# v0.7 RFID-APBench Thinking-Mode Configuration Propagation Bugfix

## Scope

This change fixes model-preset `extra_body` propagation in the public/synthetic RFID-APBench 3-sample runner. Future `qwen36-27b` RFID generation requests now carry the existing non-secret chat-template setting `chat_template_kwargs.enable_thinking=false`.

This is configuration plumbing only. No model generation or benchmark evaluation was run. No benchmark task, task prompt, prompt profile, reference, testbench, workload, generation setting, extractor, evaluator, metric, scoring policy, or report schema was changed.

## Root Cause

The endpoint-runtime investigation found that the standard benchmark path loads `configs/models.yaml`, merges model preset `extra_body`, and forwards it to `OpenAICompatibleClient.generate()`. The RFID runner instead loaded only base URL, credential, timeout, and model from environment variables and called the client without `extra_body`.

For `qwen36-27b`, this omitted the repository's intended `chat_template_kwargs.enable_thinking=false` setting. That gap is consistent with the observed parsed HTTP `2xx` responses that contained one choice, null visible content, `finish_reason=length`, and 4096 completion tokens. This bugfix corrects the gap but does not claim the runtime symptom is resolved until a separately bounded validation is performed.

## Files Changed

- `scripts/run_rfid_apbench_3sample_baseline.py`
  - resolves model preset metadata from `configs/models.yaml`
  - forwards resolved `extra_body` through `generate_samples()` to the client
- `tests/test_rfid_3sample_baseline.py`
  - verifies Qwen thinking-disable resolution, fallback behavior, and runner propagation
  - verifies the selected prompt-profile system prompt remains unchanged
- `tests/test_client_observability.py`
  - verifies `extra_body` reaches the final mocked JSON request payload

No shared client, benchmark asset, evaluator, scoring, or report schema file required modification.

## Preset Resolution

The runner resolves preset metadata after the existing endpoint configuration has selected the effective model name:

1. An exact `configs/models.yaml` preset key is preferred.
2. If no exact key exists, one unique preset whose `name` equals the effective model is accepted.
3. An unknown model returns an empty `extra_body`, preserving custom environment-selected model compatibility.
4. A missing `configs/models.yaml` also returns an empty `extra_body` deterministically.
5. Malformed preset mappings, malformed `extra_body`, or ambiguous served-name matches raise a clear `ValueError` rather than selecting unpredictably.

For the configured `qwen36-27b` preset, resolution produces:

```yaml
chat_template_kwargs:
  enable_thinking: false
```

This is public configuration metadata, not a prompt, response, credential, or model output.

## Endpoint And Generation Compatibility

Existing endpoint environment behavior is unchanged:

- `QWEN_BASE_URL` / `OPENAI_BASE_URL` still select the endpoint.
- `QWEN_API_KEY` / `OPENAI_API_KEY` still supply the credential without being recorded.
- `QWEN_TIMEOUT` / `OPENAI_TIMEOUT` still select request timeout.
- `QWEN_MODEL` / `OPENAI_MODEL` still override the effective model, with the existing `qwen36-27b` default.

The effective model is used only to resolve optional preset metadata. Unknown external model names continue with an empty body rather than losing endpoint compatibility.

Prompt profile resolution and the selected system prompt are unchanged. Task prompts, temperature, top-p, max tokens, task filtering, exactly-three-samples enforcement, extraction, evaluation, scoring, blocker rows, observability fields, and sanitized Markdown/CSV/JSONL schemas are unchanged.

## Request Propagation

`generate_samples()` now accepts the resolved `extra_body` and passes it to `OpenAICompatibleClient.generate()`. The existing client merges the mapping into the outgoing OpenAI-compatible JSON payload. For `qwen36-27b`, mocked payload inspection confirms the request contains `chat_template_kwargs.enable_thinking=false` while preserving `max_tokens=4096` and the selected system prompt.

No raw request or response body is retained by the tests or report.

## Tests

Mocked tests cover:

- `qwen36-27b` resolves `chat_template_kwargs.enable_thinking=false`
- unknown model fallback is deterministic and empty
- missing preset-file fallback is deterministic and empty
- `generate_samples()` forwards resolved `extra_body`
- the selected prompt-profile system prompt is still forwarded
- the final client JSON payload contains the thinking-disable flag
- max-token payload behavior remains unchanged
- repeatable task filtering remains covered
- existing observability and report-schema coverage remains intact

No test contacts a live model endpoint. The full repository test suite passes.

## Benchmark Asset Decision

Keep all benchmark tasks, task prompts, references, testbenches, workloads, extractor, evaluator, metrics, and scoring unchanged. The defect was confined to model request configuration propagation.

## Fine-Tuning Readiness

Fine-tuning remains out of scope and is not recommended. Correcting a request flag is infrastructure maintenance, not training evidence, and creates no training dataset or basis for model-weight or adapter changes.

## Recommended Next Step

Recommend exactly one bounded next step: `bounded_thinking_disabled_validation_spec`.

That specification should use only the previously targeted public tasks, `neutral_baseline`, unchanged 4096-token generation settings, and at most one sample per task. It should confirm sanitized request outcomes and token/content metadata after thinking is disabled. It must not run a benchmark-wide evaluation, compare models, change prompts or benchmark assets, raise global token limits, inspect or commit raw response bodies, or create training data.

## Public Synthetic Caveat

This bugfix concerns only public/synthetic RFID-APBench request configuration. It contains no private data or private RTL, is not private evaluation, and is not fine-tuning or training-readiness evidence. It makes no measured-power or final-silicon-PPA claim.
