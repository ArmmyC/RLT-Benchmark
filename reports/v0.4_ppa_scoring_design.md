# v0.4 PPA Scoring Design

## Decision Summary

v0.4 will award optimization credit only to generated RTL that passes required correctness and implementation gates. The first complete score balances measured area and power equally. Timing is enforced as a constraint when available and is not rewarded.

This document defines a scoring contract for future implementation. It contains no new benchmark results.

## Research Question

Can `qwen36-27b` produce synthesizable, functionally equivalent RTL with measured area and/or power improvement relative to a reference implementation under the same toolchain and constraints?

## Why v0.3 Is Insufficient

v0.3 measured prompt-profile functional outcomes. It did not establish controlled power measurements, timing constraints, or a correctness-gated PPA objective. The `low_area_low_power` prompt describes an optimization intent, but intent is not measurement.

An area or power claim requires, at minimum:

- successful extraction and compilation
- successful synthesis under matched assumptions
- proven functional equivalence
- comparable measured reference and generated metrics
- timing validity when a timing constraint is part of the experiment

Therefore v0.3 results cannot be reinterpreted as PPA results.

## Gate Contract

| Order | Gate | Pass condition | Failure result |
| ---: | --- | --- | --- |
| 1 | Extraction | A candidate RTL unit is extracted unambiguously. | Invalid score; category `code_extraction_failure`. |
| 2 | Compile | RTL parses/elaborates with the declared compiler and interface. | Invalid score; category `compile_failure`. |
| 3 | Synthesis | The declared synthesis flow completes and emits valid metrics. | Invalid score; category `synthesis_failure`. |
| 4 | Equivalence | Generated RTL is proven equivalent to the reference under the declared contract. | Invalid score; category `equiv_failure`. |
| 5 | Timing | The declared timing constraint passes when timing is available and required. | Invalid score; category `timing_failure`. |

Gates are evaluated before optimization scoring. Failed required gates receive no optimization credit.

For per-sample data, use a null score with `score_status: invalid`. For aggregate all-sample scoring, map invalid samples to zero. Reports must not silently discard failed samples.

## Metric Contract

Area and power ratios are meaningful only when reference and generated designs use identical measurement conditions.

Required provenance for a scored run includes:

- synthesis tool and version
- technology/library identifier or explicit generic-flow identifier
- synthesis script/configuration identifier
- clock and timing constraints, if any
- power method and activity source, if any
- metric units

Suggested units:

- `reference_area`, `generated_area`: the native area unit emitted by the declared library/flow
- `reference_power`, `generated_power`: watts, with the power method recorded
- `timing_slack`: nanoseconds

Values must be finite and greater than zero before ratios are computed. Missing, zero, negative, non-finite, or unit-mismatched values make the relevant component unavailable or invalid.

## Full PPA Score

When all required gates pass and trustworthy area and power values are available:

```text
score = 0.5 * area_score + 0.5 * power_score
area_score = min(reference_area / generated_area, 2.0)
power_score = min(reference_power / generated_power, 2.0)
```

The component cap of `2.0` limits each metric to a maximum 2x reward. The combined score therefore ranges from greater than `0` to `2.0` for valid positive inputs.

Examples:

| Result | Area score | Power score | Optimization score |
| --- | ---: | ---: | ---: |
| Reference parity | 1.0 | 1.0 | 1.0 |
| 20% smaller, power parity | 1.25 | 1.0 | 1.125 |
| Area parity, 20% lower power | 1.0 | 1.25 | 1.125 |
| 20% larger and 20% higher power | 0.8333 | 0.8333 | 0.8333 |

Scores describe relative optimization under one controlled flow. They are not portable physical constants.

## Timing Policy

Timing is a gate/constraint, not a reward.

- If timing is available and required, `timing_pass` must be true for a valid optimization score.
- Report timing constraint and slack, but do not add positive slack to the score.
- If timing is unavailable, set `timing_status: unavailable` and state that timing was not enforced.
- Do not label an area-only, timing-unavailable result as full PPA validation.

This policy avoids rewarding over-pipelining or timing optimization when the stated objective is area and power, while still preventing area/power gains that violate required performance.

## Power-Unavailable Policy

Public RTL-OPT may not provide trustworthy power data. In that case:

```text
scoring_mode = area_only
optimization_score = area_score
power_status = power_unavailable
reference_power = null
generated_power = null
power_score = null
```

Area-only scores remain subject to extraction, compile, synthesis, equivalence, and any available required timing gate.

Reports must say "area-only" and must not claim power improvement. Power must not be estimated from area, cell count, switching-related prompt language, or unsupported heuristics.

## Aggregate Reporting

For each prompt profile, report:

- total samples
- pass count for every gate
- valid full-PPA score count
- valid area-only score count
- invalid score count by failure category
- mean and median valid component scores
- mean all-sample optimization score with invalid samples mapped to zero
- task-level wins, ties, and regressions against the control profile

The all-sample score is the primary comparison view because it preserves correctness failures. Valid-only statistics are diagnostic and must be labeled accordingly.

Do not combine full-PPA and area-only scores in one unlabeled ranking.

Future commit-safe per-sample artifacts should contain fields such as:

| `model` | string | Served model name. |

| `source_run_id` | string | Stable sanitized run identifier. |

| `timing_slack` | number/null | Measured slack. |
| `reference_area` | number/null | Reference area from the matched flow. |
| `generated_area` | number/null | Generated-design area from the matched flow. |
| `area_unit` | string/null | Area unit or generic-flow label. |

| `generated_power` | number/null | Generated-design power when measured. |

Artifacts must not include raw RTL, prompts, responses, error-log text, API keys, private task text, or paths to those materials.

## `low_area_low_power` Retest Plan

The first implementation should validate scoring with `neutral_baseline` as the control. After the public flow is stable, retest `low_area_low_power` on a matched RTL-OPT subset.

The retest must:

1. use identical model and generation settings
2. use identical task ordering and sample counts
3. use identical synthesis, equivalence, timing, and power flows
4. compare correctness-gate rates before optimization scores
5. publish area-only status if power remains unavailable
6. require equivalence before claiming any optimization win

`low_area_low_power` should be promoted only if it improves the correctness-aware aggregate score or shows a credible task-level optimization advantage without degrading required gates.

## Public RTL-OPT Limitations

The public flow can validate the scoring mechanics, but current evidence may be limited by:

- absent trustworthy power/activity data
- absent or incomplete timing constraints
- synthesis estimates rather than place-and-route measurements
- tool/library dependence of area and cell metrics
- limited task scale and design diversity
- synthesis heuristic variability

Consequently, an initial v0.4 result may be area-only. It must not be presented as final PPA or final silicon performance.

## Private Task Boundary

SiliconCraft/RFID private tasks are intentionally deferred. They should be introduced only after the public RTL-OPT flow demonstrates:

- deterministic gate evaluation
- matched and versioned metric provenance
- correct invalid-score handling
- sanitized exports with no raw/private content
- reproducible summaries

Private evaluation will require a separate data-access and artifact-retention policy. It is not part of this design milestone.

## Implementation Sequence

1. Implement pure gate and score calculation functions.
2. Unit-test valid, capped, unavailable, and invalid cases.
3. Export sanitized per-task records.
4. Summarize valid and zero-filled aggregate views separately.
5. Validate the public RTL-OPT flow without changing frozen baselines.
6. Run the matched `low_area_low_power` retest only after review.

No benchmark execution occurred for this scoring-design report.
