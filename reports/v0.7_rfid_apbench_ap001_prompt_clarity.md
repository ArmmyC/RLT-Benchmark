## Previous Wording

The prior public prompt said:

> Once `count` is `8'h0f`, hold the count and assert `expired`.

The `ap_001` correctness investigation found this could plausibly be read as asserting `expired` one active clock after `count` first becomes `8'h0f`. That delayed behavior does not match the committed reference and testbench contract.

## Clarified Wording

The prompt now states:

> When `active` is high and `count` is `8'h0e`, the next rising clock edge must update `count` to `8'h0f` and assert `expired` in the same sequential update.

It also states:

> After expiration, hold `count` at `8'h0f` and keep `expired` high until reset or synchronous clear.

The reset, synchronous clear, active counting, inactive hold, saturation behavior, module name, and port list remain unchanged.

## Behavior-Preserving Rationale

This is a clarity-only update. The existing reference RTL already updates `count` from `8'h0e` to `8'h0f` and asserts `expired` in the same clocked branch. The existing testbench already checks the terminal state after the expected active count sequence and then checks that the terminal state holds while inactive.

The new wording does not relax or strengthen the evaluator beyond the existing public behavior. It documents the same simultaneous terminal update that the reference and testbench already require.
